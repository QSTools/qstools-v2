Mirra

Mirra is not:

- an autonomous agent
- an automation engine
- an accounting system
- a job management clone
- a pricing oracle
- a decision-making replacement

Mirra is:

- a financial intelligence layer

System observes

# 3. CORE SYSTEM OBJECTIVE

Mirra exists to answer:

Does this business model actually work?

That means:

- does revenue create enough margin?
- does COGS reconcile to the accounting source?
- does labour recover labour?
- do assets recover assets?
- do overheads get recovered?
- does unit margin carry the business in product/unit mode?
- does the selected recovery model work?
- do required rates expose the true commercial pressure?
- where is the leak?
- what must be reviewed before relying on the model?

Mirra must not simply show totals.

Mirra must show:

- source
- calculation
- variance
- reason
- consequence
- readiness
- trust state

# 4. UNIVERSAL BUSINESS EQUATION

Every business must recover its true cost through an output driver.

Universal equation:

Total Business Cost ÷ Output Driver = Required Recovery Per Output

The output driver may be:

- productive hours
- units sold
- m²
- m³
- tonnes
- lineal metres
- transactions
- jobs
- subscriptions
- users
- custom unit

The unit changes.

The principle does not.

1. Hours-based / service-based mode
2. Product / unit-based mode

These are not industry categories.

They are recovery-driver categories.

## Hours-based mode

Use when the business mainly sells:

- time
- labour
- services
- charge-out hours
- productive capacity
- asset-supported services

- builders
- contractors
- trades

- service businesses
- labour + plant businesses

productive hours

Primary recovery display:

$/hour

## Product / unit-based mode

Use when the business mainly sells:

- products

- repeatable outputs
- transactions
- m²

Examples:

- concrete pumping by m³
- concrete placing by m²

- subscriptions

units / selected commercial unit

# 6. CROSS-SUBSIDY PRINCIPLE

Mirra exists to expose cross-subsidy.

Cross-subsidy means one strong-looking stream hides weakness in another stream.

Examples:

- materials propping up labour
- labour hiding asset under-recovery
- assets buried inside blended margin
- strong product streams hiding weak product streams
- material margin hiding labour under-recovery
- overheads hidden inside blended gross margin
- product volume hiding weak unit margin

The system must not hide cross-subsidy.

The system must expose it.

---

# 7. HOURS-BASED RECOVERY RULE

In an hours-based business:

Labour recovers labour.
Assets recover assets.
Materials cover themselves and create margin.

Each recovery stream must stand on its own.

No cross-subsidy.

## Labour

Labour cost is real.

Mirra must not assume labour is recovered.

Mirra must enforce that labour must be paid.

Use:

- total real labour cost
- productive labour hours
- weighted productive staff type rates where available

Do not use:

- charge-out rate as cost
- target rate as cost
- desired margin as cost
- guessed productivity

## Assets

Assets need their own recovery driver.

The driver is:

hours used per week
converted to annual utilisation hours

Asset utilisation belongs in the Assets module.

Asset-to-labour or asset-to-operational-group pairing belongs in Cost Allocation.

## Materials / COGS

In hours-based mode:

COGS creates material margin.

Material margin must not automatically recover labour.

Material margin must not automatically recover assets.

Material margin is a margin layer.

It may create additional contribution, but it must not hide whether labour and assets recover themselves.

---

# 8. PRODUCT / UNIT-BASED RECOVERY RULE

In a product / unit-based business:

COGS is consumed to create the product.

Revenue minus COGS creates margin.

That margin must recover the whole business.

The unit must carry:

- labour
- assets
- overheads
- profit

There are no separate labour/material/asset recovery streams at the recovery-test level.

Product/unit mode tests:

Margin per unit
vs
Required cost burden per unit

The unit must carry the model.

---

# 9. COGS RULE

COGS treatment is mode-dependent.

## Hours-based mode

COGS creates material margin.

Material margin must not recover labour or assets by default.

Correct logic:

Revenue - COGS = Material Margin

Material Margin is tested separately from:

- labour recovery
- asset recovery
- overhead recovery

## Product / unit-based mode

COGS is part of the product.

Correct logic:

Revenue - COGS = Margin Pool

Margin Pool must recover:

- labour
- assets
- overheads
- profit

# 10. P&L SOURCE-OF-TRUTH RULE

The P&L is the accounting anchor.

Mirra must always be able to show:

- what the P&L says
- what Mirra calculates
- where the difference comes from
- whether the difference is a true problem, a timing issue, a classification issue, or an intentional modelling treatment
- how the operating result reconciles back to P&L net profit

The P&L is not always the final operational truth.

But it is always the accounting baseline.

Mirra may adjust interpretation for operational truth, but it must never hide the adjustment.

---

# 11. P&L RECONCILIATION RULE

Every major source module must either:

1. reconcile back to the P&L, or
2. clearly show why it does not reconcile.

No unreconciled value may be treated as trusted upstream.

If a module total differs from the P&L benchmark, Mirra must show:

- source P&L amount
- module amount
- variance amount
- variance percent
- variance status
- possible interpretation
- owning module
- downstream trust consequence

Do not silently overwrite the module value with the P&L value.

Do not silently overwrite the P&L value with the module value.

Always show variance.

Never silent overwrite.

---

# 12. ACCOUNTING VS OPERATIONAL TRUTH

Mirra must clearly separate:

Accounting result
from
Operational result

Accounting result comes from the P&L.

Operational result comes from Mirra’s structured modules and recovery model.

Example bridge:

Traditional P&L Net Profit
less Labour variance
less Asset finance variance
less General Overheads variance
less Other Income excluded from operating margin
equals Mirra Operating Position

This bridge must remain visible where relevant.

The user must be able to see why the operating position differs from accounting profit.

---

# 13. VARIANCE RULE

Every variance must be treated as evidence.

A variance is not automatically an error.

A variance may indicate:

- timing difference
- classification difference
- incomplete P&L
- journal not posted
- current model differs from historical P&L
- owner/director pay timing
- asset finance recognition issue
- depreciation timing issue
- COGS import mismatch
- missing or duplicate source item
- modelling treatment difference

The wording must not overclaim.

Use:

- review signal
- may indicate
- possible cause
- timing or classification issue
- not automatically an error

Do not use:

- this proves
- this is definitely wrong
- bookkeeping error
- accountant error
- overtime problem
- missing cost confirmed

unless the data proves it.

---

# 14. LEAK RULE

A leak exists when value disappears, duplicates, or is used upstream without reconciliation.

Examples:

- COGS detail does not reconcile to P&L COGS
- labour cost differs from P&L and the variance is hidden
- asset costs are double-counted in Assets and General Overheads
- overheads are over-allocated
- asset pools are over-assigned
- productive labour hours are counted twice
- COGS are used as both direct cost and overhead
- material margin is used to hide labour under-recovery
- an imported value flows upstream without source validation
- an allocation exceeds the available source pool
- a user assignment overwrites original source identity

Leaks must not be normalised.

Leaks must be shown.

Where appropriate, leaks must block downstream trust.

---

# 15. STRICT BALANCE RULE

Balanced means:

difference = 0.00

or rounding-only difference.

Default rounding tolerance:

absolute difference <= 1.00

Anything above rounding tolerance must be shown as a variance.

For control-source imports, anything above rounding tolerance should be treated as:

Leak detected

unless a later module brief explicitly defines a reviewed variance state.

Do not describe material differences as acceptable just because the percentage is small.

Small percentages can become large dollar leaks at scale.

---

# 16. SOURCE POOL RULE

Any allocation from a source pool must obey:

assigned amount <= available source pool

For every pool, Mirra must calculate:

available_pool_amount
assigned_pool_amount
remaining_pool_amount
over_allocated_amount
allocation_status

Formula:

remaining_pool_amount =
available_pool_amount - assigned_pool_amount

over_allocated_amount =
max(0, assigned_pool_amount - available_pool_amount)

Over-allocation must never be silently capped.

Over-allocation must never be automatically redistributed.

The user must correct the allocation.

# 18. MODULE BOUNDARY RULE

No module may:

- rebuild upstream maths
- rename upstream variables casually
- use display names as join keys
- hide source state
- change another module’s source truth

# 21. STANDARD STATUS VALUES

Where practical, modules should use consistent readiness/status values.

Allowed broad states:

ready
ready_with_warnings
review_required
blocked
leak_detected
preview_only
not_ready

Meaning:

ready =
source data complete, reconciled, and trusted

ready_with_warnings =
usable but warnings remain visible

review_required =
not blocked, but user review required before relying downstream

blocked =
must not be used downstream as trusted

leak_detected =
source, allocation, or reconciliation mismatch that may cause hidden loss

preview_only =
can display but must not be treated as final

not_ready =
required setup is missing

Modules may define additional local statuses only where required.

# 22. TRUST STATE RULE

Every major module should expose a trust state.

Suggested variable:

model_trust_state

Allowed values:

ready
warning
blocked
preview_only

Trust must flow downstream.

If an upstream module is blocked, downstream modules must not show final trusted outputs.

If upstream is warning, downstream may calculate but must carry the warning.

If upstream is preview_only, downstream must show preview_only.

# 27. ACCOUNTING IMPORT RULE

Excel or structured data import is preferred over PDF.

Preferred order:

1. API import
2. Excel export
3. CSV export
4. manual entry
5. PDF only if no structured data exists

Reason:

Structured imports provide:

- rows
- columns
- account headings
- exact numeric values
- repeatable validation
- easier reconciliation

PDF introduces:

- OCR errors
- layout extraction issues
- hidden row problems
- merged heading problems
- poor validation
