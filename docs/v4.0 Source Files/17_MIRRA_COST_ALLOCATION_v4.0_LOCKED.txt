PRIMARY RULE:
Cost Allocation is the working-group builder. It assigns existing productive labour, productive assets, and remaining overheads into operating groups. It does not test recovery, build rates, create recovery strategy, or issue the business verdict.

---

# 1. PURPOSE

Cost Allocation builds the operating structure of the business.

It answers:

What operating groups does the business run?
Which productive labour belongs to each group?
Which productive assets belong to each group?
Which assets are shared across groups?
How should remaining overheads be distributed across groups?
What labour, asset, or overhead cost is still unassigned?
Has any labour, asset, or overhead pool been over-assigned?
What cost stack does each group carry before recovery and rate testing?

Cost Allocation should be simple to use.

A basic business should be able to use this page quickly with one default operating group.

Advanced users may create more detailed groups.

Cost Allocation must stay simple.

It should answer one question:

How is the business operationally structured?

It should not ask the user to solve recovery, pricing, or final business strategy.

Those questions belong downstream.

Cost Allocation should feel like:

Create group
Assign labour
Assign assets
Distribute overheads
Check everything balances

That is all.

# 6. DEFAULT HOLDING POOL PATTERN

Each assignable cost pool must have a visible default holding pool.

Default holding pools are not extra costs.

They are the visible representation of unassigned source values.

Required default pools:

1. Default Productive Labour Pool
2. Default Productive Asset Pool
3. Default Overhead Pool

The default pool starts with 100% of its source value.

As the user assigns cost to working groups, values move out of the default pool.

Nothing is copied.

Nothing is duplicated.

Nothing is hidden.

---

# 7. DEFAULT POOL LIFECYCLE

Each default pool has three possible states.

## State 1 — Default pool > 0

Meaning:

Some source value remains unassigned.

Status:

review_required unless the user intentionally leaves it unassigned.

## State 2 — Default pool = 0

Meaning:

The pool has been fully allocated.

Status:

clean / balanced.

This is usually the ideal state.

## State 3 — Default pool < 0

Meaning:

The source pool has been over-assigned.

Status:

blocked.

The system must never allow trusted downstream output when a pool is negative.

---

# 8. SOURCE POOL RULE

Cost Allocation works only from existing source pools.

Core source pools:

1. Productive Labour Pool
2. Productive Asset Pool
3. Remaining / Net General Overheads Pool

Each pool must expose:

- available amount
- assigned amount
- remaining amount
- over-allocated amount
- allocation status

No source cost may disappear.

No source cost may be counted twice.

No pool may be silently over-assigned.

# 9. SIMPLE DEFAULT SETUP RULE

To keep the page easy to use, Cost Allocation should support a simple default setup.

Default setup may create:

Primary Operations Group

Then optionally assign:

- all productive labour
- all productive assets
- all remaining overheads

to that one group.

This gives a simple business a working, balanced allocation quickly.

The user may then split into additional groups only if useful.

Cost Allocation should not force complex structure.

# 13. PRODUCTIVE LABOUR ONLY RULE

Only productive labour enters Cost Allocation labour assignment.

Productive labour includes labour that directly supports production, delivery, output, or chargeable work.

Non-productive labour must not enter the Productive Labour Pool.

Non-productive labour belongs in overheads / business support cost pathway.

Examples of productive labour:

- operators
- site labourers
- installers
- drivers where delivery is productive
- production staff
- field staff
- crew members
- tradespeople
- chargeable working roles

Examples of non-productive labour:

- admin
- office support
- management where non-productive
- accounting support
- sales support unless classified productive
- general business support

Non-productive labour must not appear in Default Productive Labour Pool.

---

# 15. LABOUR ASSIGNMENT RULE

Labour assignment moves productive labour from Default Productive Labour Pool to working groups.

It does not copy labour.

It does not create labour.

It does not create a new labour rate.

Example:

Total productive labour = $100,000

Start:

Default Productive Labour Pool = $100,000
Boxing Crew = $0
Steel Fixing Crew = $0

After assignment:

Boxing Crew = $40,000
Steel Fixing Crew = $30,000
Default Productive Labour Pool = $30,000

The totals still reconcile:

$40,000 + $30,000 + $30,000 = $100,000

# 17. LABOUR / ASSET SEPARATION RULE

Labour and assets are independent pools.

Do not assign labour to assets.

Do not attach an operator directly to a machine inside Cost Allocation.

Correct model:

Working Group =
Labour
+ Productive Assets
+ Overheads

Example:

Digger Crew:
- Operator from Productive Labour Pool
- Digger from Productive Asset Pool
- Allocated overheads

Incorrect model:

Operator assigned to Labour Group
and also attached separately to Digger

That would double count labour.

Labour and assets meet only at the working-group level.

# 21. ASSET ASSIGNMENT RULE

Assets behave like labour structurally, but with percentage splitting.

Labour usually moves by hours/cost.

Assets may be split across working groups by assigned percentage.

Example:

Digger annual cost = $60,000
Digger annual utilisation = 600 hours

Assignment:

Site Works = 60%
Drainage = 40%

Then:

Site Works asset burden = $36,000
Site Works asset hours = 360

Drainage asset burden = $24,000
Drainage asset hours = 240

Default Productive Asset Pool remaining = 0%

If only 80% is assigned:

Default Productive Asset Pool holds remaining 20%.

If 110% is assigned:

blocked.

# 23. ASSET UTILISATION CARRY-FORWARD RULE

Cost Allocation must carry forward asset utilisation from Assets.

Required variables:

assigned_utilisation_hours

Formula:

assigned_utilisation_hours =
asset utilisation_hours_annual × assigned_percent

Cost Allocation must not use labour hours as asset hours.

Cost Allocation must not invent asset usage.

Cost Allocation must not overwrite asset utilisation.

Asset utilisation is sourced from Assets.

Cost Allocation only splits that utilisation according to group assignment.

---

# 24. SHARED ASSET RULE

Shared assets are created in Cost Allocation only.

Assets module does not define shared assets.

A shared asset means:

one productive asset is used across multiple working groups.

Example:

Digger:
- 60% Site Works
- 25% Drainage
- 15% General Site Support

Cost Allocation must ensure:

sum assigned_percent for asset_id <= 100%

If sum assigned_percent > 100%:

allocation_status = over_allocated
cost_allocation_status = blocked

Do not silently cap percentages.

Do not auto-correct shares.

Do not create shared asset logic inside Assets.

---

# 25. SHARED ASSET IMPACT ON GROUP COST

For each group:

group_asset_burden =
sum assigned_asset_burden for that group

For each asset assignment:

assigned_asset_burden =
asset_total_cost_annual × assigned_percent

assigned_utilisation_hours =
asset_utilisation_hours_annual × assigned_percent

This creates group-level asset burden and group-level asset hours.

It does not change the original asset cost.

It does not change the original asset utilisation.

It does not change Asset Reality.

# 42. POOL BALANCE RULE

For each source pool:

available_amount
- assigned_amount
= remaining_amount

over_allocated_amount =
max(0, assigned_amount - available_amount)

allocation_status values:

balanced
under_allocated
over_allocated
review_required
not_available

Rules:

- over_allocated blocks trusted output.
- under_allocated is visible review state.
- balanced means within tolerance.
- no silent capping.
- no silent auto-balance.
- no hidden remaining pool.

Default tolerance:

absolute difference <= 1.00

---

# 43. TOTAL COST ALLOCATION RECONCILIATION

Cost Allocation must reconcile assigned group totals to source pools.

Required checks:

1. Labour assigned vs Productive Labour Pool
2. Asset assigned vs Productive Asset Pool
3. Overhead assigned vs General Overheads Pool
4. Group total cost vs assigned source totals
5. Assigned + remaining = available for each source pool

Formula:

total_group_assigned_cost =
sum group_cost_stacks.total_group_cost

total_assigned_source_cost =
assigned_labour_cost
+ assigned_asset_burden
+ assigned_overhead_amount

These must match.

If not:

cost_allocation_status = blocked or leak_detected

---

# 44. NO DOUBLE COUNTING RULE

Cost Allocation must prevent double counting.

Examples:

- A staff type assigned above available hours.
- A productive asset assigned above 100%.
- An overhead pool allocated 100% to one group and again 100% to another.
- Asset running costs transferred into Assets and also distributed in General Overheads.
- Operator assigned once as labour and again as if attached to an asset.

All must be blocked or clearly review-required.

No double counting may flow downstream as trusted.

---

# 45. OVERHEAD DISTRIBUTION RULE

Overheads are distributed across working groups.

They are not ignored.

They are not forced into Assets.

They are not forced into Labour.

They are distributed according to the selected overhead allocation method.

If overheads remain unallocated:

- show remaining overhead amount
- status = review_required
- downstream recovery tests must know overheads remain unallocated

If overheads are over-allocated:

- status = blocked
- downstream trusted output blocked

blocked:

preview_only:
Values display but should not be trusted.

not_ready:
Required inputs or group setup missing.
