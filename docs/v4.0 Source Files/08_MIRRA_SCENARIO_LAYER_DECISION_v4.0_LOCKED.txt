# 1. PURPOSE

This brief records the v4.0 architecture decision for scenario modelling.

The previous concept of a standalone Labour Scenario Layer is not required as an active v4.0 module.

Instead, labour scenario logic will be included later inside the Business Modelling layer.

This keeps the active source chain cleaner and avoids creating isolated scenario modules too early.

---

# 2. DECISION

Do not build:

08_MIRRA_LABOUR_SCENARIO_LAYER_v4.0_ACTIVE.txt

Do not create a separate active Labour Scenario page, hook, storage layer, calculation layer, or selector layer at this point.

Instead:

Labour scenario testing will be handled inside:

20_MIRRA_BUSINESS_MODELLING_v4.0_FUTURE_OR_ACTIVE_AS_DECIDED.txt

when Business Modelling is reached.

---

# 3. WHY THIS DECISION WAS MADE

A labour scenario is not isolated.

Changing labour affects:

- Labour cost
- Productive hours
- Staff type rates
- Cost Summary
- Business Summary
- Revenue Reality
- Cost Allocation
- Recovery Summary
- Rate Builder
- Business Outcome

A user modelling labour will often also want to model:

- adding an operator
- adding a labourer
- changing productive hours
- changing staff role mix
- increasing wages
- adding an asset
- increasing utilisation
- changing overheads
- changing unit volume
- changing sale rates
- changing recovery strategy

That is not a Labour-only problem.

That is a Business Modelling problem.

---

# 4. ACTIVE MODULE RESPONSIBILITY

Labour remains the active source truth.

Labour owns:

- staff identity
- staff type identity
- selected staff role/type
- productive/non-productive status
- annual labour cost
- productive hours
- weighted productive staff type rates
- total labour cost
- total productive labour hours
- labour readiness
- labour trust state

Business Modelling will later own:

- scenario staff rows
- scenario labour cost
- scenario productive hours
- scenario staff type mix
- scenario weighted staff type rates
- scenario asset changes
- scenario overhead changes
- scenario revenue / COGS changes
- scenario recovery changes
- scenario rate changes
- scenario Business Outcome impact

---

# 5. CORRECT SOURCE TRUTH SPLIT

Correct split:

Labour Module
= current labour truth

Business Modelling
= what-if labour changes

Cost Allocation
= assigning current or scenario labour groups to operating structure when the selected model is being tested

Recovery Summary
= testing whether the selected recovery model works

Rate Builder
= converting the selected recovery model into required rates

Business Outcome
= showing the final consequence

---

# 6. WHAT LABOUR MUST STILL CREATE

The Labour module must still create the variables needed for Cost Allocation and Rate Builder.

Required active Labour variables include:

staff_type_id
staff_type_name
productive_staff_type_rates[]
weighted_all_productive_labour_rate
total_productive_labour_cost
total_productive_labour_hours
total_productive_output

The selected staff role/type must be stored on each staff row using:

staff_type_id
staff_type_name

The weighted productive rate by staff type must be exposed through:

productive_staff_type_rates[]

Each productive_staff_type_rates row must include:

staff_type_id
staff_type_name
staff_count
total_annual_cost
total_productive_hours
weighted_productive_hourly_rate
productive_share_percent
is_productive
rate_status
source

Formula:

weighted_productive_hourly_rate =
total_annual_cost / total_productive_hours

Cost Allocation must consume staff_type_id as the stable assignment key.

Cost Allocation must not use staff names or display labels as keys.

# 14. NO-LEAK SCENARIO RULE

No scenario value may become active source truth without explicit user approval.

Future scenario values must never silently overwrite:

- Labour
- Assets
- General Overheads
- Revenue / COGS
- Cost Summary
- Business Summary
- Cost Allocation
- Recovery Summary
- Rate Builder
- Business Outcome

Scenario values are preview-only unless accepted through a controlled baseline update process.

---

# 15. FUTURE ACCEPTANCE RULE

When Business Modelling is built, accepting a scenario must require explicit user confirmation.

Correct pattern:

User creates scenario
↓
Mirra calculates scenario impact
↓
Mirra shows comparison against active baseline
↓
User explicitly accepts change
↓
Controlled baseline update request is created
↓
Source module updates only after approval

Incorrect pattern:

User edits scenario
↓
Source module silently changes

That is not allowed.

---

# 16. WHY THIS PREVENTS LEAKS

Keeping scenario logic inside Business Modelling prevents:

- Labour source truth being polluted by what-if values
- Rate Builder accidentally consuming scenario rates as active rates
- Cost Summary using scenario labour cost as active cost
- Cost Allocation assigning scenario staff as if they exist
- Business Outcome issuing a final verdict from mixed active/scenario inputs
- multiple scenario modules duplicating logic

One scenario engine is safer than separate scenario modules for every source area.

---

# 17. DO NOT CHANGE

Do not change Labour source truth for scenario purposes.

Do not create a Labour Scenario route.

Do not create a Labour Scenario hook.

Do not create Labour Scenario storage.

Do not let Cost Allocation consume scenario Labour unless Business Modelling has created an explicit scenario preview contract.

Do not let Rate Builder consume scenario Labour unless Business Modelling has created an explicit scenario preview contract.

Do not let Business Outcome show scenario result as active baseline.

---

# 18. ACCEPTANCE CRITERIA

This decision brief is accepted when:

1. No standalone active Labour Scenario module is built.
2. Labour remains the source truth for active labour data.
3. Labour still creates staff_type_id and staff_type_name.
4. Labour still creates productive_staff_type_rates[].
5. Cost Allocation consumes Labour staff type rates from Labour, not scenario data.
6. Business Modelling later owns scenario labour changes.
7. Scenario labour variables are clearly marked as scenario variables.
8. Scenario values remain preview-only unless accepted.
9. Scenario acceptance requires explicit user approval.
10. The active v4.0 build continues to Assets after this decision brief.
11. No hidden source mutation is allowed.
12. No scenario value becomes active truth silently.

---

# 19. FINAL PRINCIPLE

Labour is current truth.

Business Modelling is what-if truth.

Scenario values must not pollute source modules.

The selected staff role/type and weighted productive staff type rates are created in Labour for active use.

Scenario staff roles and scenario weighted rates will be created later inside Business Modelling for preview use only.

No assumptions.
No silent scenario mutation.
No duplicate scenario modules.
No active-source contamination.
No leaks.
