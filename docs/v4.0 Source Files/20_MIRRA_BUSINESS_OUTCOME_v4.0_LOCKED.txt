# Mirra — Business Outcome Brief

## v4.0 — BUSINESS DIAGNOSIS, FIRST MODELLING LAYER, SUPPLIER / MATERIAL SENSITIVITY, RATE IMPACT, AND QUOTE CHECKER HANDOFF

STATUS: ACTIVE SOURCE BRIEF
VERSION: v4.0
FILE NUMBER: 20

REPLACES:
- older QS Tools Business Outcome assumptions
- older logic where Business Outcome is only a static summary
- older logic where modelling is deferred entirely to a later page
- older logic that does not connect supplier/material pressure to outcome
- older logic that does not consume Rate Builder outputs
- older logic that does not prepare Quote Checker targets

PRODUCT NAME:
Mirra

FORMER NAME:
QS Tools

PRIMARY RULE:
Business Outcome is the first modelling layer. It explains what is happening, why it is happening, what levers can change it, and what target rates / assumptions should flow to Quote Checker. It does not create source truth, assign costs, or rebuild rates.

---

# 1. PURPOSE

Business Outcome answers:

What is the result of the business model?
Where is the pressure coming from?
What needs to change?
Which levers move the result?
What happens if supplier prices increase or decrease?
What happens if labour recovery improves?
What happens if asset utilisation improves?
What happens if charge-out rates change?
What targets should flow into Quote Checker?

Business Outcome is not just a report.

It is the first modelling layer.

It converts all upstream diagnostics into practical business decisions.

# 3. CORE PRINCIPLE

Business Outcome answers:

What does this mean, and what can we do about it?

It consumes:

- financial truth
- cost structure
- recovery test
- required rates
- supplier/material context
- margin pressure
- asset pressure
- labour pressure

Then it lets the user model changes.

It must not hide bad results.

It must not treat commercial failure as data failure.

It must not mutate upstream source data.

# 13. MATERIAL MARKUP RULE

Business Outcome may calculate and display one blended material markup.

There is one blended material markup at the model level.

The system must not pretend there is a true supplier-level markup unless source data supports that.

Supplier/category detail shows contribution to the blended markup result.

It does not create a separate true markup per supplier.

Correct interpretation:

- material markup = one blended result
- supplier spend = contribution / influence on that result
- category spend = contribution / influence on that result

# 14. SUPPLIER / CATEGORY CONTRIBUTION RULE

Business Outcome should show supplier/category influence.

It may show:

- supplier spend amount
- supplier % of total COG
- supplier % of selected category
- category spend amount
- category % of total COG
- rank by spend
- rank by sensitivity impact

Purpose:

Show which suppliers/categories move the business result the most.

Example:

Concrete purchases are $191,000.

If this is a large share of total COG, small price movement has meaningful outcome impact.

# 15. SUPPLIER SENSITIVITY MODELLING

Business Outcome must support supplier sensitivity modelling.

- supplier cost increase %

- category cost decrease %

- total COG decrease %

Example:

Supplier cost reduction = 1%

Model should calculate:

- COG reduction
- new total COG
- new gross margin
- new margin pool
- new material markup
- recovery pressure change
- required rate impact
- net position impact

---

# 16. SUPPLIER SENSITIVITY FORMULA

For selected supplier:

supplier_adjustment_amount =
supplier_spend × adjustment_percent

new_supplier_spend =
supplier_spend + supplier_adjustment_amount

new_total_cog =
current_total_cog + supplier_adjustment_amount

If adjustment_percent is negative:

COG reduces.

If adjustment_percent is positive:

COG increases.

new_margin_pool =
total_revenue - new_total_cog

new_gross_margin_percent =
new_margin_pool / total_revenue

Do not divide by zero.

---

# 17. CATEGORY SENSITIVITY FORMULA

For selected COG category:

category_adjustment_amount =
category_spend × adjustment_percent

new_total_cog =
current_total_cog + category_adjustment_amount

new_margin_pool =
total_revenue - new_total_cog

new_gross_margin_percent =
new_margin_pool / total_revenue

This allows modelling:

- concrete purchases up/down
- pumping up/down
- subcontracting up/down
- purchases other up/down

# 26. DIAGNOSTIC STATEMENT RULE

Business Outcome should create a clear diagnostic statement.

Examples:

COG pressure:
Material / COG costs are compressing margin. Supplier/category spend shows where changes would move the result most.

Labour pressure:
Labour consumes too much of available margin. Required rates need to increase or productive recovery must improve.

Asset pressure:
Productive assets require higher recovery than current utilisation supports.

Rate pressure:
Current rates are below minimum recoverable rates.

Mixed pressure:
Multiple pressure points exist. Model each lever separately before relying on one fix.

---

# 27. BUSINESS OUTCOME STATUS VALUES

Allowed business_outcome_status values:

- ready
- ready_with_warnings
- review_required
- blocked
- preview_only
- negative_outcome
- modelled_improvement
- not_ready

Status rules:

ready:
Outcome is calculated and source trust is good.

ready_with_warnings:
Outcome is calculated but warnings remain.

negative_outcome:
Current model shows deficit or under-recovery but source data is trusted.

modelled_improvement:
A modelled scenario improves outcome.

review_required:
Inputs are valid but need review.

blocked:
Upstream source blocks trusted output.

preview_only:
Values display but are not trusted final output.

not_ready:
Required upstream values are missing.

A negative outcome is not a data failure.

---

# 28. TRUST STATE

Business Outcome must expose:

business_outcome_model_trust_state

Allowed values:

ready
warning
blocked
preview_only

Mapping:

If Model Readiness is blocked:
business_outcome_model_trust_state = blocked

If Cost Allocation / Recovery Summary / Rate Builder is blocked:
business_outcome_model_trust_state = blocked

If outcome is negative but source data is trusted:
business_outcome_model_trust_state = warning

If modelled scenario is preview-only:
business_outcome_model_trust_state = preview_only

If outcome is calculated and no warnings:
business_outcome_model_trust_state = ready
