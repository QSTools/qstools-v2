# Mirra — Database / Storage / Export Layer Brief

## v4.0 — PERSISTENT DATA, MODEL VERSION STORAGE, SOURCE TRACE, REPORTING TABLES, EXPORT CONTRACTS, AND NO-CALCULATION-OWNERSHIP BOUNDARY

STATUS: FUTURE BUILD BRIEF
VERSION: v4.0
FILE NUMBER: 28

REPLACES:
- older assumptions that module state is enough for long-term reporting
- older assumptions that exports can be disconnected from source trace
- older logic where model versions, quote checks, job actuals, and approvals are not stored consistently
- older logic where Power BI / external reporting tables are created without stable contracts

PRODUCT NAME:
Mirra

FORMER NAME:
QS Tools

PRIMARY RULE:
The Database / Storage / Export Layer stores Mirra outputs, approved model versions, quote checks, job actuals, source trace, and reporting-ready fact/dimension tables. It does not own business calculations, source classification, rate building, modelling, or quote checking logic.

---

# 1. PURPOSE

The Database / Storage / Export Layer answers:

Where is the model stored?
Where are approved versions stored?
Where are quote checks stored?
Where are job actuals stored?
Where is source trace preserved?
How do reporting tables get created?
How does Power BI consume Mirra data?
How do we avoid losing history?
How do we keep outputs consistent across modules?

This layer is the persistence and export backbone.

It turns Mirra from a live page system into a durable business data system.

---

# 2. CORE PRINCIPLE

Modules calculate.

The database stores.

Exports publish.

Reporting consumes.

The Database / Storage / Export Layer must not become a hidden calculation engine.

It stores calculated outputs from source modules and preserves their trace.

---

It sits underneath:

Mirra may use multiple storage layers.

# 6. LOCAL STATE RULE

Local state may be used for:

- page drafts
- temporary user edits
- unsaved working profiles
- UI state
- active selections
- draft model scenarios
- draft quote checks

Local state must not be the only storage for:

- approved model versions
- quote checker targets
- job actuals
- model approval audit trail
- source trace
- reporting history

Approved or historical data must be persisted.

---

# 7. DATABASE RULE

The database stores approved and historical truth.

It should store:

- source imports
- cleaned source rows
- module output snapshots
- approved model versions
- approved rate sets
- approved scenarios
- quote checker target sets
- quote checks
- SOQ rows
- PO rows
- job actuals
- job profitability
- feedback signals
- approval decisions
- reporting facts
- reporting dimensions

The database should not silently recalculate module logic.

---

# 8. SOURCE TRACE RULE

Every stored calculated output must preserve source trace.

source_trace should include:

- source_module
- source_record_id
- source_variable
- source_file
- source_import_id
- calculation_rule
- model_version_id
- created_at
- trust_state

No approved model value should exist without trace.

No job actual cost should exist without trace.

No quote checker target should exist without trace.

# 21. POWER BI HANDOFF RULE

Power BI should consume reporting fact/dimension tables.

Power BI should not consume raw UI state.

Power BI should receive:

- stable table names
- stable column names
- stable IDs
- source trace fields
- model version fields
- trust state fields

This prevents reports from breaking when UI changes.

# 23. HISTORY RULE

Historical records must not be overwritten once approved or used.

Immutable once used:

- approved model versions
- approved rate sets
- quote checker target sets
- quote checks
- job actuals
- job profitability results
- approval audit trail

Corrections should create:

- amended rows
- replacement versions
- or adjustment rows

Do not silently overwrite history.

---

# 24. MIGRATION RULE

Database schema changes must be versioned.

migration rows should include:

{
  migration_id,
  migration_version,
  migration_name,
  applied_at,
  status,
  notes
}

Rules:

- migrations must be ordered.
- migrations must be repeat-safe where possible.
- schema changes must preserve existing data.
- breaking changes require explicit migration notes.

# 32. STATUS VALUES

Allowed storage_status values:

- ready
- ready_with_warnings
- blocked
- preview_only
- not_ready

Definitions:

ready:
Storage/export layer is available and trusted.

ready_with_warnings:
Storage works but warnings exist.

blocked:
Storage cannot be trusted or export failed.

preview_only:
Values display but are not persisted.

not_ready:
Storage layer not configured.
