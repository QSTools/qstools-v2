# Mirra — Reporting / Analytics Brief

## v4.0 — BUSINESS REPORTING, MODEL PERFORMANCE, JOB PROFITABILITY TRENDS, SUPPLIER INSIGHT, RATE PERFORMANCE, AND POWER BI HANDOFF

STATUS: FUTURE BUILD BRIEF
VERSION: v4.0
FILE NUMBER: 27

REPLACES:
- older assumptions that reporting is only dashboard display
- older logic where Power BI / reporting is disconnected from model outputs
- older logic where quote, job, supplier, and rate performance are not traceable back to source
- older logic that does not separate live modelling from historical reporting

PRODUCT NAME:
Mirra

FORMER NAME:
QS Tools

PRIMARY RULE:
Reporting / Analytics displays historical and model performance across the Mirra system. It reports on what happened, what the model predicted, what the quote assumed, what the job actually achieved, and where performance trends are emerging. It does not mutate source modules, approve rates, or change model assumptions.

---

# 1. PURPOSE

Reporting / Analytics answers:

How is the business performing over time?
Are rates improving?
Are quotes aligning with the model?
Are jobs meeting target margin?
Which suppliers are causing pressure?
Which categories dominate COG?
Which labour types are under-recovering?
Which assets are underutilised or under-recovered?
Which quote types are failing?
Which jobs are profitable?
Which jobs are below target?
Are model changes improving outcomes?
What trends should be reviewed?

This layer is for visibility, trends, and reporting.

It is not a modelling page.

It is not an approval page.

It is not a source editing page.

# 29. SOURCE TRACE REQUIREMENT

Every report metric must preserve source trace.

Trace should show:

- source module
- source variable
- source record ID
- model version
- calculation rule
- trust state
- export timestamp

If metric has no trace:

trust_state = warning or blocked depending severity.

---

# 30. REPORTING STATUS VALUES

Allowed reporting_status values:

- ready
- ready_with_warnings
- blocked
- preview_only
- not_ready

Definitions:

ready:
Reports are available and trusted.

ready_with_warnings:
Reports are available but source warnings exist.

blocked:
Critical source data is blocked.

preview_only:
Reports display but are not trusted final output.

not_ready:
Insufficient data exists.

---

# 31. TRUST STATE

Reporting must expose:

reporting_trust_state

Allowed values:

- ready
- warning
- blocked
- preview_only

Mapping:

If Data Quality Dashboard global trust state is blocked:
reporting_trust_state = blocked

If key source modules are preview_only:
reporting_trust_state = preview_only

If reports are available with warnings:
reporting_trust_state = warning

If source data is trusted:
reporting_trust_state = ready
