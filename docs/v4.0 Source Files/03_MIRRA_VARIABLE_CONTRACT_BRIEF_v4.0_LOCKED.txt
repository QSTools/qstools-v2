# Mirra — Variable Contract Brief

## v4.0 — MODULE OWNERSHIP, DOWNSTREAM CONTRACTS, TRUST STATES, AND NO-LEAK VARIABLE STANDARD

STATUS: ACTIVE SOURCE BRIEF
VERSION: v4.0
FILE NUMBER: 03

REPLACES:
- QS Tools v3.x variable contract briefs where naming has changed
- older variable contracts that place Rate Builder after Business Outcome
- older Cost Allocation contracts that do not include default labour, asset, and overhead pools
- older Revenue / COGS contracts that treat P&L COG categories as final operational COG classifications
- any variable contract that allows downstream modules to rebuild upstream maths

PRODUCT NAME:
Mirra

FORMER NAME:
QS Tools

CONTROLLING BRIEFS:
00_MIRRA_SYSTEM_PRINCIPLES_v4.0_LOCKED.txt
01_MIRRA_ACTIVE_SYSTEM_BRIEF_v4.0_LOCKED.txt

PRIMARY RULE:
Every variable has one owner. Every downstream value has a traceable source. No duplicate truth. No hidden leaks.

---

# 1. PURPOSE

This brief defines the active Mirra v4.0 variable contract.

It controls:

- variable ownership
- module boundaries
- upstream inputs
- downstream outputs
- standard status variables
- warning variables
- reconciliation variables
- trust-state variables
- mode-aware variables
- source trace variables
- no-leak guardrail variables

Every module must create only the variables it owns.

Downstream modules may consume upstream variables.

Downstream modules must not recreate upstream variables.

---

# 2. CONTRACT PRINCIPLES

## 2.1 One variable, one owner

Every core variable must have one owning module.

Example:

total_labour_cost_annual belongs to Labour.

Cost Summary may consume it.

Business Summary may display it.

Recovery Summary may use it.

Rate Builder may use it.

But no downstream module may recreate it.

---

## 2.2 Source truth vs interpretation

Mirra must distinguish:

- source truth
- calculated output
- interpretation
- warning
- reconciliation
- scenario value

Example:

P&L account amount is source truth.

Labour module annual people cost is module source truth.

Module Reconciliation compares them.

Business Summary explains the operating bridge.

Business Outcome gives the consequence.

Do not collapse these into one variable.

---

## 2.3 No display names as IDs

Never use display labels as primary IDs.

Use stable IDs.

Allowed:

staff_id
asset_id
pnl_line_id
cog_category_id
operational_group_id

Not allowed:

"Paul"
"Truck"
"Concrete Purchase"
"Operators"
"Materials"

Display names may change.

IDs must remain stable.

---

## 2.4 Snake case

All internal variables must use snake_case.

Examples:

total_cost_burden
required_recovery_per_driver
productive_staff_type_rates
asset_pool_guardrail_status

Do not use camelCase for contract variables unless an existing code path already requires it and migration has not yet been done.

UI component props may use existing project conventions temporarily, but the source contract should be snake_case.

---

## 2.5 No silent aliases

Do not create multiple names for the same concept unless migration requires it.

If an alias is required, mark it clearly.

Example:

business_outcome_status
outcome_status

These may temporarily mirror each other for compatibility, but the owner must define which is primary.

---

## 2.6 Future-only variables

Future-only variables must be marked:

future_only: true

or listed in a future section.

Do not implement future-only variables inside active calculations until the relevant module is active.

---

# 5. STANDARD WARNING OBJECT

Warnings must use structured objects.

Standard shape:

{
  warning_id,
  source_module,
  severity,
  message,
  affected_variable,
  blocking,
  review_action
}

Allowed severity values:

info
warning
critical

Rules:

- warning_id must be stable
- source_module must be included
- message must be plain English
- blocking must be true or false
- review_action should tell the user where to review

Example:

{
  warning_id: "cog_reconciliation_leak_detected",
  source_module: "revenue_cogs",
  severity: "critical",
  message: "COG Account Transactions do not reconcile to P&L Cost of Sales.",
  affected_variable: "cog_reconciliation_status",
  blocking: true,
  review_action: "Review the Account Transactions import and P&L Cost of Sales lines."
}

---

# 6. STANDARD RECONCILIATION OBJECT

Any module that reconciles to a source must use a reconciliation object.

Standard shape:

{
  reconciliation_id,
  source_module,
  source_name,
  source_amount,
  module_name,
  module_amount,
  variance_amount,
  variance_percent,
  reconciliation_status,
  tolerance_amount,
  explanation,
  blocking,
  reviewed
}

Allowed reconciliation_status values:

balanced
rounding_only
variance_review
leak_detected
blocked
not_available

Default tolerance:

tolerance_amount = 1.00

Rules:

- difference = 0.00 means balanced
- absolute difference <= 1.00 may be rounding_only
- anything above tolerance must be visible
- control-source imports above tolerance should be leak_detected unless reviewed state is explicitly supported
- never hide a variance
- never overwrite one side with the other

---

Standard shape:

# 30. FINAL VARIABLE CONTRACT PRINCIPLE

Every Mirra number must answer:

Where did this come from?
Who owns it?
How was it calculated?
Does it reconcile?
Can it be trusted?
Where does it go next?

If those questions cannot be answered, the variable is not ready for downstream use.
