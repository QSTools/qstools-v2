# Mirra — Model Review / Approval Brief

## v4.0 — CONTROLLED MODEL UPDATES, APPROVAL WORKFLOW, RATE REVIEW, SCENARIO ACCEPTANCE, SOURCE TRACE, AND NO-SILENT-CHANGE BOUNDARY

STATUS: FUTURE BUILD BRIEF
VERSION: v4.0
FILE NUMBER: 25

REPLACES:
- older assumptions that feedback automatically updates rates
- older assumptions that model changes can be applied without review
- older logic where actual job feedback silently changes future assumptions
- older logic with no approval workflow between feedback and model updates
- older logic that does not preserve prior approved model versions

PRODUCT NAME:
Mirra

FORMER NAME:
QS Tools

PRIMARY RULE:
Model Review / Approval is the controlled change layer. It lets the user review feedback, accept or reject model changes, approve new target rates or assumptions, and create a new approved model version. It never silently changes source modules or approved rates.

---

# 1. PURPOSE

Model Review / Approval answers:

Which feedback signals should change the model?
Which rate reviews should be accepted?
Which supplier assumptions should be modelled or approved?
Which labour productivity assumptions need review?
Which asset utilisation assumptions need review?
Which Business Outcome scenario should become the approved target?
Which Rate Builder rates should become approved rates?
What model version should Quote Checker use next?

This layer converts feedback into approved model changes.

# 3. CORE PRINCIPLE

No model value changes without approval.

Feedback is evidence.

Approval is decision.

A rate review signal does not become a new rate until approved.

A supplier pressure signal does not become a new assumption until approved.

A Business Outcome scenario does not become the target model until approved.

Quote Checker must use approved targets only.

---

# 4. MODULE ROLE

Model Review / Approval owns:

- feedback review queue
- approval workflow
- reject / defer / accept decisions
- approved model version creation
- approved rate set creation
- approved scenario selection
- approved quote checker target set
- decision notes
- reviewer identity where available
- effective date
- rollback reference
- model version status
- approval warnings
- approval trust state

Model Review / Approval does not own:

- original P&L source values
- raw Revenue / COG values
- Labour source calculations
- Asset source calculations
- General Overheads source values
- Cost Allocation assignments
- Recovery Summary tests
- Rate Builder calculations
- Business Outcome scenario calculations
- Quote Checker checks
- Job Costing actuals
- Model Feedback signal generation

# 16. APPROVAL DECISION RULE

Every approval item must have a decision.

Allowed decisions:

- approve
- reject
- defer
- request_more_data
- mark_as_one_off
- send_to_data_quality
- create_rate_review
- create_business_outcome_scenario
- create_quote_checker_target

No silent approval.

No automatic approval from feedback alone.

---

# 17. EFFECTIVE DATE RULE

Approved changes must have an effective date.

Effective date controls:

- which Quote Checker target applies
- which rate set applies
- which Business Outcome scenario applies
- which future jobs are checked against the new model

Past quotes and jobs must preserve the model version they were checked against.

---

# 18. ROLLBACK RULE

Model Review must support rollback.

Rollback should preserve:

- old approved model version
- new approved model version
- rollback reason
- rollback date
- affected targets
- source trace

Rollback must not delete history.

It should create a new active version pointing back to the prior approved state.

---

# 19. AUDIT TRAIL RULE

Every approved, rejected, deferred, or rolled-back decision must be auditable.

Audit trail should include:

- review_item_id
- decision
- decision_date
- reviewer
- source evidence
- current value
- proposed value
- approved value
- notes

No model change without audit trail.

---

# 20. DATA QUALITY BLOCK RULE

If feedback is data-quality related, it should not directly update the model.

Examples:

- unmapped invoice
- missing quantity
- missing labour hours
- duplicate cost
- unmatched quote line

These should create:

data_quality_fix_required

They must not create approved rate or scenario changes until resolved.

---

# 21. ONE-OFF EXCEPTION RULE

The user may mark feedback as one-off.

Examples:

- bad weather
- unusual site condition
- customer scope change
- staff absence
- supplier mistake
- one-off rework

One-off feedback should remain in history but not drive model changes unless repeated later.

---

# 22. REPEATED ISSUE ESCALATION RULE

Repeated issues should be prioritised.

If a feedback signal repeats across multiple jobs, the review queue should elevate:

- severity
- confidence
- review priority

Repeated issue threshold should default to 3 jobs unless later configured.

---

# 23. POSITIVE SIGNAL APPROVAL RULE

Positive feedback can also support approval.

Examples:

- rate consistently exceeds target
- supplier consistently under allowance
- labour productivity better than model
- asset recovery above target

Positive feedback may support:

- model confidence
- no change decision
- optional target tightening

Do not ignore positive evidence.

# 26. STATUS VALUES

Allowed model_review_status values:

- ready
- ready_with_warnings
- pending_review
- approved
- active
- blocked
- preview_only
- not_ready

Definitions:

ready:
Review queue is available and no blocking source issue exists.

ready_with_warnings:
Review available but warnings remain.

pending_review:
Items require decision.

approved:
A model version or target set has been approved.

active:
An approved model version is active.

blocked:
Approval cannot proceed due to source trust or missing required values.

preview_only:
Review values display but should not be applied.

not_ready:
No feedback or candidate model exists.

---

# 27. TRUST STATE

Model Review must expose:

model_review_trust_state

Allowed values:

- ready
- warning
- blocked
- preview_only

Mapping:

If feedback source is blocked:
model_review_trust_state = blocked

If Rate Builder or Business Outcome is blocked:
model_review_trust_state = blocked

If pending review items exist:
model_review_trust_state = warning

If active model version exists and sources are trusted:
model_review_trust_state = ready

If values are preview-only:
model_review_trust_state = preview_only
