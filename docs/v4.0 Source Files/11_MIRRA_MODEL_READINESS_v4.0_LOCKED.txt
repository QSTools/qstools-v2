PRIMARY RULE:
Model Readiness is the trust gate. It does not calculate source values. It decides whether downstream outputs can be trusted, warned, previewed, or blocked.

---

# 1. PURPOSE

Model Readiness converts source-module readiness and Module Reconciliation evidence into a system-wide trust state.

It answers:

Can Mirra use the current model as trusted input downstream?

Model Readiness does not create business maths.

It does not fix source values.

It does not allocate costs.

It does not calculate rates.

It does not issue the business verdict.

It controls whether downstream modules may show:

- ready output
- ready with warnings
- review required
- preview only
- blocked output
- leak detected

---

# 4. CORE PRINCIPLE

A model is not ready just because numbers exist.

A model is ready only when:

- source modules are complete enough
- material data is not missing
- blocking warnings are resolved
- source pools are not over-allocated
- COGS control data is reconciled where required
- Module Reconciliation has no blocking checks
- trust state is not blocked
- downstream modules can safely consume the outputs

If the model is not trusted, downstream modules must not pretend it is final.

---

# 5. TRUST STATE RULE

Model Readiness must expose:

model_trust_state

Allowed values:

ready
warning
blocked
preview_only

Meaning:

ready:
The model is trusted for downstream use.

warning:
The model can be used, but warnings remain visible.

blocked:
The model must not be used as trusted downstream output.

preview_only:
The model can display calculations for review, but cannot be treated as final.

---

# 6. MODEL READINESS STATUS VALUES

Model Readiness must expose:

model_readiness_status

Allowed values:

ready
ready_with_warnings
review_required
blocked
leak_detected
preview_only
not_ready

Status meanings:

ready:
All required source modules and reconciliation checks are ready.

ready_with_warnings:
Model is usable but warnings remain visible.

review_required:
Model can calculate but user review is required before relying downstream.

blocked:
At least one blocking source/module/reconciliation issue exists.

leak_detected:
A source, allocation, reconciliation, or transfer mismatch may create hidden loss, duplication, or unreconciled output.

preview_only:
Data exists but cannot be treated as trusted final model.

not_ready:
Required setup or source modules are missing.

---

# 7. MODEL READY BOOLEAN

Model Readiness must expose:

model_ready

Rule:

model_ready = true only when:

model_readiness_status = ready

or, where explicitly allowed:

model_readiness_status = ready_with_warnings

if downstream module can safely carry warnings.

Default stricter rule:

model_ready = true only when model_trust_state = ready

If model_trust_state is warning, downstream modules may calculate but must show warnings.

If model_trust_state is blocked or preview_only, downstream modules must not show trusted final output.

# 11. BLOCKING MODULES

Model Readiness must expose:

blocking_modules[]

Each row:

{
  module_id,
  module_name,
  module_status,
  trust_state,
  blocking_reason,
  review_action
}

A module should be listed as blocking when:

- required source data is missing
- source module status is blocked
- trust state is blocked
- source values are invalid
- material unclassified rows exist
- source pool is over-allocated
- COG reconciliation has leak_detected
- Module Reconciliation has blocking checks
- required period is missing
- required source import failed

---

# 12. WARNING MODULES

Model Readiness must expose:

warning_modules[]

Each row:

{
  module_id,
  module_name,
  module_status,
  trust_state,
  warning_reason,
  review_action
}

A module should be listed as warning when:

- source module is ready_with_warnings
- source module is review_required
- non-blocking variance exists
- defaults are used
- manual source is preview-only but still calculable
- non-material review items exist
- under-allocated but visible pools exist
- non-productive assets are excluded from recovery
- P&L to module variance exists but is not blocking

---

# 13. BLOCKING CHECKS

Model Readiness consumes and may aggregate:

blocking_checks[]

from Module Reconciliation and source modules.

Each check should use:

{
  check_id,
  source_module,
  severity,
  message,
  affected_variable,
  blocking,
  review_action
}

Examples of blocking checks:

- P&L missing
- P&L source period missing
- Revenue / COGS COG reconciliation leak
- Labour missing productive hours
- Labour missing staff type for productive staff
- Assets source pool over-allocated
- Asset running cost transfer over-allocated
- Asset running cost double-count risk
- General Overheads invalid
- Module Reconciliation blocked
- Cost Summary source invalid
- source period mismatch

---

# 14. WARNING CHECKS

Model Readiness consumes and may aggregate:

warning_checks[]

Examples of warning checks:

- Labour variance vs P&L
- Asset finance variance vs P&L
- General Overheads variance vs P&L
- Asset utilisation default used
- non-productive assets excluded
- overhead pool under-allocated
- asset pool under-allocated
- Other Income present
- manual source used
- review-required but non-blocking item exists

Warnings must remain visible downstream.

# 17. NO-LEAK READINESS RULES

Model Readiness must block trusted downstream output when any of the following exist:

- unreconciled COG control data above tolerance
- over-allocated asset source pool
- over-allocated overhead source pool
- productive labour assigned beyond source pool
- productive asset assigned beyond 100% in Cost Allocation once that module exists
- overhead assigned beyond available overhead pool once Cost Allocation exists
- asset running cost transfer double-count risk
- source period mismatch between control reports
- missing P&L source when accounting reconciliation is required
- source module blocked
- Module Reconciliation blocked
- source values invalid

A leak must not become trusted output.

# 18. COG READINESS RULE

Revenue / COGS owns COG reconciliation.

Model Readiness must consume:

cog_reconciliation_status

If:

cog_reconciliation_status = leak_detected

or:

cog_reconciliation_status = blocked

then Model Readiness must mark COG-dependent downstream modules as blocked or preview_only.

COG-dependent downstream modules include:

- Revenue / COGS final trusted output
- Recovery Summary where COG margin/unit margin is used
- Rate Builder where COG is used
- Business Outcome where COG affects verdict

---

# 19. ASSET READINESS RULE

Assets owns asset guardrails.

Model Readiness must consume:

assets_status
asset_pool_guardrail_status
assets_model_trust_state

If:

asset_pool_guardrail_status = blocked

then downstream modules that consume asset burden must be blocked or preview_only.

Asset-dependent downstream modules include:

- Cost Summary
- Cost Allocation
- Recovery Summary
- Rate Builder
- Business Outcome

---

# 20. ASSET TRANSFER READINESS RULE

If productive asset running cost transfers exist, Model Readiness must ensure they do not create double counting.

Inputs:

total_asset_running_cost_transferred
remaining_overhead_after_asset_transfers
asset_running_cost_transfer_context
reconciliation_asset_transfer_double_count_risk

If transfer double-count risk exists:

model_readiness_status = leak_detected

or:

model_readiness_status = blocked

depending severity.

The user must correct transfer mapping before trusted downstream use.

---

# 21. LABOUR READINESS RULE

Labour owns productive staff type rates.

Model Readiness must block trusted downstream labour use if:

- Labour is blocked
- total productive hours are zero
- productive staff have missing staff type
- productive staff type rate has invalid_zero_productive_hours
- productive staff type rate has blocking warnings

Affected downstream modules:

- Cost Summary
- Cost Allocation
- Recovery Summary
- Rate Builder
- Business Outcome

---

# 22. GENERAL OVERHEADS READINESS RULE

General Overheads owns overhead source pool.

Model Readiness must block trusted downstream overhead use if:

- General Overheads is blocked
- overhead total is invalid
- overhead source is missing
- asset finance exclusion is invalid
- transferred asset running costs create double counting
- overhead source pool is over-allocated once Cost Allocation exists

Affected downstream modules:

- Cost Summary
- Cost Allocation
- Recovery Summary
- Rate Builder
- Business Outcome

---

# 23. P&L READINESS RULE

P&L is the accounting anchor.

Model Readiness must block trusted accounting-based model if:

- P&L is missing
- P&L import failed
- P&L period is missing
- P&L totals are invalid
- material unclassified P&L lines exist
- source period mismatch exists

If P&L is missing but the user is intentionally creating a manual preview model, status should be:

preview_only

not ready.

---

# 24. PERIOD MATCH RULE

Where two accounting reports are used as control sources, periods must match.

Examples:

- P&L period
- Account Transactions COG period
- future Xero imports
- future monthly actuals

If periods do not match, mark affected check as blocking.

Do not silently month-shift.

Do not assume timing correction.

Do not auto-adjust report period.

The user must supply matching periods or explicitly accept a reviewed scenario where allowed.

# 26. DOWNSTREAM DISPLAY RULE

If model_trust_state = ready:

Downstream modules may show trusted outputs.

If model_trust_state = warning:

Downstream modules may show calculated outputs but must display warnings.

If model_trust_state = blocked:

Downstream modules must not show final trusted outputs.

They may show:

- blocked state
- preview-only calculations
- source issues
- review actions

If model_trust_state = preview_only:

Downstream modules must clearly label outputs as preview.

---

# 27. RELATIONSHIP TO COST SUMMARY

Cost Summary consumes Model Readiness.

If Model Readiness is blocked:

Cost Summary should not present its output as final trusted cost burden.

It may display preview values if source totals exist.

Cost Summary must show:

- model trust state
- blocking reasons
- source module warnings

Cost Summary must not override Model Readiness.

---

# 28. RELATIONSHIP TO BUSINESS SUMMARY

Business Summary consumes Model Readiness.

If Model Readiness is blocked:

Business Summary should show preview-only or blocked state.

It may still explain what can be calculated, but must not imply final diagnosis.

Business Summary must carry warnings into its bridge and forward proof.

---

# 29. RELATIONSHIP TO COST ALLOCATION

Cost Allocation consumes Model Readiness and source module statuses.

If Model Readiness is blocked due to Labour, Assets, or General Overheads:

Cost Allocation must not treat those pools as ready.

If only COG is blocked, Cost Allocation may still show structural setup, but downstream recovery/rate use must carry COG warning where relevant.

Cost Allocation must inherit blocked status from:

- Labour source blocked
- Assets source blocked
- Asset pool blocked
- General Overheads blocked
- Overhead source invalid
- Model Reconciliation blocked

---

# 30. RELATIONSHIP TO RECOVERY SUMMARY

Recovery Summary consumes Model Readiness.

If Model Readiness is blocked:

Recovery Summary may show preview calculations only.

If COG reconciliation is blocked:

COG margin and unit margin tests must be blocked or preview-only.

If Labour is blocked:

Labour recovery test must be blocked.

If Assets are blocked:

Asset recovery test must be blocked.

If General Overheads are blocked:

Total recovery tests involving overheads must be blocked.

---

# 31. RELATIONSHIP TO RATE BUILDER

Rate Builder consumes Model Readiness.

If Model Readiness is blocked:

Rate Builder must not show rates as final.

If upstream source is blocked:

- COG blocked → COG/unit/material margin rates blocked
- Labour blocked → labour and crew rates blocked
- Assets blocked → asset rates blocked
- Cost Allocation blocked → group/unit stack rates blocked
- Recovery Summary blocked → rate consequence is preview-only or blocked

No rate can be trusted if its source is blocked.

---

# 32. RELATIONSHIP TO BUSINESS OUTCOME

Business Outcome consumes Model Readiness.

If Model Readiness is blocked:

Business Outcome must not issue a final viable/not-viable verdict as trusted.

It should instead show:

- model not trusted
- blocking source issues
- required review actions
- preview-only consequence where useful

Business Outcome must not hide readiness failures behind a verdict.

# 45. FINAL PRINCIPLE

Model Readiness is the trust gate.

It decides whether the model can be relied on downstream.

It does not fix numbers.

It does not create new source truth.

It does not hide warnings.

It does not let leaks become trusted output.

Ready means trusted.
Warning means visible caution.
Blocked means do not rely.
Preview means not final.

No assumptions.
No hidden warnings.
No source mutation.
No unreconciled downstream use.
No leaks.
