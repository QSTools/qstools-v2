PRIMARY RULE:
Module Reconciliation is evidence-only. It compares P&L benchmark values against Mirra module values and shows variance. It does not change either side.

---

# 1. PURPOSE

Module Reconciliation compares accounting evidence from the P&L against Mirra source module outputs.

It answers:

Where does Mirra agree with the P&L?
Where does Mirra differ from the P&L?
Which module owns the variance?
Is the variance a warning, a review item, or a blocking leak?
Can the model be trusted downstream?

Module Reconciliation is a visibility and evidence layer.

It does not correct data.

It does not make decisions.

It does not calculate the business outcome.

---

# 2. MODULE ROLE

Module Reconciliation owns:

- P&L benchmark vs module total comparison
- variance amount
- variance percent
- reconciliation status
- blocking check list
- warning check list
- evidence for Model Readiness
- normalised reconciliation inputs
- P&L-to-module variance context
- other income treatment visibility
- source module trust context

Module Reconciliation does not own:

- P&L source data
- Labour source data
- Assets source data
- General Overheads source data
- Revenue / COGS source data
- COG Account Transactions reconciliation calculation
- Cost Summary aggregation

# 4. CORE PRINCIPLE

Always variance.

Never silent overwrite.

If the P&L says one value and Mirra says another, the system must show:

- P&L amount
- Mirra module amount
- difference
- variance percent
- likely review area
- status
- downstream consequence

Module Reconciliation must not decide that one side is correct.

The user decides after reviewing the evidence.

---

# 5. P&L AS ACCOUNTING BENCHMARK

The P&L is the accounting anchor.

P&L provides benchmark values such as:

- labour benchmark
- asset finance benchmark
- general overheads benchmark
- total operating expenses benchmark
- Cost of Sales benchmark
- revenue
- gross profit
- net profit
- other income
- other expenses

These are benchmark values.

They are not automatically operational truth.

Module Reconciliation compares them to Mirra module values.

# 14. STANDARD RECONCILIATION OBJECT

Every reconciliation check should use this shape:

{
  reconciliation_id,
  source_module,
  source_name,
  source_amount,
  module_name,
  module_amount,
  variance_amount,
  variance_percent,
  reconciliation_status,
  tolerance_amount,
  explanation,
  blocking,
  reviewed
}

Allowed reconciliation_status values:

balanced
rounding_only
variance_review
leak_detected
blocked
not_available

Default tolerance:

1.00

---

# 15. VARIANCE FORMULA

For each check:

variance_amount =
module_amount - source_amount

variance_percent =
variance_amount / source_amount

If source_amount = 0 and module_amount != 0:

variance_percent = null
reconciliation_status = variance_review or leak_detected depending check

If both source_amount and module_amount = 0:

variance_amount = 0
variance_percent = 0
reconciliation_status = balanced

---

# 16. BALANCE RULE

Balanced means:

variance_amount = 0.00

Rounding-only means:

absolute variance_amount <= 1.00

Anything above tolerance must be visible.

For control-source imports such as COG Account Transactions:

anything above tolerance should be leak_detected unless a reviewed variance state is explicitly supported.

For module vs P&L comparisons:

above tolerance may be variance_review unless it creates downstream risk.

Do not hide small percentage variances.

Small percentage leaks can become large dollar leaks at scale.

---

# 17. REQUIRED RECONCILIATION CHECKS

Module Reconciliation should include, where data exists:

1. Labour vs P&L Labour Benchmark
2. Asset Finance vs P&L Asset Finance Benchmark
3. General Overheads vs P&L General Overheads Benchmark
4. Total Operating Cost Modules vs P&L Operating Expense Benchmark
5. Cost Summary Total Cost Burden context
6. Revenue / COGS COG Reconciliation status reference
7. Other Income visibility
8. P&L net profit to Mirra operating position bridge context

Not every check must be blocking.

Every material check must be visible.

---

# 18. LABOUR RECONCILIATION

Check:

labour_to_pnl_benchmark

source_amount =
labour_benchmark_total

module_amount =
total_labour_cost_annual or total_people_cost_annual

variance_amount =
module_amount - source_amount

Purpose:

Show whether Labour module true people cost aligns with the P&L labour benchmark.

Possible reasons for variance:

- timing difference
- payroll classification difference
- owner/director pay treatment
- employee overheads included elsewhere
- missing staff
- P&L period mismatch
- module uses forward/current cost while P&L is historical

Do not call variance an error automatically.

---

# 19. ASSET FINANCE RECONCILIATION

Check:

asset_finance_to_pnl_benchmark

source_amount =
asset_finance_benchmark_total

module_amount =
asset_finance_interest_annual or selected asset finance total

Purpose:

Show whether asset finance / interest in Assets aligns with the P&L benchmark.

Possible reasons for variance:

- end-of-year journal not posted
- loan interest timing
- principal/interest split issue
- asset finance classified elsewhere
- asset not entered
- P&L period mismatch

Do not include principal repayment as operating cost.

---

# 20. GENERAL OVERHEADS RECONCILIATION

Check:

general_overheads_to_pnl_benchmark

source_amount =
general_overheads_benchmark_total or adjusted_general_overheads_benchmark

module_amount =
total_general_overheads

Purpose:

Show whether General Overheads module aligns with P&L operating expense benchmark.

Possible reasons for variance:

- asset finance excluded
- running costs transferred to Assets
- staff overheads reclassified
- COGS accidentally included in overheads
- manual overhead adjustment
- operating expense classification issue

If asset running cost transfers are active, reconciliation must make clear whether General Overheads is gross or net of transferred asset costs.

---

# 21. ASSET RUNNING COST TRANSFER CONTEXT

If Assets transfers running cost pools from General Overheads into productive asset burden, Module Reconciliation must preserve visibility.

Required context:

asset_running_cost_transfer_context

Suggested shape:

{
  total_asset_running_cost_transferred,
  gross_general_overheads_before_transfer,
  net_general_overheads_after_transfer,
  transfer_status,
  double_count_risk,
  affected_pools
}

Purpose:

Prevent double counting between Assets and General Overheads.

Module Reconciliation must show if:

- transferred amount appears in Assets
- same amount still appears fully in General Overheads
- net overhead pool has not been adjusted or marked
- transfer exceeds available source pool

Over-allocated transfer should be blocking.

---

# 22. TOTAL BUSINESS COST RECONCILIATION

Check:

module_total_business_costs_to_pnl_operating_expense_benchmark

source_amount =
operating_expense_benchmark_total or relevant P&L benchmark

module_amount =
module_total_business_costs

module_total_business_costs =
total_people_cost_annual
+ total_asset_cost_annual
+ total_general_overheads

If asset running cost transfers are included in Assets, ensure total_general_overheads is net of transferred amounts or transfer is otherwise removed from duplicate count.

Purpose:

Show whether the combined Mirra operating cost modules align with P&L operating expense evidence.

This is evidence only.

Do not force the totals to match.

# 26. BLOCKING CHECKS

blocking_checks[] should include reconciliation or source issues that prevent trusted downstream use.

Examples:

- P&L missing or blocked
- Labour source invalid
- Assets source invalid
- asset pool over-allocated
- asset running cost transfer over-allocated
- General Overheads invalid
- COG reconciliation leak detected
- total business cost cannot be calculated
- source period mismatch
- material unclassified P&L lines

Blocking check shape:

{
  check_id,
  source_module,
  severity,
  message,
  affected_variable,
  blocking,
  review_action
}

---

# 27. WARNING CHECKS

warning_checks[] should include visible issues that do not necessarily block all downstream output.

Examples:

- Labour variance vs P&L
- Asset finance variance vs P&L
- General Overheads variance vs P&L
- Other Income present
- asset utilisation defaults used
- non-productive assets excluded from recovery
- overhead pool under-allocated
- manual source used
- review-required but non-material items

Warnings must remain visible.

---

# 28. MODULE RECONCILIATION STATUS VALUES

Allowed module_reconciliation_status values:

ready
ready_with_warnings
review_required
blocked
leak_detected
not_ready

Status rules:

ready:
All required checks balanced or acceptable with no warnings.

ready_with_warnings:
No blocking checks, but warnings remain visible.

review_required:
Material variances require review but do not fully block.

blocked:
At least one blocking check exists.

leak_detected:
A source, allocation, or reconciliation mismatch may create hidden loss or double counting.

not_ready:
Required inputs are missing.

# 31. WARNING MEANINGS

## reconciliation_labour_variance

Use when Labour module amount differs from P&L labour benchmark.

Blocking:

false by default unless Model Readiness rules make it blocking.

## reconciliation_asset_finance_variance

Use when Assets finance/interest differs from P&L benchmark.

Blocking:

false by default unless material and unresolved.

## reconciliation_general_overheads_variance

Use when General Overheads differs from P&L benchmark.

Blocking:

review_required.

## reconciliation_cog_leak_detected

Use when Revenue / COGS COG import does not reconcile to P&L COG.

Blocking:

true for COG-dependent downstream modules.

## reconciliation_asset_transfer_double_count_risk

Use when transferred running costs appear in both Assets and full General Overheads.

Blocking:

true if unresolved.

## reconciliation_source_period_mismatch

Use when P&L and imported detail reports have different periods.

Blocking:

true for affected modules.

# 51. ACCEPTANCE CRITERIA

Module Reconciliation v4.0 is accepted when:
