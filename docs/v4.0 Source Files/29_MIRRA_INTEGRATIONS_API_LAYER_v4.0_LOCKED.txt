# Mirra — Integrations / API Layer Brief

## v4.0 — XERO, SOURCE IMPORTS, EXTERNAL DATA CONNECTORS, AUTHENTICATED SYNC, NORMALISED IMPORT CONTRACTS, AND NO-CALCULATION-OWNERSHIP BOUNDARY

STATUS: FUTURE BUILD BRIEF
VERSION: v4.0
FILE NUMBER: 29

REPLACES:
- older assumptions that all data must be manually imported
- older assumptions that Xero integration can write directly into downstream modules
- older logic where external imports bypass source validation
- older logic where connector data is trusted without reconciliation
- older logic where API imports become calculation owners

PRODUCT NAME:
Mirra

FORMER NAME:
QS Tools

PRIMARY RULE:
The Integrations / API Layer imports, normalises, validates, and syncs external data into Mirra source contracts. It does not own business calculations, classifications, model outputs, recovery logic, rate logic, quote checking, or business decisions.

---

# 1. PURPOSE

The Integrations / API Layer answers:

Where does external data come from?
How does Mirra connect to Xero?
How are P&L rows imported?
How are supplier bills imported?
How are invoices, credit notes, journals, payments, and contacts imported?
How are supplier names normalised?
How are external rows mapped to Mirra source contracts?
How does Mirra avoid duplicate imports?
How does Mirra preserve source trace?
How does Mirra know whether imported data reconciles?

This layer makes external data usable.

It does not decide what the data means commercially.

---

# 3. CORE PRINCIPLE

External data is source evidence.

It is not automatically trusted business logic.

The integration layer must:

- import
- normalise
- deduplicate
- preserve trace
- expose status
- hand data to the correct source module

It must not:

- classify final business outcomes
- calculate recovery
- build rates
- approve models
- mutate downstream results
- silently override user-reviewed mappings

---

# 10. SIGNAGE RULE

The integration layer must preserve debit / credit reality as a signed value.

Required fields:

- gross_amount
- net_amount
- gst_amount
- signed_amount
- source_document_type

Credit notes should reduce cost/revenue where appropriate.

Do not discard credit notes.

Do not convert all values to positive without a sign/source document type.

If source provides separate debit and credit columns, normalise into signed_amount while preserving original fields in raw payload.

---

# 11. GST RULE

GST must be preserved separately.

Required values where available:

- gross_amount
- net_amount
- gst_amount
- tax_type
- tax_rate

Source modules may decide whether to use gross or net.

For business modelling, values should usually use net of GST unless user/config says otherwise.

The integration layer must not hide GST.

---

# 14. CATEGORY MAPPING RULE

The integration layer may suggest mappings.

Suggested mappings may include:

- revenue account
- other/unallocated

# 16. PERIOD RULE

Every import should have a defined period.

Required fields:

- period_start
- period_end
- transaction_date

P&L reports use report period.

Transactions use transaction date.

Supplier bills use invoice/bill date.

Payment date may be stored separately but should not replace transaction date unless explicitly required.

---

# 17. RECONCILIATION HANDOFF RULE

The integration layer must provide enough information for source modules to reconcile.

Examples:

P&L total imported from Xero should reconcile to sum of relevant imported account rows where same basis exists.

Supplier bills by COG account should reconcile to Revenue / COG imported totals where possible.

If not reconcilable:

status = review_required

Do not hide mismatch.

# 28. IMPORT TRUST STATE

Integration layer must expose:

integration_trust_state

Allowed values:

- ready
- warning
- blocked
- preview_only

Mapping:

ready:
Connected/synced and no blocking import issues.

warning:
Imported but mapping/review/stale warnings exist.

blocked:
No source data, failed import, auth failure, or unreconciled critical import.

preview_only:
Mock, partial, or draft import.

---

# 29. STATUS VALUES

integration_status values:

- ready
- ready_with_warnings
- review_required
- blocked
- preview_only
- not_configured
- syncing

Definitions:

ready:
Imports available and trusted.

ready_with_warnings:
Imports available but warnings exist.

review_required:
Mappings or duplicates require review.

blocked:
Import failed or required source missing.

preview_only:
Mock or partial import only.

not_configured:
No connector configured.

syncing:
Import currently running.
