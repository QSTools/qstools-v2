# Mirra — Model Feedback Loop Brief

## v4.0 — ACTUAL PERFORMANCE FEEDBACK, RATE REVIEW SIGNALS, SUPPLIER PRESSURE FEEDBACK, MODEL UPDATE RECOMMENDATIONS, AND NO-AUTOMATIC-SOURCE-MUTATION BOUNDARY

STATUS: FUTURE BUILD BRIEF
VERSION: v4.0
FILE NUMBER: 24

REPLACES:
- older assumptions that job actuals automatically update the model
- older assumptions that rates should be overwritten from job results
- older logic that does not preserve a review step before model changes
- older logic where job costing feedback is disconnected from Business Outcome
- older logic where supplier, labour, asset, and rate issues are not fed back into the model

PRODUCT NAME:
Mirra

FORMER NAME:
QS Tools

PRIMARY RULE:
The Model Feedback Loop converts actual job performance into structured review signals for Rate Builder, Business Outcome, Labour, Assets, Revenue / COGS, and future modelling. It does not automatically overwrite approved model values.

---

# 1. PURPOSE

The Model Feedback Loop answers:

What did real job results teach us?
Which rates may need review?
Which suppliers are causing cost pressure?
Which material categories are consistently over allowance?
Which labour tasks are taking longer than expected?
Which assets are under-recovering?
Which overhead assumptions are not being recovered?
Which quote assumptions failed?
Which model assumptions should be reviewed?
What should be flagged before future quotes are checked?

This layer turns actual performance into structured improvement signals.

It is not an automatic correction engine.

It is a review and learning layer.

# 3. CORE PRINCIPLE

Actual job results should influence future model decisions, but they must not automatically rewrite the model.

A single bad job may be:

- bad pricing
- poor production
- supplier issue
- scope issue
- site issue
- one-off exception
- incorrect coding
- bad source data

Therefore every feedback signal must remain a review signal until accepted.

No automatic rate overwrite.

No automatic supplier adjustment.

No automatic labour productivity change.

No automatic asset utilisation change.

- info
- watch
- review
- high
- critical
- blocked

Useful signal, no immediate action.

Requires user review.

high:
Likely model impact.

blocked:
Source data cannot be trusted.

# 11. FEEDBACK STATUS VALUES

Allowed feedback_status values:

- pending_review
- accepted
- rejected
- deferred
- applied_to_scenario
- applied_to_rate_review
- ignored_as_one_off
- blocked

Rules:

pending_review:
Default state for feedback.

accepted:
User agrees signal should affect future model review.

rejected:
User decides signal is not valid.

deferred:
Signal is valid but not acted on yet.

applied_to_scenario:
Signal has been used in Business Outcome modelling.

applied_to_rate_review:
Signal has been used to create a Rate Builder review item.

ignored_as_one_off:
Signal is valid but treated as exceptional.

blocked:
Signal cannot be used due to bad source data.

# 19. QUOTE ALIGNMENT SIGNALS

Quote alignment signals identify whether failure began before or after quote approval.

# 20. REPEATED ISSUE DETECTION

Model Feedback Loop should identify repeated issues.

- same supplier
- same material category

# 27. DATA QUALITY FEEDBACK

Some feedback is about bad data, not business performance.

Examples:

- invoice not matched to PO
- labour hours missing
- asset hours missing
- quote line unmatched
- supplier unmapped
- zero quantity
- duplicate cost

Data quality feedback should be separated from commercial feedback.

Bad data should block or reduce trust.

Commercial failure should not be treated as bad data.

# 29. TRUST STATE

Model Feedback Loop must expose:

model_feedback_trust_state

Allowed values:

- ready
- warning
- blocked
- preview_only

Mapping:

If Job Costing is blocked:
model_feedback_trust_state = blocked

If source actuals are incomplete:
model_feedback_trust_state = warning or preview_only

If feedback is based on trusted actuals:
model_feedback_trust_state = ready

If feedback includes data quality issues:
model_feedback_trust_state = warning or blocked depending severity

blocked:
Feedback cannot be trusted due to missing or invalid source data.

preview_only:
Signals display but should not be treated as final.

not_ready:

# 51. FINAL PRINCIPLE

The Model Feedback Loop is the learning layer.

It turns actual job performance into structured evidence.

It tells the business:

which rates need review
which suppliers create pressure
which labour assumptions failed
which assets under-recovered
which overheads were missed
which quote checks failed
which model assumptions worked

It feeds Rate Builder and Business Outcome with review signals.

It never silently changes the model.

No automatic overwrites.
No source mutation.
No untraceable feedback.
No confusing bad performance with bad data.
No leaks.
