# Mirra — Data Quality Dashboard Brief

## v4.0 — SOURCE TRUST, MAPPING ISSUES, BLOCKING DATA PROBLEMS, WARNING CONTROL, AND NO-COMMERCIAL-RESULT-CONFUSION

STATUS: FUTURE BUILD BRIEF
VERSION: v4.0
FILE NUMBER: 26

REPLACES:
- older assumptions that negative commercial results are data failures
- older logic where data quality warnings are scattered across modules
- older logic where missing mappings are hidden inside each page
- older logic where blocked source data is not clearly separated from business underperformance
- older logic where users cannot see what must be fixed before trusting the model

PRODUCT NAME:
Mirra

FORMER NAME:
QS Tools

PRIMARY RULE:
The Data Quality Dashboard identifies source, mapping, reconciliation, missing-data, and trust issues across the whole Mirra system. It must clearly separate data problems from valid commercial results.

---

# 1. PURPOSE

The Data Quality Dashboard answers:

Can the model be trusted?
Which source data is missing?
Which mappings are incomplete?
Which modules are blocked?
Which warnings are commercial results?
Which warnings are actual data problems?
What must be fixed before downstream outputs can be trusted?
Where did the issue originate?
Which module owns the fix?

This page is not a business outcome page.

It is a trust and cleanup layer.

---

# 2. CORE PRINCIPLE

Data quality problems are not the same as bad business results.

Examples of data problems:

- missing P&L
- missing revenue
- missing COG mapping
- missing labour hours
- missing asset utilisation
- missing overhead source
- unmatched supplier
- duplicate cost
- failed reconciliation
- zero driver where driver is required
- quote line unmatched
- invoice not linked to PO

Examples of commercial results:

- margin is low
- labour consumes margin
- asset recovery is high
- rate is under-recovering
- quote fails target
- job loses money
- supplier costs increased

Commercial results should create warnings or outcome flags.

They should not automatically block the model if source data is trusted.

# 4. MODULE ROLE

Data Quality Dashboard owns:

- global warning aggregation
- source trust overview
- blocking issue list
- mapping issue list
- reconciliation issue list
- missing data list
- duplicate / double count risk list
- module readiness rollup
- issue owner mapping
- recommended fix location
- issue severity
- issue status
- data quality score
- global trust state

Data Quality Dashboard does not own:

- source imports
- categorisation edits
- labour calculations
- asset calculations
- overhead calculations
- cost allocation assignments
- recovery testing
- rate building
- business modelling
- quote checking
- job costing calculations
- model approvals

It points users to the correct module to fix issues.

# 8. SEVERITY VALUES

Allowed severity values:

- info
- warning
- review_required
- blocking
- critical

Definitions:

info:
Useful context, not a problem.

warning:
Needs attention but does not block.

review_required:
User should review before relying on output.

blocking:
Trusted downstream output must be blocked.

critical:
Major source trust failure or systemic issue.

---

# 9. ISSUE STATUS VALUES

Allowed issue_status values:

- open
- acknowledged
- ignored
- fixed
- deferred
- blocked
- resolved_by_upstream
- not_applicable

Rules:

open:
Default state.

acknowledged:
User has seen it but not fixed.

ignored:
User intentionally ignores non-blocking issue.

fixed:
Issue resolved.

deferred:
Valid issue but not addressed yet.

blocked:
Cannot resolve because required source is unavailable.

resolved_by_upstream:
Issue fixed by changing upstream module.

not_applicable:
Issue does not apply to current mode or setup.

---

# 10. COMMERCIAL RESULT FLAG

Every issue row must include:

commercial_result: true / false

If true:

The issue is not a data problem.

Examples:

- under-recovered rate
- negative net position
- low margin
- job loss
- quote margin shortfall

If false:

The issue is a data / mapping / trust problem.

This distinction is mandatory.

---

# 11. BLOCKING RULE

A warning blocks only when data cannot be trusted or a required calculation cannot run.

Blocking examples:

- missing P&L
- missing business mode
- failed source reconciliation
- over-allocated source pool
- zero required driver
- missing quote target
- missing job quantity
- duplicate cost included twice
- source trace missing for actual cost

Non-blocking examples:

- low margin
- high labour burden
- high asset pressure
- under-recovery
- supplier cost pressure
- job loss where actuals are trusted

# 13. GLOBAL TRUST STATE

Data Quality Dashboard must expose:

global_data_quality_trust_state

Allowed values:

- ready
- warning
- blocked
- preview_only

Mapping:

If any critical source module is blocked:
global_data_quality_trust_state = blocked

If no blocking issues but review issues exist:
global_data_quality_trust_state = warning

If source data is preview only:
global_data_quality_trust_state = preview_only

If no blocking/review issues:
global_data_quality_trust_state = ready

Commercial warnings alone should not make global trust blocked.

---

# 14. DATA QUALITY SCORE

The dashboard may calculate a data quality score.

data_quality_score =
0 to 100

Suggested weighting:

- blocking source issues reduce heavily
- missing mappings reduce moderately
- review-required issues reduce lightly
- commercial warnings do not reduce data quality score unless they also indicate missing data

This score is for UX only.

It must not replace trust-state logic.

# 29. FIX QUEUE PANEL

Show recommended order of fixing issues.

Example order:

1. Business Setup missing
2. P&L missing
3. Labour productive hours missing
4. Asset utilisation missing
5. Cost Allocation over-allocated
6. Rate Builder target missing
7. Quote Checker unmatched lines

This helps user avoid random fixing.

---

# 30. HELP PANEL

Explain:

- data quality issues are different from bad commercial outcomes
- blocked means calculations cannot be trusted
- warning means review but not necessarily wrong
- commercial result means the model found a real business issue
- fix issues in the source module shown
- the dashboard does not edit source data directly

---

# 31. WHAT TO KEEP OUT

Keep out:

- direct source editing
- rate editing
- business scenario modelling
- quote checking
- job costing
- model approval
- supplier recategorisation controls unless opened through source module link

The dashboard identifies and routes.

It does not fix directly.

When implementing:

1. Consume warning/status/trust outputs from all modules.
2. Normalise warnings into global_data_quality_issues.
3. Add commercial_result flag.
4. Separate commercial warnings from data problems.
5. Build module_trust_rollup.
6. Build blocking issue list.
7. Build review required list.
8. Build mapping issue list.
9. Build reconciliation issue list.
10. Build fix queue.
11. Add global trust state.
12. Add data quality score.
13. Add source trace display.
14. Do not mutate source modules.
15. Add warnings and trust state.
16. Run build.

---

# 34. ACCEPTANCE CRITERIA

Data Quality Dashboard v4.0 is accepted when:

1. It consumes all module warning/status outputs.
2. It normalises warnings into a global issue list.
3. It clearly separates data problems from commercial results.
4. It identifies blocking issues.
5. It identifies review-required issues.
6. It identifies commercial warnings.
7. It identifies mapping issues.
8. It identifies reconciliation issues.
9. It identifies missing source trace.
10. It points every issue to a fix module.
11. It creates a fix queue.
12. It calculates global trust state.
13. It does not block trusted output for valid negative commercial results.
14. It does not edit source data directly.
15. It exposes warnings and trust state.
16. Build passes.

---

# 35. FINAL PRINCIPLE

The Data Quality Dashboard protects trust.

It tells the user:

what is broken
what is only a warning
what is a real business problem
where to fix each issue

It prevents the system from confusing:

bad data

with

bad business results

No source mutation.
No hidden blockers.
No scattered warnings.
No confusing commercial losses with data failure.
No leaks.
