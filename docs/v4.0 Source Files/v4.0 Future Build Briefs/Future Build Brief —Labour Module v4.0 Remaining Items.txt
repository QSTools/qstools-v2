## 1. Purpose

This brief captures Labour Module items that are **not required for the current Labour v4.0 source-truth pass**, but should be considered for future builds.

The current Labour module now has the core v4.0 foundations:

* reusable user-controlled staff types
* custom staff type creation
* `staff_type_id`
* `staff_type_name`
* compatibility alias through `staff_role`
* productive / non-productive Labour class separation
* `productive_staff_type_rates[]`
* `weighted_all_productive_labour_rate`
* zero-hour / zero-cost / unassigned staff type warnings
* Productive Staff Type Rates panel

This brief prevents unfinished ideas from being lost while keeping the current Labour build from expanding beyond its correct source-module role.

## 4. Future Item A — Staff Type Management Panel

### Purpose

Create a dedicated Staff Type Management panel so users can manage staff types separately from creating or editing individual staff.

### Current state

Staff types can be created from the Labour profile/staff form.

### Future improvement

Add a panel that allows users to:

* view all staff types
* create staff types
* rename custom staff types
* deactivate custom staff types
* see whether a staff type is system-created or user-created
* see how many staff are currently assigned to each staff type

### Rules

* Do not allow deleting a staff type currently used by staff.
* Do not use `staff_type_name` as a key.
* Do not infer productivity from staff type.
* Labour class remains the productive / non-productive control.
* System default staff types may be renamed only if a migration/alias strategy is defined.

### Likely files

* `components/labour/StaffTypeManagementPanel.jsx`
* `hooks/useLabour.js`
* `lib/storage/labourStorage.js`
* `lib/selectors/labourSelectors.js`
* `app/labour/page.jsx`

## 5. Future Item B — Staff Type Rename / Migration Handling

### Purpose

Allow safe renaming of custom staff types without breaking historical staff records or downstream grouping.

### Required behaviour

When a custom staff type is renamed:

* keep the same `staff_type_id`
* update `staff_type_name`
* update display only
* preserve historical staff records where required
* avoid creating duplicate staff type IDs
* avoid merging similar names automatically

### Rules

* `staff_type_id` remains stable.
* `staff_type_name` may change.
* Do not rewrite historical accepted snapshots unless explicitly approved by a future snapshot/migration brief.

## 15. Future Acceptance Criteria

A future Labour enhancement is acceptable only if:

1. Labour remains a source module.
2. Staff identity remains system-controlled and hidden.
3. Staff type identity remains system-controlled and hidden.
4. User-facing names remain editable/display only.
5. Productive / non-productive classification remains explicit.
6. Productive staff type rates remain weighted by productive hours.
7. Labour does not create allocation, recovery, rate, or outcome logic.
8. Downstream modules consume Labour outputs instead of rebuilding them.
9. Warnings are structured and visible.
10. Build passes.

Do not continue expanding Labour unless the improvement strengthens Labour as a source-truth module.

The next active build should move downstream to Cost Allocation cleanup.
