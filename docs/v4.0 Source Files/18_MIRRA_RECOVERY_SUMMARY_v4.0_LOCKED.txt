PRIMARY RULE:
Recovery Summary is a zero-input, automatic, mode-dependent recovery test layer. It tests whether the allocated operating structure can recover its cost burden. It does not create source costs, assign costs, ask the user to choose recovery method, build sell rates, or issue the final business verdict.

---

# 1. PURPOSE

Recovery Summary is the automatic recovery testing layer.

It answers:

Does the allocated business structure recover its cost burden?
Does each working group recover its assigned labour, productive assets, and overhead burden?
Is the business recovering by hours, product/unit, or mixed group logic?
Where is the recovery shortfall or surplus?
Which groups are under-recovered?
Which costs remain unassigned or unrecovered?
Does the recovery model respect Revenue Reality and Asset Reality pressure signals?

Recovery Summary does not build the operating structure.

Cost Allocation builds the structure.

Recovery Summary tests it automatically.

Rate Builder turns recovery requirements into sellable rates.

Business Outcome explains the final consequence.

# 4. CORE PRINCIPLE

Recovery Summary answers one question:

Does this structure recover?

It is not a setup page.

It is not an allocation page.

It is not a pricing page.

It is not an outcome page.

The user should not need to input anything on this page.

Recovery Summary should calculate automatically from upstream mode, groups, drivers, and source values.

---

# 5. ZERO USER INPUT RULE

Recovery Summary has no user inputs.

The user must not be asked to:

- choose recovery method
- choose recovery basis
- choose hybrid mode
- enter recovery driver
- enter hours
- enter units
- enter sell rates
- enter margin
- create groups
- assign labour
- assign assets
- distribute overheads

All required setup must come from upstream modules.

If required source values are missing, Recovery Summary should show:

not_available
review_required
or blocked

It must not ask the user to fix the missing source inside Recovery Summary.

The review action should send the user back to the correct upstream module.

---

# 6. MODE-DEPENDENT RULE

Recovery Summary is controlled by Business Setup mode.

Business Setup owns business mode.

Recovery Summary must consume:

business_type
is_labour_based
is_product_based

If business_type = labour_based:

Recovery Summary uses hours-based recovery logic.

If business_type = product_based:

Recovery Summary uses product/unit-based recovery logic.

If Cost Allocation groups contain different valid recovery drivers:

Recovery Summary may display mixed group recovery.

Do not ask the user to choose the recovery mode here.

Do not infer the business mode from group labels, supplier names, asset names, or account names.

# 16. HOURS-BASED RECOVERY INTERPRETATION

In labour-based mode, Recovery Summary automatically tests whether available productive hour recovery can cover:

- assigned labour burden
- assigned productive asset burden
- assigned overhead burden

Group recovery basis may come from:

- assigned labour hours
- assigned asset hours
- productive hours from Labour
- group driver values from Cost Allocation
- future production/quote hours if supplied upstream

Hours-based recovery must show:

- required recovery per hour
- group cost per hour where driver exists
- recovery gap if recoverable value exists
- not_available where actual recoverable value is not known

Recovery Summary does not create final hourly rates.

Rate Builder does.

---

# 17. PRODUCT / UNIT-BASED RECOVERY INTERPRETATION

In product/unit mode, Recovery Summary automatically tests whether product/unit margin and assigned group structure can recover:

- assigned labour burden
- assigned productive asset burden
- assigned overhead burden
- COG burden where mapped later

Unit-based recovery basis may come from:

- unit_driver_rows from Revenue / COGS
- group unit drivers from Cost Allocation
- future quote/production data if supplied upstream

Recovery Summary may test whether current unit margin can carry assigned burden.

It must not infer product consumption without user input upstream.

It must not ask for new unit input on this page.

Rate Builder owns final product/unit rate construction.

---

# 18. MIXED GROUP RECOVERY INTERPRETATION

Mixed group recovery applies when different working groups recover through different drivers.

Example:

Group A:
Digger Crew → hourly

Group B:
Concrete Pumping → m3

Group C:
Kitchen → meals / covers

Recovery Summary must allow group-level recovery display where valid group drivers already exist.

It should not force the whole business into one recovery driver if upstream structure supports multiple valid drivers.

It must not ask the user to choose those drivers here.

# 22. REQUIRED RECOVERY AMOUNT

Recovery Summary must calculate:

required_recovery_amount =
total_group_cost

At group level.

Also:

total_required_recovery_amount =
sum group total_group_cost

This is the cost burden that must be recovered.

This is not the sell rate.

This is the amount that Rate Builder must cover.

---

# 23. REQUIRED RECOVERY PER DRIVER

Recovery Summary may calculate required recovery per driver:

required_recovery_per_driver =
total_group_cost / recovery_driver_value

If recovery_driver_value = 0:

required_recovery_per_driver = 0
status = review_required or not_available

This is a required recovery signal only.

It is not a final Rate Builder sell rate.

Rate Builder owns final rate construction.

# 25. RECOVERY TEST FORMULA

Where recoverable_value_available exists:

recovered_cost =
recoverable_value_available

unrecovered_cost =
max(0, total_group_cost - recovered_cost)

recovery_surplus =
max(0, recovered_cost - total_group_cost)

recovery_coverage_percent =
recovered_cost / total_group_cost

If total_group_cost = 0:

recovery_coverage_percent = 0
status = review_required or not_available

If recoverable_value_available is not available:

recovered_cost = not_available
unrecovered_cost = not_available or total_group_cost depending display choice
recovery_coverage_percent = not_available
status = not_available or review_required

Do not divide by zero.

Do not invent recovered values.

# 28. RECOVERY STATUS VALUES

blocked
preview_only

Status rules:

ready:
Recovery test can run and no blocking warnings exist.

covered:
Recovery test shows costs are covered.

under_recovered:
Recovery test shows unrecovered cost remains, but source data is trusted.

ready_with_warnings:
Recovery test is usable but warnings remain visible.

review_required:

blocked:
Upstream source or allocation is blocked.

preview_only:
Values display but should not be trusted.

not_available:
Required actual recovery value is not yet available.

not_ready:
Required recovery inputs are missing.

Under-recovered is not a data failure if source data is trusted.

Not_available is not a data failure if Rate Builder has not yet created rates.

# 30. READINESS CONDITIONS

Recovery Summary is ready when:

- Business Setup mode exists
- Revenue / COGS context exists
- Business Summary context exists
- Cost Allocation groups exist
- group cost stacks exist
- Cost Allocation source pools are not over-allocated
- required recovery drivers exist upstream where required
- Model Readiness is not blocked
- no blocking Recovery Summary warning exists

Recovery Summary is blocked when:

- Cost Allocation is blocked
- source pool reconciliation failed
- group cost stack is invalid
- required upstream driver is missing where required
- recovery calculation cannot be performed
- Model Readiness blocks trusted output

Recovery Summary may be ready_with_warnings when:

- groups are under-recovered
- Revenue Reality shows labour/margin pressure
- Asset Reality shows asset pressure
- overheads were under-allocated but accepted for review
- unassigned pools remain but are visible and intentionally reviewed

Recovery Summary may be not_available when:

- required recovery amounts can be shown
- but actual recoverable value does not exist yet because Rate Builder has not built rates

# 32. WARNING MEANINGS

## recovery_summary_cost_allocation_blocked

Use when Cost Allocation is blocked or leaking.

Blocking:

true.

## recovery_summary_missing_upstream_recovery_driver

Use when a group has cost but no upstream recovery driver.

Blocking:

true for trusted recovery result.

Review action:

Return to Business Setup, Revenue / COGS, Labour, Assets, or Cost Allocation depending missing source.

## recovery_summary_zero_recovery_driver

Use when a recovery driver is zero.

Blocking:

true for per-driver recovery result.

## recovery_summary_no_recoverable_value

Use when required recovery can be calculated but actual recovered value is unavailable.

Blocking:

false.

Meaning:

Rate Builder or future revenue/rate source has not yet provided recoverable value.

## recovery_summary_group_under_recovered

Use when one group is under-recovered.

Blocking:

false if source data is trusted.

## recovery_summary_model_under_recovered

Use when total model recovery is under-recovered.

Blocking:

false if source data is trusted.

## recovery_summary_revenue_reality_pressure

Use when Revenue Reality shows labour/margin pressure.

Blocking:

false if source data is trusted.

## recovery_summary_asset_reality_pressure

Use when Asset Reality shows asset pressure.

Blocking:

false if source data is trusted.

## recovery_summary_unassigned_labour_pool

Use when productive labour remains unassigned in Cost Allocation.

Blocking:

false by default if visible and intentionally reviewed; true if workflow requires full allocation.

## recovery_summary_unassigned_asset_pool

Use when productive asset burden remains unassigned.

Blocking:

false by default if visible and intentionally reviewed; true if workflow requires full allocation.

## recovery_summary_unassigned_overhead_pool

Use when overhead remains unassigned.

Blocking:

false by default if visible and intentionally reviewed; true if workflow requires full allocation.
