Simple in hindsight.

## CORE SYSTEM PHILOSOPHY

QS Tools uses historical accounting data to build a forward-looking commercial model.

The P&L shows what happened.

QS Tools uses that information to classify, reconcile, and model what the business needs to recover going forward.

Past accounting data is input evidence.

Forward-looking commercial cost is the decision basis.

The system must not blindly price from historical P&L values when a specialised module owns a clearer forward-looking cost.

## COST OWNERSHIP RULE

Every cost must have one owner.

No duplicate cost ownership is allowed.

Examples:

Salary & Wages → Labour

Employer KiwiSaver → Labour

ESCT → Labour

Employer ACC Levy → Labour

Staff Expenses → General Overheads / Staff Overheads category

Entertainment → General Overheads / Staff Overheads category

Vehicle fuel / servicing / repairs / tyres / registration → General Overheads / Vehicle Running Costs

Vehicle finance / asset ownership → Assets

Asset finance interest → Assets

Paid-off asset finance → $0 current finance cost

Non-asset interest / overdraft / working capital interest → General Overheads

P&L interest line → historical / reported context and reconciliation evidence only where it includes asset finance

Principal repayments → Cash Flow later, not Cost Summary

WIP Adjustment → Review Required until explicitly resolved

Penalties / Non-Deductible → Excluded unless explicitly included

## P&L INTEREST TREATMENT RULE

P&L interest treatment is factual metadata only.

Allowed states:

- not_reviewed
- contains_asset_finance_interest
- no_asset_finance_interest

Legacy mappings:

- unknown → not_reviewed
- asset_finance_exclude → contains_asset_finance_interest
- general_overhead_keep → no_asset_finance_interest

Selecting contains_asset_finance_interest must not:

- delete the line
- exclude the line
- zero the line
- unwire the line
- remove the line from classification visibility

However, selecting contains_asset_finance_interest also must not make P&L the authority for asset finance interest.

Asset finance interest belongs to Assets.

P&L interest is historical / reported context.

The P&L interest value may support reconciliation or classification of non-asset interest, but it must not override the forward-looking asset interest calculated by Assets.

## MODEL READINESS RULE

The user must never accidentally trust an incomplete model.

If the model is incomplete, QS Tools must say so clearly.

If a number is preliminary, QS Tools must say so clearly.

If a cost is excluded, QS Tools must keep it visible.

If a material item is unresolved, QS Tools must block readiness.

If a model is ready, the user should have confidence that:

- source data has been classified
- module ownership is clean
- reconciliation has been checked
- downstream outputs are safe to use

Readiness formula:

model_ready =
pnl_ready
AND general_overheads_ready
AND labour_ready
AND assets_ready
AND reconciliation_ready
AND blocking_checks.length = 0

Status formula:

if blocking_checks.length > 0:
  model_readiness_status = "blocked"

else if warning_checks.length > 0:
  model_readiness_status = "warning"

else:
  model_readiness_status = "ready"

Business benchmark variance may create warning status.

Business benchmark variance must not block readiness when all source modules are valid.

## LABOUR RECOVERY VISIBILITY

QS Tools must expose baseline labour recovery before quote-level analysis exists.

Labour recovery is expected to be one of the main hidden leaks in fixed-price / quote-based businesses.

The summary layers should eventually show whether the model actually recovers the Labour module’s cost burden over productive hours.

The system must distinguish:

- revenue
- external/direct costs
- subcontract labour
- internal labour recovery
- general overhead recovery
- asset ownership recovery
- target profit

The system must not simply assume:

Revenue - COGS = Labour content

That is too broad and can hide the real labour recovery issue.

The defensible baseline concept is:

Revenue
- external/direct costs
= margin / recovery pool

Then:

margin / recovery pool
- required general overhead recovery
- required asset ownership recovery
- target profit
= revenue available for internal labour recovery

Then:

effective_internal_labour_recovery_rate =
revenue_available_for_internal_labour_recovery
/ total_productive_output

This should be compared against:

required_labour_recovery_rate =
total_labour_cost_annual
/ total_productive_output

Quote Benchmark later proves this quote-by-quote.

## MARGIN POOL RECOVERY RESULT

For fixed-price and quote-based businesses, QS Tools must not treat gross margin as proof of profit.

A quote must generate enough margin / recovery pool to recover:

- internal labour cost
- general overheads
- asset ownership costs
- target profit

The core fixed-price test is:

Revenue
- external/direct costs
= margin / recovery pool

Then:

margin / recovery pool
- required internal labour recovery
- required general overhead recovery
- required asset ownership recovery
- target profit
= recovery surplus / shortfall

If the recovery surplus / shortfall is negative, the quote is commercially loss-making against the model, even if it appears profitable using a simple gross-margin view.

The product should eventually surface this as a clear commercial result:

- recovering
- under-recovering
- commercially loss-making against the model

This is a core QS Tools insight and must be protected in future builds.

---

## SUBCONTRACT LABOUR VISIBILITY

Subcontract labour must be visible separately from internal labour recovery.

Subcontract labour may appear in COGS, but it is not internal productive labour capacity.

For labour recovery analysis, subcontract labour should normally be treated as external labour cost unless explicitly reclassified.

Future quote/revenue analysis should distinguish:

- internal labour recovery
- subcontract labour cost
- total labour content
- internal vs external labour mix

This prevents QS Tools from overstating internal labour recovery when the work is actually being delivered by subcontractors.

## FUTURE XERO API RULE

Xero API integration should come after the core engine is stable.

The Xero API should be used to import P&L/accounting data into an already-defined QS Tools classification structure.

The API should not define the commercial model.

The commercial model defines how imported Xero lines are treated.

Future Xero flow:

Xero P&L line
→ saved QS Tools mapping
→ category
→ review subcategory
→ treatment
→ readiness state

If a Xero line is new, changed, or unmapped, QS Tools must flag it for review.

The API makes the workflow faster.

It does not replace first-principles classification.

## BUILD DISCIPLINE

Every build should follow this sequence:

1. first-principles rule
2. source brief update
3. narrow implementation task
4. build passes
5. visual / workflow check
6. commit
7. move to the next small piece

Do not make broad changes when a narrow change is enough.

Do not hide a conceptual problem with code.

Do not allow convenience to override commercial correctness.

Do not run Codex against unclear behaviour.

If the rule is not written, update the brief before coding.

---

## FINAL STANDARD

QS Tools is successful when the user can trust the model because every cost, assumption, treatment, variance, and downstream number can be explained.

The product goal is not to produce a number quickly.

The product goal is to produce a number that can be defended.

The long-term product goal is to connect baseline, quote behaviour, accepted work, and actual monthly P&L performance so the business can see what is happening commercially at any given moment.

QS Tools must show whether the model and future fixed-price quotes actually recover the business model.

The system must not allow a quote or model to look commercially healthy merely because gross margin is positive.

The key question is:

Does the margin / recovery pool recover internal labour, general overheads, asset ownership, and target profit?

If not, QS Tools must make the shortfall visible.
