## MODULE OWNERSHIP SUMMARY

### P&L owns

- raw P&L lines
- P&L classification
- P&L benchmark totals
- review-required status
- excluded status
- WIP / accounting adjustment visibility
- reusable P&L mapping over time

P&L does not own downstream source-of-truth cost totals.

### General Overheads owns

- staff overheads as a category
- vehicle running costs
- office/admin costs
- finance/admin overheads
- insurance/compliance
- sales/growth overheads
- travel overheads
- total_general_overheads

General Overheads does not own Labour employer obligations.

General Overheads does not own asset ownership costs.

### Labour owns

- staff_id
- labour calculations
- salary/wages
- entitlement cost
- employer KiwiSaver
- ESCT
- employer ACC levy
- total_labour_cost_annual
- total_productive_output

Labour does not own staff overheads.

### Assets owns

- asset_id
- ownership-only asset cost calculations
- finance cost
- lease cost
- depreciation cost
- replacement/ownership recovery if explicitly enabled
- total_asset_cost_annual
- no_active_assets_confirmed

Assets does not own vehicle running costs.

Assets must distinguish active asset from active finance.

Paid-off finance must not continue creating current finance cost.

### Module Reconciliation owns

- evidence checks
- cross-module consistency
- benchmark variance visibility
- blocker/warning generation
- reconciliation_ready

Module Reconciliation does not recalculate upstream modules.

### Model Readiness owns

- model_ready
- model_readiness_status
- blocking modules
- warning modules
- final trust verdict

Model Readiness does not create cost totals.

### Cost Summary owns

- visible cost summary presentation
- trusted / warning / blocked display
- consumption of upstream outputs

Cost Summary must not rebuild upstream maths.

## DOWNSTREAM CONTRACT RULE

Upstream modules own inputs and calculations.

Downstream modules consume outputs only.

Never rebuild upstream maths downstream.

Never read raw upstream form state in downstream modules unless the relevant brief explicitly allows it.

Never rename locked contract variables without a matching source brief update.

Never use display names as join keys.

Use IDs only for joins.

If a downstream module needs a value, the upstream module must expose it through its output contract.

---

## COST OWNERSHIP RULE

Every cost must have one owner.

No duplicate cost ownership is allowed.

Examples:

```text
Salary & Wages → Labour
Employer KiwiSaver → Labour
ESCT → Labour
Employer ACC Levy → Labour
Staff Expenses → General Overheads / Staff Overheads
Entertainment → General Overheads / Staff Overheads
Vehicle fuel/servicing/repairs/tyres/rego → General Overheads / Vehicle Running Costs
Vehicle finance / asset finance → Assets
Paid-off asset finance → $0 current finance cost
WIP Adjustment → Review Required until explicitly resolved
Penalties / Non-Deductible → Excluded unless explicitly included
```

## P&L CLASSIFICATION RULE

P&L is the first classification layer.

P&L must distinguish:

```text
accounting section
QS top-level category
review subcategory
treatment
readiness impact
```

Operating Expenses is an accounting section.

General Overheads is a QS category.

Operating Expenses may contain:

- Labour
- General Overheads
- Assets
- Review Required
- Excluded
- Unassigned

The system must not force all Operating Expenses into General Overheads.

---

## WIP / ACCOUNTING ADJUSTMENT RULE

WIP Adjustment must not be automatically assigned to:

- COGS
- General Overheads
- Labour
- Assets
- Income
- Excluded

WIP Adjustment must default to:

```text
category: review_required
review_subcategory: wip_accounting_adjustment
```

WIP remains a blocker until explicitly resolved.

Allowed WIP treatments:

```text
Leave as Review Required
Exclude from QS Cost Model
Include as COGS / Direct Job Cost
Treat as Income / Revenue Timing Adjustment
```

Do not auto-resolve WIP.

Do not silently exclude WIP.

Do not silently include WIP in COGS.

## PRODUCT STANDARD RULE

Every important output must survive challenge.

If someone questions a number, QS Tools must be able to show:

- where the number came from
- what source line or module produced it
- what category owns it
- what assumptions were made
- whether the user manually resolved it
- whether it is included or excluded
- whether it blocks readiness
- what downstream output it affects

If the system cannot explain a number, the number should not be trusted.

## LONG-TERM COMMERCIAL CONTROL LOOP

The current priority is to build the baseline model correctly.

Future monthly P&L import, Xero API integration, accepted quote tracking, and actual-vs-model performance analysis must be built only after the baseline model is commercially defensible.

The long-term QS Tools loop is:

```text
Baseline setup
→ trusted cost model
→ quote benchmark
→ quote accepted / rejected
→ monthly P&L import
→ actual performance vs model
→ variance analysis
→ better future pricing decisions
```

The first P&L setup creates the reusable business mapping layer that future monthly imports and Xero API integration will rely on.

A weak baseline creates weak monthly analysis.

A trusted baseline creates meaningful commercial control.

---

## REUSABLE P&L MAPPING PRINCIPLE

The first P&L setup may take time because QS Tools is making the business’s P&L structure explicit.

This setup effort is intentional.

Once a P&L line has been classified and resolved, QS Tools should be able to reuse that treatment for future P&L periods.

The reusable mapping layer should connect:

```text
P&L account / Xero account / report line
→ QS category
→ review subcategory
→ treatment
→ owning module
→ readiness impact
```

Future monthly P&L entry or Xero API import should become a controlled update rather than a full setup exercise.

Known mapped lines should map automatically.

New, changed, or unresolved lines should be highlighted for review.

The system must not automate a weak baseline.

---

## FUTURE MONTHLY PERFORMANCE LOOP

Once the baseline model is trusted, QS Tools should eventually compare:

```text
baseline model
```

against:

```text
monthly actual P&L performance
```

The system should show whether the business is performing in line with the model it set.

Future monthly analysis should answer:

- did actual sales align with the required recovery model?
- did actual labour cost align with the baseline?
- did actual general overheads align with the baseline?
- did actual asset costs align with the baseline?
- did actual gross margin align with the quote model?
- did accepted quotes produce the expected recovery?
- where did actual performance drift from the model?

The user does not need perfect job-level tracking to see whether the business model is working.

Job-level tracking may be useful later, but the first powerful layer is business-level model control.

## LABOUR RECOVERY VISIBILITY

QS Tools must expose baseline labour recovery before quote-level analysis exists.

Labour recovery is expected to be one of the main hidden leaks in fixed-price / quote-based businesses.

The summary layers should eventually show whether the model actually recovers the Labour module’s cost burden over productive hours.

The system must distinguish:

- revenue
- external/direct costs
- subcontract labour
- internal labour recovery
- general overhead recovery
- asset ownership recovery
- target profit

The system must not simply assume:

```text
Revenue - COGS = Labour content
```

That is too broad and can hide the real labour recovery issue.

The defensible baseline concept is:

```text
Revenue
- external/direct costs
= margin / recovery pool
```

Then:

```text
margin / recovery pool
- required general overhead recovery
- required asset ownership recovery
- target profit
= revenue available for internal labour recovery
```

Then:

```text
effective_internal_labour_recovery_rate =
revenue_available_for_internal_labour_recovery
/ total_productive_output
```

This should be compared against:

```text
required_labour_recovery_rate =
total_labour_cost_annual
/ total_productive_output
```

Quote Benchmark later proves this quote-by-quote.

---

## MARGIN POOL RECOVERY RESULT

For fixed-price and quote-based businesses, QS Tools must not treat gross margin as proof of profit.

A quote must generate enough margin / recovery pool to recover:

- internal labour cost
- general overheads
- asset ownership costs
- target profit

The core fixed-price test is:

```text
Revenue
- external/direct costs
= margin / recovery pool
```

Then:

```text
margin / recovery pool
- required internal labour recovery
- required general overhead recovery
- required asset ownership recovery
- target profit
= recovery surplus / shortfall
```

If the recovery surplus / shortfall is negative, the quote is commercially loss-making against the model, even if it appears profitable using a simple gross-margin view.

The product should eventually surface this as a clear commercial result:

- recovering
- under-recovering
- commercially loss-making against the model

This is a core QS Tools insight and must be protected in future builds.

---

## SUBCONTRACT LABOUR VISIBILITY

Subcontract labour must be visible separately from internal labour recovery.

Subcontract labour may appear in COGS, but it is not internal productive labour capacity.

For labour recovery analysis, subcontract labour should normally be treated as external labour cost unless explicitly reclassified.

Future quote/revenue analysis should distinguish:

- internal labour recovery
- subcontract labour cost
- total labour content
- internal vs external labour mix

This prevents QS Tools from overstating internal labour recovery when the work is actually being delivered by subcontractors.

---

## FUTURE XERO API RULE

Xero API integration should come after the core engine is stable.

The Xero API should be used to import P&L/accounting data into an already-defined QS Tools classification structure.

The API should not define the commercial model.

The commercial model defines how imported Xero lines are treated.

Future Xero flow:

```text
Xero P&L line
→ saved QS Tools mapping
→ category
→ review subcategory
→ treatment
→ readiness state
```

If a Xero line is new, changed, or unmapped, QS Tools must flag it for review.

The API makes the workflow faster.

It does not replace first-principles classification.

---

## MODEL READINESS RULE

The user must never accidentally trust an incomplete model.

If the model is incomplete, QS Tools must say so clearly.

If a number is preliminary, QS Tools must say so clearly.

If a cost is excluded, QS Tools must keep it visible.

If a material item is unresolved, QS Tools must block readiness.

If a model is ready, the user should have confidence that:

- source data has been classified
- module ownership is clean
- reconciliation has been checked
- downstream outputs are safe to use

## FINAL STANDARD

QS Tools is successful when the user can trust the model because every cost, assumption, treatment, variance, and downstream number can be explained.

The product goal is not to produce a number quickly.

The product goal is to produce a number that can be defended.

The long-term product goal is to connect baseline, quote behaviour, accepted work, and actual monthly P&L performance so the business can see what is happening commercially at any given moment.

QS Tools must show whether the model and future fixed-price quotes actually recover the business model.

The system must not allow a quote or model to look commercially healthy merely because gross margin is positive.

The key question is:

```text
Does the margin / recovery pool recover internal labour, general overheads, asset ownership, and target profit?
```

If not, QS Tools must make the shortfall visible.
