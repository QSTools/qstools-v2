# 1. P&L INTEREST TREATMENT

## Problem Solved

The previous P&L interest logic used outcome-driven wording such as:

- Contains asset finance — exclude from QS cost
- Pure general finance — keep in QS cost

This was incorrect because it gave the user an apparent permission path to exclude costs from the model.

## Locked Rule

P&L interest treatment is factual metadata only.

The user classifies whether the P&L interest line contains asset finance interest.

Allowed states:

- not_reviewed
- contains_asset_finance_interest
- no_asset_finance_interest

Legacy compatibility:

- unknown → not_reviewed
- asset_finance_exclude → contains_asset_finance_interest
- general_overhead_keep → no_asset_finance_interest

## Critical Behaviour

Selecting contains_asset_finance_interest must NOT:

- delete the line
- exclude the line
- zero the line
- unwire the line
- remove the line from classification visibility

However, selecting contains_asset_finance_interest must also NOT make P&L the source of truth for asset finance interest.

P&L interest is historical / reported accounting context.

Asset finance interest belongs to Assets.

# 2. GENERAL OVERHEADS SYNC

## Problem Solved

Interest lines marked as containing asset finance were previously being removed or unwired from General Overheads.

That was incorrect because the user still needs visibility of the reported P&L interest line.

## Updated Locked Rule

General Overheads owns non-asset overhead costs.

General Overheads owns:

- non-asset interest
- overdraft interest
- working capital interest
- bank interest
- miscellaneous finance costs

General Overheads does NOT own:

- asset finance interest
- asset ownership cost
- asset principal repayments

If P&L interest includes asset finance, the P&L value remains visible as reported historical context.

It must not become the authority for asset finance interest.

The system must preserve the distinction between:

- P&L reported interest
- asset finance interest from Assets
- non-asset / other interest in General Overheads where available

## Current Behaviour Intent

P&L interest lines may sync into General Overheads visibility with metadata including:

- interest_treatment
- contains_asset_finance_interest
- source_pnl_line_id
- source_line_name
- source_category
- source_review_subcategory

But General Overheads must not duplicate asset finance interest when Assets owns it.

# 3. ASSETS FINANCE LIFECYCLE

## Problem Solved

Assets previously calculated finance cost without properly accounting for paid-off or term-ended finance.

## Locked Rule

Active asset does not always mean active finance.

The asset may remain active while finance is:

- active
- paid off
- term ended
- not financed

## Finance Fields

The Assets module supports or should support:

- finance_start_date
- finance_paid_off
- finance_end_date
- finance_active
- finance_status
- purchase_date
- finance_term_years
- interest_rate
- loan_amount / financed_amount

Effective From remains profile/version metadata only.

It must NOT be used as the finance start date.

## Calculation Rule

If finance_paid_off is true:

- finance_cost_annual = 0
- asset_interest_annual = 0
- finance_active = false

If the finance term has ended:

- finance_cost_annual = 0
- asset_interest_annual = 0
- finance_active = false

If finance is active:

- asset finance interest should be calculated from the current / forward-looking asset finance position
- paid-off or term-ended finance must not keep creating current finance cost

# 4. FORWARD-LOOKING ASSET INTEREST

## New Locked Rule

Assets is the source of truth for asset finance interest.

Asset interest must be calculated from the Assets module, not from the P&L.

The P&L may contain historical or accountant-journaled interest values, but those values must not be treated as the source of truth for asset finance.

The purpose is to estimate the asset finance interest the business needs to recover going forward, not to reconstruct the exact historical interest charged in the previous P&L period.

## Accepted v3.6 Approximation

The system does not need a full amortisation schedule in v3.6.

For each active financed asset, the accepted simple model is:

- calculate elapsed time since purchase / finance start
- calculate progress through the finance term
- estimate remaining balance
- calculate current annual interest from estimated remaining balance

Simple model:

remaining_term_percent =
1 - finance_progress_percent

estimated_remaining_finance_balance =
loan_amount × remaining_term_percent

asset_interest_annual =
estimated_remaining_finance_balance × interest_rate

This is an approximation only.

It is good enough for forward-looking Cost Summary unless a later locked Cash Flow / finance schedule brief requires a full amortisation model.

# 5. ASSETS READINESS

## Problem Solved

Assets readiness was previously blocked by P&L benchmark reconciliation.

The old logic effectively required:

assets_ready = reconciliation_state === "green"

This was incorrect.

## Locked Rule

Assets readiness is based on ownership record validity, not P&L benchmark matching.

P&L benchmark variance remains diagnostic only.

## Readiness Passes When

Assets readiness passes when either:

### Active Assets Path

- at least one active saved asset exists
- active asset records have required identity/setup fields
- asset is active and not retired
- total_asset_cost_annual is a valid number, including 0
- ownership-only rules are followed
- running costs are excluded
- asset finance interest is owned by Assets where finance applies
- principal repayments do not feed total_asset_cost_annual

A zero cost is valid where finance is paid off, term-ended, not financed, or no ownership cost is currently active.

### No Active Assets Path

- no active saved assets exist
- no_active_assets_confirmed is true
- total_asset_cost_annual is 0

## Readiness Must Fail When

- no active assets exist and no_active_assets_confirmed is false
- active asset records are incomplete or invalid
- running_cost_annual feeds Assets
- vehicle running costs feed Assets
- principal repayments feed total_asset_cost_annual
- asset finance interest is duplicated in Assets and General Overheads

## Benchmark Variance

P&L asset benchmark reconciliation remains visible through:

- reconciliation_state
- variance amount
- variance percent
- diagnostic warning text

It must not block Assets readiness when the Assets module itself is valid.

# 6. ASSET INTEREST / PRINCIPAL OUTPUTS

## Problem Solved

Assets calculated interest_annual per asset but did not always preserve or aggregate it.

This was needed for later Cost Summary transparency and finance / interest explanation.

## Locked Rule

Assets must preserve and expose:

- full asset ownership cost
- asset interest-only amounts

## Outputs

Saved asset records may preserve:

- asset_interest_annual
- finance_cost_annual
- finance_active
- finance_status
- finance_start_date
- finance_end_date
- finance_paid_off
- estimated_remaining_finance_balance
- finance_progress_percent

Assets output contract should expose:

- total_asset_cost_annual
- total_asset_interest_annual
- assets_ready
- no_active_assets_confirmed

## Principal Treatment

Principal repayments are real cash pressure but are not operating cost.

Principal repayments must not feed:

- total_asset_cost_annual
- total_general_overheads
- total_cost_burden
- required_recovery_rate
- Cost Summary

Principal belongs in the future Cash Flow layer.

Assets may later expose principal-related outputs for Cash Flow, such as:

- asset_principal_repayment_annual
- asset_principal_repayment_monthly
- asset_total_finance_payment_annual
- asset_total_finance_payment_monthly

These are future Cash Flow support outputs only.

They must not feed Cost Summary.

# 7. BUSINESS COST VARIANCE

## Problem Solved

Business Cost Variance previously blocked Model Readiness when module totals differed from P&L benchmark totals.

This was too strict because P&L benchmark variance is diagnostic evidence, not necessarily module invalidity.

## Locked Rule

Business Cost Variance is diagnostic unless a source module fails readiness.

It must remain visible, but it must not block Model Readiness when all source modules are individually ready.

## Current Behaviour

Business Cost Variance may expose:

- module_total_business_costs
- pnl_business_cost
- business_cost_variance_amount
- business_cost_variance_percent
- setup_module_benchmark_total
- full_pnl_business_cost_variance
- full_pnl_business_cost_variance_percent

## Interpretation

Readiness means:

- source modules are complete and usable

Variance means:

- benchmark comparison needs review

Readiness and variance must not be treated as the same thing.

# 9. LABOUR AGGREGATION

## Problem Solved

Module Reconciliation was consuming only the current Labour form/profile total.

This caused Model Readiness to understate labour cost and created a false high variance.

## Example Before Fix

Observed Labour profiles:

- Profile 1 total labour cost: approximately $92,422
- Profile 2 total labour cost: approximately $93,686
- Profile 3 total labour cost: approximately $108,841

Expected active Labour total:

- approximately $294,949

But Module Reconciliation was using only the current form/profile.

## Locked Rule

Labour output contract must aggregate all active saved Labour profiles.

Active means:

profile.is_active !== false

Inactive profiles must not count.

## Aggregate Labour Outputs

Labour output contract should expose aggregate totals including:

- total_labour_cost_annual
- total_productive_output
- total_gross_wages_annual
- total_entitlements_annual
- total_employer_kiwisaver_annual
- total_esct_annual
- total_acc_levy_annual
- total_employer_contribution_annual
- active_staff[]
- labour_ready

current_staff may remain available for current form/UI use.

## Reconciliation Consumption Rule

Module Reconciliation must consume:

labour_output_contract.total_labour_cost_annual

first.

It may fall back to:

labour.outputs.total_labour_cost_annual

only for compatibility.

No individual Labour calculation formulas were changed.

## Productive Output Rule

Cost Summary must consume Labour’s final total_productive_output.

This must be the final productive output after entitlement and productivity logic has already been applied.

Cost Summary must not use:

- paid hours
- raw working hours
- pre-productivity hours
- estimated local hours

# 13. LOCKED PRINCIPLES

## Source of Truth

Each module owns its own inputs, calculations, and output contract.

## P&L Role

P&L provides benchmark classification and diagnostic comparison.

P&L does not own final module source totals.

P&L interest is historical / reported context when asset finance is involved.

## General Overheads Role

General Overheads owns shared, vehicle-running, staff-related, and non-asset overhead costs.

General Overheads owns non-asset interest.

General Overheads does not own asset finance interest.

## Assets Role

Assets models ownership-only asset cost.

Assets owns asset finance interest.

Assets must calculate asset finance interest from the current / forward-looking asset finance position.

Assets does not own running costs.

Assets does not own principal repayments for Cost Summary.

## Labour Role

Labour aggregates all active saved staff profiles.

Labour owns final productive output.

## Reconciliation Role

Module Reconciliation checks readiness and shows variance evidence.

It may show historical P&L vs forward-looking module differences as diagnostic information.

It must not change source-of-truth module outputs.

## Model Readiness Role

Model Readiness decides whether the setup modules are structurally ready.

## Cost Summary Role

Cost Summary consumes ready upstream outputs and presents the forward-looking operating cost baseline.

## Cash Flow Role Later

Cash Flow owns principal repayments, full finance payment pressure, and liquidity view.

Cash Flow is separate from Cost Summary.

The Cost Setup Readiness milestone confirms that QS Tools can pass Model Readiness using active P&L, Labour, General Overheads, and Assets data, while preserving diagnostic variance evidence without allowing P&L benchmark mismatches to incorrectly block source-module readiness.

- Assets owns asset finance interest
- P&L interest is historical / reconciliation context where asset finance is involved
- General Overheads owns non-asset interest only
