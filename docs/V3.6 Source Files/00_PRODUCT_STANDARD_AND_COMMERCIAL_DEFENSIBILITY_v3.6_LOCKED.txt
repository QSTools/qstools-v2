# QS Tools — Product Standard & Commercial Defensibility Brief (v3.6 — LOCKED)

STATUS: ACTIVE SYSTEM PRINCIPLE  
VERSION: v3.6

---

## PURPOSE

This brief defines the product quality standard for QS Tools.

QS Tools is not being built to be merely functional.

QS Tools is being built to be commercially defensible, first-principles aligned, traceable, and difficult to pick holes in.

The target standard is 10/10 across every core product metric.

---

## PRODUCT STANDARD

QS Tools must be:

```text
Good enough to defend, not just good enough to work.
```

A working number is not enough.

A trusted number must be:

- traceable
- explainable
- reconciled
- owned by one module
- free from hidden assumptions
- supported by visible treatment decisions
- safe to use downstream

---

## CORE COMMERCIAL AMBITION

QS Tools should become the best commercial decision tool for construction/business costing.

It should not compete with accounting platforms as a bookkeeping system.

It should sit beside accounting systems and answer:

- what does this business really cost?
- which cost belongs where?
- what assumptions are hidden?
- what needs review?
- can the model be trusted?
- what revenue/rate is required to recover the true cost burden?
- should the business take this work at this price?
- is the business actually performing against the model it set?

---

## FIRST-PRINCIPLES PRODUCT RULE

Every part of QS Tools must follow:

```text
First principles for everything.

Simple in hindsight.
Invisible beforehand.
```

The system should reveal what was previously hidden in the business.

The user should be able to look at the result and say:

```text
That is obvious now that I can see it.
```

But before QS Tools exposed it, the issue was hidden in the P&L, accounting treatment, cost structure, quoting process, or operational setup.

---

## DEFENSIBILITY RULE

Every important output must survive challenge.

If someone questions a number, QS Tools must be able to show:

- where the number came from
- what source line or module produced it
- what category owns it
- what assumptions were made
- whether the user manually resolved it
- whether it is included or excluded
- whether it blocks readiness
- what downstream output it affects

If the system cannot explain a number, the number should not be trusted.

---

## 10/10 STANDARD

QS Tools should aim for 10/10 in:

- commercial logic
- cost ownership
- traceability
- reconciliation
- readiness gating
- user clarity
- source data treatment
- module contracts
- downstream trust
- decision usefulness

A 10/10 system means:

```text
Every cost has one owner.
Every assumption is visible.
Every unresolved item blocks trust.
Every resolved item has a treatment.
Every downstream number is traceable.
Every variance has an explanation.
Every module has a clean contract.
```

---

## NOT GOOD ENOUGH

The following are not acceptable for a market-grade QS Tools build:

- numbers that work but cannot be explained
- costs included only because they were convenient
- WIP automatically treated as COGS
- WIP automatically treated as General Overheads
- WIP automatically excluded
- penalties silently included in recovery
- finance costs mixed between Assets and General Overheads without visibility
- paid-off asset finance still creating current cost
- staff overheads treated as Labour
- vehicle running costs treated as Assets
- raw P&L values feeding Cost Summary directly
- downstream modules rebuilding upstream maths
- unassigned costs hidden behind summary totals
- warnings that do not tell the user what to fix
- trusted outputs produced from unresolved source data

---

## ACCOUNTING SYSTEM POSITION

Accounting systems such as Xero are the record-of-truth for bookkeeping and financial reporting.

QS Tools is not trying to replace the accounting system.

QS Tools uses accounting data as an input and turns it into a commercial decision model.

The accounting system answers:

```text
What happened?
```

QS Tools answers:

```text
What does this mean commercially?
What must the business recover?
Can the model be trusted?
What should the business charge or avoid?
Is the business performing against the model?
```

---

## FOUNDATION-FIRST RULE

Monthly P&L import, Xero API integration, quote acceptance tracking, and month-by-month performance analysis must not be built until the baseline model is commercially defensible.

The foundation must reach the required standard first:

```text
P&L classification
module ownership
reconciliation
model readiness
Cost Summary
Recovery Summary
Quote Benchmark
```

The system must not automate a weak model.

A weak baseline creates weak monthly analysis.

A trusted baseline creates meaningful variance analysis.

---

## ONE-TIME SETUP / REUSABLE MAPPING RULE

The first QS Tools setup may take time because the system is making the business’s P&L structure explicit.

This setup effort is intentional.

Once a P&L line has been classified and resolved, QS Tools should be able to reuse that treatment for future P&L periods.

The system should progressively build a reusable business mapping layer:

```text
P&L account / Xero account / report line
→ QS category
→ review subcategory
→ treatment
→ owning module
→ readiness impact
```

This means future P&L imports should become faster and more consistent.

The user should not need to repeatedly reclassify the same known accounts every month unless something has changed.

---

## SETUP VALUE RULE

The first setup should create durable value.

The goal is not just to classify one P&L.

The goal is to create a reusable commercial map of the business.

Once the business’s explicit P&L structure is mapped, future monthly P&L entry or Xero API import becomes a controlled update rather than a full setup exercise.

This is why setup accuracy matters.

A rushed setup creates weak future imports.

A clear setup creates scalable monthly review.

---

## P&L INPUT STANDARD

The P&L classification layer is production-critical.

Every P&L line must end as one of:

- included in the QS cost model
- excluded from QS recovery
- review required
- income / timing adjustment
- direct cost / COGS
- Labour
- Asset ownership
- General Overheads

Known lines should default to the correct treatment.

Unclear lines should remain visible and unresolved.

Material unresolved lines must block Model Readiness.

---

## MONTHLY P&L IMPORT RULE

Future monthly P&L imports should use the saved explicit P&L structure.

When a new monthly P&L is imported:

- known lines should map automatically
- known excluded items should remain excluded
- known review-required items should remain review-required
- known General Overheads subcategories should remain aligned
- known Labour items should flow to Labour
- known Asset ownership items should flow to Assets
- WIP / accounting adjustments should still require explicit review unless a locked treatment rule exists

New or changed lines should be highlighted for review.

The system should distinguish between:

```text
known mapped line
```

and:

```text
new or changed line requiring review
```

Monthly import should not remove the need for commercial review.

Monthly import should reduce repeated setup effort.

---

## BASELINE VS ACTUAL PERFORMANCE LOOP

QS Tools should eventually compare:

```text
baseline model
```

against:

```text
monthly actual P&L performance
```

The system should show whether the business is performing in line with the model it set.

Future monthly analysis should answer:

- did actual sales align with the required recovery model?
- did actual labour cost align with the baseline?
- did actual general overheads align with the baseline?
- did actual asset costs align with the baseline?
- did actual gross margin align with the quote model?
- did accepted quotes produce the expected recovery?
- where did actual performance drift from the model?

This turns QS Tools from a setup tool into an ongoing commercial control system.

---

## QUOTE ENGINE FEEDBACK RULE

The Quote Engine should eventually compare each quote against the trusted model.

For each quote, QS Tools should be able to show:

- quoted revenue
- expected cost
- expected margin
- expected recovery contribution
- alignment with the baseline model
- whether the quote was accepted
- whether the quote was rejected
- whether accepted work later appears in monthly P&L performance

Accepted quote tracking should create future evidence.

The system should learn from:

```text
quotes prepared
quotes accepted
quotes rejected
actual monthly sales
actual monthly cost recovery
```

This evidence should help the user understand whether the business is quoting in line with its commercial model.

---

## MODEL FEEDBACK STANDARD

QS Tools should eventually connect:

```text
planned model
quoted work
accepted work
actual P&L
monthly variance
```

The purpose is not only to calculate a rate.

The purpose is to create a feedback loop that shows whether the business model is working in reality.

The user does not need perfect job-level tracking to see whether the business model is working.

Job-level tracking may be useful later, but the first powerful layer is business-level model control.

The system should show:

- what the business planned
- what it quoted
- what work it won
- what the P&L shows
- where actual performance differs from the model

---

## COMMERCIAL CONTROL LOOP

The long-term QS Tools loop is:

```text
Baseline setup
→ trusted cost model
→ quote benchmark
→ quote accepted / rejected
→ monthly P&L import
→ actual performance vs model
→ variance analysis
→ better future pricing decisions
```

This creates a commercial control system.

The value is not only knowing what to charge.

The value is knowing whether the business is actually behaving the way the model says it should.

---

## USER TRUST RULE

The user must never accidentally trust an incomplete model.

If the model is incomplete, QS Tools must say so clearly.

If a number is preliminary, QS Tools must say so clearly.

If a cost is excluded, QS Tools must keep it visible.

If a material item is unresolved, QS Tools must block readiness.

If a model is ready, the user should have confidence that:

- source data has been classified
- module ownership is clean
- reconciliation has been checked
- downstream outputs are safe to use

---

## BUILD DISCIPLINE

Every build should follow this sequence:

1. first-principles rule
2. source brief update
3. narrow implementation task
4. build passes
5. visual / workflow check
6. commit
7. move to the next small piece

Do not make broad changes when a narrow change is enough.

Do not hide a conceptual problem with code.

Do not allow convenience to override commercial correctness.

---

## FUTURE XERO API RULE

Xero API integration should come after the core engine is stable.

The Xero API should be used to import P&L/accounting data into an already-defined QS Tools classification structure.

The API should not define the commercial model.

The commercial model defines how imported Xero lines are treated.

Future Xero flow:

```text
Xero P&L line
→ saved QS Tools mapping
→ category
→ review subcategory
→ treatment
→ readiness state
```

If a Xero line is new, changed, or unmapped, QS Tools must flag it for review.

The API makes the workflow faster.

It does not replace first-principles classification.

---

## FINAL STANDARD

QS Tools is successful when the user can trust the model because every cost, assumption, treatment, variance, and downstream number can be explained.

The product goal is not to produce a number quickly.

The product goal is to produce a number that can be defended.

The long-term product goal is to connect baseline, quote behaviour, accepted work, and actual monthly P&L performance so the business can see what is happening commercially at any given moment.
