# PURPOSE

This brief defines the active build sequence for QS Tools v3.8.

It protects the system from building downstream pages before upstream contracts, source ownership, readiness gates, recovery strategy, structural dependency, and outcome contracts are stable.

The build order must follow the first-principles rule:

```text
Brief first. Narrow build second. Verify. Commit. Move on.
```

Do not run broad Codex tasks against unclear behaviour.

Do not allow Codex to redesign the architecture.

If a rule is not written, update the source brief before coding.

---

# 1. ACTIVE BUILD PRINCIPLE

QS Tools v3.8 is governed by the recovery-chain flow:

```text
Business Summary
→ Recovery Summary
→ Cost Allocation
→ Business Outcome
```

The recovery chain must follow this separation:

```text
Business Summary = current reality
Recovery Summary = chosen strategy + visible consequence
Cost Allocation = delivery structure + dependency
Business Outcome = verdict
```

Core product rule:

```text
You can choose anything, but you cannot hide from what it causes.
```

Every downstream page must consume upstream output contracts only.

No downstream page may rebuild upstream maths.



Business Modelling and Quote Engine remain downstream and must not be built until this chain is stable.
