# QS Tools — First Principles System Brief

## v3.7 — QUOTE ENGINE AND FEEDBACK LOOP UPDATED

**STATUS:** ACTIVE SYSTEM PRINCIPLE
**VERSION:** v3.7
**REPLACES:** First Principles System Brief v3.6
**UPDATED FOR:** Quote Engine leak detection, accepted quote snapshots, monthly actuals, and macro feedback loop

---

## PURPOSE

This brief defines the first-principles operating philosophy behind QS Tools.

It applies to every module, calculation, contract, page, and downstream decision.

QS Tools must not hide assumptions, invent recovery numbers, duplicate costs, or create commercial outputs that cannot be traced back to real business costs, real productive capacity, clearly owned source modules, and visible user assumptions.

---

## CORE PRINCIPLE

**Simple in hindsight. Invisible beforehand.**

QS Tools reflects the business back to the user.

The system exists to make hidden commercial reality visible.

QS Tools uses historical accounting data to build a forward-looking commercial model.

The P&L shows what happened.

QS Tools uses that information to classify, reconcile, and model what the business needs to recover going forward.

Then QS Tools applies that model to scenarios, quotes, accepted quote predictions, and later monthly actuals.

---

## CORE PRODUCT QUESTION

QS Tools should continually answer:

**Do your assumptions about how your business runs match reality?**

This question applies at multiple levels:

* P&L classification
* cost ownership
* productive labour recovery
* operating cost burden
* revenue and margin pool
* current business summary
* scenario modelling
* quote pricing
* accepted quote behaviour
* monthly actual performance later

---

## FIRST PRINCIPLES RULE

Every output must be traceable back to:

* a real input
* a stated assumption
* a source module
* a calculation rule
* a downstream contract
* a trust state

If an output cannot be traced, it should not be trusted.

---

## COST OWNERSHIP RULE

Every cost must belong to exactly one module.

If a cost appears in more than one module, the system is incorrect.

If cost ownership is unclear, it must be classified before being used downstream.

Cost ownership must be explicit.

No duplicate ownership is allowed.

Examples:

* Labour cost belongs to Labour
* Employer obligations belong to Labour
* Vehicle running costs belong to General Overheads
* Staff overheads belong to General Overheads
* Asset ownership cost belongs to Assets
* Asset finance interest belongs to Assets
* Non-asset interest belongs to General Overheads
* Revenue and Direct Costs belong to Revenue / COGS
* Quote-level direct cost packages belong to Quote Engine
* Accepted quote snapshots belong to Quote Engine / Feedback Loop later
* Monthly actuals belong to P&L / Revenue / COGS reality layer later
* Principal repayments belong to Cash Flow later

---

## ASSUMPTION RULE

The system must remove assumptions wherever feasible.

Where an assumption remains, it must be:

* visible
* named
* explainable
* reviewable
* separated from actuals
* separated from locked predictions

Hidden assumptions are not allowed.

The system must not silently infer:

* no active assets
* clean P&L classification
* valid recovery strategy
* asset finance treatment
* cost ownership
* productive capacity
* quote viability
* quote acceptance
* cash-flow pressure
* monthly actual performance

If a treatment decision is required, the user must resolve it or the system must clearly mark it as unresolved.

---

## NO MAGIC NUMBER RULE

QS Tools must not create a recovery number out of thin air.

A recovery number must come from:

* actual cost burden
* productive capacity
* margin / recovery pool
* stated target
* recovery strategy
* selected model
* real-world quote benchmark

If the number cannot be explained, it is not valid.

Required Recovery Rate must be traceable to:

```text
total_cost_burden / total_productive_output
```

The denominator must be Labour’s final productive output after entitlement and productivity logic has already been applied.

Required Recovery Rate is not a labour charge-out rate.

It is the business break-even pressure per productive hour.

---

## FORWARD-LOOKING MODEL RULE

QS Tools is not only a historical reporting layer.

It is a commercial decision system.

Historical P&L values are used as evidence and classification context.

Forward-looking specialised module outputs are used as the decision basis where they are more accurate than historical accounting values.

Examples:

* Assets calculates forward-looking asset finance interest
* Labour calculates current productive output
* General Overheads produces current clean overhead burden
* Cost Summary uses current / forward operating cost
* Revenue / COGS produces current trading Margin Pool from classified P&L
* Business Summary compares current trading output with current operating burden
* Business Modelling starts from current reality and tests future changes
* Quote Engine tests individual quotes against the baseline or selected model
* Feedback Loop later compares locked accepted predictions against monthly actuals

The system must not blindly price from historical P&L values when a specialised module owns a clearer forward-looking cost.

---

## PAST VS FUTURE RULE

Past data is input evidence.

Future-facing cost is the decision basis.

Predictions must be preserved separately from actuals.

The system must distinguish:

* historical accounting data
* current operating cost
* forward-looking recovery requirement
* scenario model
* quote prediction
* accepted quote snapshot
* monthly actuals
* cash-flow pressure

Do not use past accounting values as final decision numbers when the current business structure has changed.

Do not reconstruct the past merely to force the future model to match the P&L.

Do not overwrite locked quote predictions with actuals.

---

## MODULE ROLE RULE

Each module must answer one clear question.

P&L:

What does the business financial reality currently show, and how should it be classified?

Revenue / COGS:

What did the business sell, what direct costs were required, and what Margin Pool remains?

General Overheads:

Which shared, unclear, staff-related, vehicle-running, and non-asset interest costs belong to business overhead?

Labour:

What does labour truly cost, and how much productive time exists?

Assets:

What does asset ownership cost, including current / forward-looking asset finance interest?

Module Reconciliation:

Is the upstream model internally consistent?

Model Readiness:

Can the downstream model be trusted?

Cost Summary:

What does the business cost to operate going forward?

Business Summary:

What is the current business position when Margin Pool is compared with operating cost burden?

Business Modelling:

What changes if the user adjusts controlled scenario levers from a locked baseline?

Quote Engine:

Does this individual quote recover the selected trusted business model?

Accepted Quote Snapshot:

What did we believe when we priced and accepted the work?

Feedback Loop later:

Do accepted quote predictions match monthly business reality at macro level?

Cash Flow later:

What principal and finance payment pressure affects liquidity?

Production Capacity later:

Can the priced work be delivered?

---

## COST SUMMARY PRINCIPLE

Cost Summary is the forward-looking operating cost layer.

It must answer:

**What does this business cost to operate per productive hour?**

Cost Summary must include:

* Labour cost
* Asset ownership cost
* Asset finance interest from Assets
* General Overheads

Cost Summary must exclude:

* Revenue
* COGS / Direct Costs
* Margin Pool
* Gross Margin
* Target Profit
* Quote Margin
* Quote Engine outputs
* Accepted quote snapshots
* Monthly actuals
* Principal repayments
* Cash-flow pressure
* Scenario outputs

Cost Summary must not rebuild upstream maths.

Cost Summary must not consume raw P&L lines directly.

Cost Summary must not use P&L interest as the source of truth for asset finance interest.

---

## REVENUE / COGS PRINCIPLE

Revenue / COGS owns the trading side of the model.

It must answer:

**What did the business sell, what did it directly spend to create those sales, and what Margin Pool remains?**

Core formula:

```text
Revenue
- Direct Costs
= Margin Pool
```

Margin Pool is not final profit.

Margin Pool is the pool available to carry the operating cost burden.

Revenue / COGS must not calculate Cost Summary.

Revenue / COGS must not calculate Net Position.

Revenue / COGS must not define pricing.

Revenue / COGS must not use optional micro / activity classifications in v3.7 core calculations unless a later locked brief promotes them.

---

## BUSINESS SUMMARY PRINCIPLE

Business Summary is the mirror.

It must show what the business currently is.

It combines:

```text
Margin Pool from Revenue / COGS
```

with:

```text
Total Cost Burden from Cost Summary
```

to show:

```text
Net Position
```

It must also show operational reality:

* Required Recovery Rate
* Current Margin per Productive Hour
* Recovery Gap per Hour

Business Summary must not suggest fixes.

Business Summary must not model scenarios.

Business Summary must not test individual quotes.

Business Summary must not mutate source modules.

---

## BUSINESS MODELLING PRINCIPLE

Business Modelling is the control layer.

It starts from the current business mirror.

It must create:

* locked baseline snapshot
* editable scenario copy
* live delta against baseline
* baseline / upside / downside benchmark roles

Baseline = reality.

Scenario = control.

The baseline must not be overwritten automatically.

Minimum recoverable rate is a reference line, not a scenario.

Business Modelling must not mutate P&L, Labour, Assets, General Overheads, Cost Summary, Revenue / COGS, or Business Summary.

---

## QUOTE ENGINE PRINCIPLE

Quote Engine is the quote-level leak detection layer.

It applies the trusted baseline or selected model to an individual quote.

It asks:

**Does this quote recover what the business actually needs?**

At quote level:

```text
quote_price_total
- direct_cost_total
= quote_margin_pool
```

Then:

```text
base_labour_hours_allowed
× required_recovery_rate
= business_recovery_required_total
```

Then:

```text
quote_margin_pool
- business_recovery_required_total
= quote_gap
```

Quote Engine must make the gap visible.

Quote Engine must preserve original quote vs adjusted test quote.

Quote Engine must let the user try changes and watch the gap close.

Quote Engine must not treat gross margin as proof of profit.

Quote Engine must not rebuild Cost Summary, Labour, Revenue / COGS, Business Summary, or Business Modelling.

---

## ACCEPTED QUOTE SNAPSHOT PRINCIPLE

Quotes are predictions.

Accepted quotes are locked predictions.

When a quote is marked accepted, the system must preserve a locked accepted quote snapshot.

That snapshot represents:

**What we believed when we priced the job.**

Accepted quote snapshots must not be overwritten by later edits or monthly actuals.

If the quote changes after acceptance, create a new version or variation later.

---

## FEEDBACK LOOP PRINCIPLE

The Feedback Loop is macro-level first, not job-level first.

V1 must compare:

```text
Accepted quote predictions in a month
vs
Monthly P&L actuals
```

It must not try to perfectly job-cost each individual job in V1.

The purpose is to detect whether the business is consistently pricing work correctly, not to prove every job line item.

Core idea:

**Quotes are predictions. Monthly P&L is reality. The feedback loop compares the two.**

---

## CASH FLOW PRINCIPLE

Cash Flow is separate from Cost Summary.

Principal repayments are real cash pressure.

Principal repayments are not operating cost.

Principal repayments must not feed:

* total_asset_cost_annual
* total_general_overheads
* total_cost_burden
* required_recovery_rate
* Cost Summary

Interest belongs in cost.

Principal belongs in Cash Flow.

Cash Flow later should show liquidity pressure, finance payments, and principal repayment burden without corrupting operating cost logic.

---

## ASSET FINANCE PRINCIPLE

Assets is the source of truth for asset finance interest.

P&L interest may be historical accounting context.

P&L interest must not override asset finance interest calculated by Assets.

If P&L interest includes asset finance, the system must not treat the P&L value as the asset finance authority.

Asset finance interest must be current / forward-looking.

Any mismatch between historical P&L interest and forward-looking asset interest is informational only.

---

## MACRO / MICRO CLASSIFICATION PRINCIPLE

Macro classification drives the v3.7 core engine.

Macro classification includes:

* income
* cogs
* labour
* assets
* general_overheads
* excluded
* unassigned
* review_required where applicable

Micro / activity classification may be captured for future insight.

Micro classification is optional in v3.7.

Micro classification must not block readiness.

Micro classification must not drive Cost Summary.

Micro classification must not drive Revenue / COGS unless a later locked brief explicitly promotes it.

Rule:

**Macro = active and required.**
**Micro = optional and stored for later.**

---

## DOWNSTREAM TRUST RULE

Downstream modules must consume upstream outputs only.

Downstream modules must not rebuild upstream maths.

If an upstream module is incomplete, downstream outputs may display for preview only but must not be treated as trusted.

If model_ready = false, downstream values may be shown only as untrusted / preview values.

A downstream module must not silently turn an upstream warning into a trusted output.

Quote Engine must inherit selected model trust state.

Accepted quote snapshots must preserve the trust state that existed when the quote was accepted.

Feedback Loop must not rewrite historical trust state.

---

## USER CLARITY RULE

The user must be able to understand:

* where the number came from
* what module owns it
* whether it is actual, assumed, diagnostic, calculated, scenario-based, or a locked prediction
* whether it is historical or forward-looking
* whether it is trusted
* what needs fixing if it is not trusted

The user must not need to inspect code to understand the business logic.

---

## RECONCILIATION RULE

The system must distinguish between:

* source-of-truth totals
* benchmark values
* diagnostic comparisons
* warnings
* blockers
* locked predictions
* monthly actuals

A diagnostic comparison must not replace a source-of-truth module output.

P&L benchmark variance is evidence.

It is not the calculation source for Cost Summary.

Historical P&L asset interest vs modelled asset interest is an explanatory bridge only.

It must not replace Assets output.

---

## COMMERCIAL REALITY RULE

QS Tools must reflect the business as it is, not as the user hopes it is.

If the business cannot recover its cost burden through available productive capacity, the system must show that clearly.

If a quote appears profitable but does not recover the required cost burden, the system must show that clearly.

If gross margin exists but internal labour, overheads, asset ownership, or target profit are under-recovered, the system must show that clearly.

---

## LABOUR RECOVERY RULE

QS Tools must expose labour recovery as early as possible.

The system must distinguish:

* internal labour recovery
* subcontract labour cost
* external/direct costs
* general overhead recovery
* asset ownership recovery
* target profit later

The system must not assume:

```text
Revenue - COGS = Labour content
```

That hides the real recovery problem.

Internal labour recovery must be compared against Labour’s required recovery burden where relevant.

---

## QUOTE / MODEL RECOVERY RULE

QS Tools must not treat gross margin as proof of profit.

A quote or model must generate enough margin / recovery pool to recover:

* internal labour
* general overheads
* asset ownership
* target profit later

If the margin / recovery pool is insufficient, the system must make the shortfall visible.

This does not necessarily mean the job is immediately cash-negative.

It means the quote or model fails the business recovery model.

---

## NOT ALLOWED

QS Tools must not:

* hide assumptions
* duplicate costs
* create untraceable recovery rates
* treat benchmarks as source-of-truth totals
* rebuild upstream maths downstream
* use raw P&L lines directly in Cost Summary
* use P&L interest as the authority for asset finance interest
* include principal repayments in Cost Summary
* mix cash-flow pressure into operating cost
* use display names as join keys
* silently ignore readiness blockers
* present untrusted outputs as final
* treat gross margin as proof of profitability
* treat required_recovery_rate as a labour charge-out rate
* confuse subcontract labour with internal productive labour
* overwrite accepted quote snapshots
* let monthly actuals overwrite quote predictions
* let monthly actuals overwrite source module assumptions automatically
* reintroduce removed modules without a locked source brief
* run Codex against unclear behaviour

---

## BUILD DISCIPLINE RULE

Every build must follow this sequence:

1. first-principles rule
2. source brief update
3. narrow implementation task
4. build passes
5. visual / workflow check
6. commit
7. move to the next small piece

Do not make broad changes when a narrow change is enough.

Do not hide a conceptual problem with code.

Do not allow convenience to override commercial correctness.

If the rule is not written, update the brief before coding.

---

## FINAL DEFINITION

QS Tools v3.7 is a first-principles commercial decision system.

It does not guess what the business should charge.

It shows:

* what the financial data says
* which module owns each cost
* what the business costs to operate going forward
* what revenue and direct costs produced as a Margin Pool
* whether that Margin Pool covers the operating burden
* what changes under controlled modelling
* whether quotes align with the model
* what was believed when accepted quotes were won
* whether monthly actuals later match accepted predictions
* what cash-flow pressure exists later without corrupting operating cost logic

Every number must be traceable.

Every assumption must be visible.

Every cost must belong to exactly one module.

Historical data is evidence.

Forward-looking commercial cost is the decision basis.

Accepted quote snapshots are locked predictions.

Monthly actuals are reality.

The Feedback Loop compares them without overwriting either.
