# 3. HIGHEST-LEVEL PRINCIPLE

QS Tools is built on:

```text
First principles for everything.
Simple in hindsight. Invisible beforehand.
```

The system must reveal what was previously hidden in the business.

Every cost must belong to exactly one owner.

Every important output must be traceable, explainable, and defensible.

If a number cannot be defended, it should not be trusted.

Core product question:

```text
Do your assumptions about how your business runs match reality?
```

---

# 4. CORE SYSTEM PHILOSOPHY

QS Tools uses historical accounting data to build a forward-looking commercial model.

The P&L shows what happened.

QS Tools uses that information to classify, reconcile, and model what the business needs to recover going forward.

Past accounting data is input evidence.

Forward-looking commercial cost is the decision basis.

Business Setup defines whether the business is interpreted through:

```text
productive hours
units sold
```

This allows QS Tools to support both service / labour-based businesses and product / unit-based businesses without changing the first-principles model.

---

# 5. CORE MODE-AWARE PRINCIPLE

QS Tools uses one universal recovery equation:

```text
Total Business Cost ÷ Activity Driver = Required Recovery Per Output
```

The activity driver is selected by Business Setup and applied by Business Summary.

```text
labour_based  → productive hours → $/hour
product_based → units sold       → $/unit
```

This is not an industry category.

It is the output driver used by the system to interpret recovery pressure.

---

# 6. CORE RECOVERY CHAIN PRINCIPLE

The recovery chain must follow this separation:

```text
Business Summary = current reality
Recovery Summary = chosen strategy + visible consequence
Cost Allocation = delivery structure + dependency
Business Outcome = verdict
```

Core recovery rule:

```text
You can choose anything, but you cannot hide from what it causes.
```

This means:

```text
REALITY      = current business position
STRATEGY     = user-selected recovery plan
CONSEQUENCE  = live pressure, gap, dependency, and feasibility
VERDICT      = whether the model works
```

No hidden assumptions.

No fantasy modelling.

No downstream page may calculate what an upstream page already owns.

```text
↓
Quote Engine later
↓

↓

```

Operationally, Cost Summary depends on General Overheads, Labour, Assets, Module Reconciliation, and Model Readiness.

Commercially, Revenue / COGS creates Margin Pool and Business Summary compares that Margin Pool against Cost Summary’s operating burden through the selected activity driver.

Strategically, Recovery Summary chooses how the burden is intended to be recovered.

Structurally, Cost Allocation checks what the strategy depends on.

Outcome-wise, Business Outcome produces the verdict.

# 9. BUSINESS SETUP OWNS

Business Setup owns:

```js
business_name
business_type
is_labour_based
is_product_based
setup_completed
```

Allowed `business_type` values:

```js
"labour_based"
"product_based"
```

Business Setup answers:

```text
What business is being modelled, and does it recover cost through hours or units?
```

Business Setup controls downstream interpretation:

```text
labour_based  → productive hours → required recovery per hour
product_based → units sold       → required recovery per unit
```

Business Setup does not calculate revenue, direct costs, margin pool, cost burden, recovery, scenarios, quotes, allocation, outcome, or cash flow.

Business Setup must remain editable during the current build phase so the mode-aware system can be tested.

Core formula:

```text

```

In product-based mode, Revenue / COGS captures:

```js

```

This is the annual product / unit activity driver used by Business Summary.

Margin Pool is not final profit.

Margin Pool is the pool available to carry the business operating cost burden.

# 24. DOWNSTREAM CONTRACT RULE

Upstream modules own inputs and calculations.

Downstream modules consume outputs only.

Never rebuild upstream maths downstream.

Never read raw upstream form state in downstream modules unless the relevant brief explicitly allows it.

Never rename locked contract variables without a matching source brief update.

Never use display names as join keys.

Use IDs only for joins.

If a downstream module needs a value, the upstream module must expose it through its output contract.

# 25. COST OWNERSHIP RULE

Every cost must have one owner.

No duplicate cost ownership is allowed.

Examples:

```text
Salary & Wages → Labour
Employer KiwiSaver → Labour
ESCT → Labour
Employer ACC Levy → Labour
Staff Expenses → General Overheads / Staff Overheads category
Entertainment → General Overheads / Staff Overheads category
Vehicle fuel / servicing / repairs / tyres / registration → General Overheads / Vehicle Running Costs
Vehicle finance / asset ownership → Assets
Asset finance interest → Assets
Paid-off asset finance → $0 current finance cost
Non-asset interest / overdraft / working capital interest → General Overheads
Revenue / COGS → Revenue / COGS
Quote direct cost packages → Quote Engine later
Principal repayments → Cash Flow later, not Cost Summary
WIP Adjustment → Review Required until explicitly resolved
Penalties / Non-Deductible → Excluded unless explicitly included
```

# 26. MODEL READINESS RULE

The user must never accidentally trust an incomplete model.

If the model is incomplete, QS Tools must say so clearly.

If a number is preliminary, QS Tools must say so clearly.

If a cost is excluded, QS Tools must keep it visible.

Readiness formula:

```text
model_ready =
pnl_ready
AND general_overheads_ready
AND labour_ready
AND assets_ready
AND reconciliation_ready
AND blocking_checks.length = 0
```

Status formula:

```text
if blocking_checks.length > 0:
  model_readiness_status = "blocked"

else if warning_checks.length > 0:
  model_readiness_status = "warning"

else:
  model_readiness_status = "ready"
```

Recovery Summary, Cost Allocation, and Business Outcome must inherit Model Readiness state.

# 29. FINAL DEFINITION

QS Tools v3.8 introduces a mode-aware recovery chain.

Business Setup defines whether the business is interpreted through hours or units.

Revenue / COGS creates Margin Pool and captures `units_sold_annual` where product mode requires it.

Cost Summary defines the operating cost burden and productive-hour recovery baseline.

Business Summary converts the trading result and operating burden into a universal activity-driver view:

```text
Required Recovery Per Driver
Current Margin Per Driver
Recovery Gap Per Driver
```

Recovery Summary turns that current reality into a selected strategy and visible consequence.

Cost Allocation tests the delivery structure and dependency created by that strategy.

Business Outcome gives the verdict.

The system remains first-principles, traceable, and ownership-safe.

Every downstream module must consume mode-aware outputs without rebuilding upstream maths.
