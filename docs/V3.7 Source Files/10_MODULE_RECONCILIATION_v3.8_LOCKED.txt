# 1. PURPOSE

Module Reconciliation checks whether the upstream model is internally consistent, contract-compliant, and safe to pass into Model Readiness.

It is not a user workflow page.

It is a system evidence layer.

Module Reconciliation answers:

```text
Is the upstream model evidence clean enough for Model Readiness to issue a trust verdict?
```

It does not answer:

```text
what the final verdict is
what recovery strategy should be selected
whether the selected recovery strategy can be delivered
whether the business model works
what quote should be accepted
whether a quote is leaking
whether accepted quotes matched monthly actuals
what scenario should be selected
what cash-flow pressure exists
```

Those belong to other layers.

---

# 2. PRODUCT ROLE

Module Reconciliation is the evidence layer.

Model Readiness is the trust verdict layer.

Cost Summary and all later downstream layers depend on the verdict.

Module Reconciliation is not:

```text
Recovery Summary
Cost Allocation
Business Outcome
Business Modelling
Quote Engine
Feedback Loop
Cash Flow
```

It proves whether source modules are contract-compliant before trusted downstream use.

---

# 3. CORE PRINCIPLE

Reconciliation is the evidence layer.

Model Readiness is the verdict layer.

Module Reconciliation does not create source-of-truth totals.

Module Reconciliation does not create recovery strategy.

Module Reconciliation does not create allocation status.

Module Reconciliation does not create business outcome verdicts.

Cost Summary must not be trusted until reconciliation has checked the upstream modules.

Recovery Summary, Cost Allocation, and Business Outcome must inherit trust through Model Readiness.

# 11. HARD CHECKS — GENERAL OVERHEADS

General Overheads checks include:

```text
total_general_overheads exists
general_overheads_ready is true
General Overheads categories exist
Staff Overheads is category only
Staff Overheads is not a separate module
total_general_overheads is the only downstream overhead output
vehicle running costs are classified to General Overheads
General Overheads does not include Labour employer obligations
General Overheads does not include asset ownership costs
General Overheads does not include asset finance interest
General Overheads does not expose quote-level repair inputs
```

---

# 12. HARD CHECKS — LABOUR

Labour checks include:

```text
total_labour_cost_annual exists
total_productive_output exists
total_productive_output > 0 where hour-based recovery is required
total_employer_contribution_annual exists
employer KiwiSaver is included
ESCT is included
employer ACC levy is included
Labour does not include staff overheads
Labour does not include asset ownership costs
Labour does not include general overheads
Labour does not expose quote-level accepted snapshot fields
```

In product-based mode, zero productive hours may still require warning depending on downstream use, but Business Summary uses units sold as the primary recovery driver.

---

# 13. HARD CHECKS — ASSETS

Assets checks include:

```text
total_asset_cost_annual exists
assets_ready is true
Assets are ownership-only
running_cost_annual does not feed Cost Summary
Assets do not include fuel
Assets do not include servicing
Assets do not include maintenance
Assets do not include registration
Assets do not include tyres
Assets do not include shared fleet costs
Assets do not include labour cost
Assets do not include principal repayments in total_asset_cost_annual
```

If no active assets exist, Module Reconciliation must check whether the user has explicitly confirmed that no active assets are required.

Allowed confirmation field:

```js
no_active_assets_confirmed
```

Rules:

```text
If active assets exist:
  Assets readiness should be based on ownership-only cost completeness and reconciliation.

If no active assets exist and no_active_assets_confirmed = true:
  Assets may pass readiness with an informational check.

If no active assets exist and no confirmation exists:
  Assets readiness must fail.
```

A missing or zero P&L asset benchmark must not be treated as confirmation that no assets are required.

# 17. BLOCKER VS WARNING RULE

## Blockers

A blocker means Cost Summary may display totals but must not be trusted.

Examples:

```text
missing total_labour_cost_annual
missing total_general_overheads
missing total_asset_cost_annual
employer ACC levy missing from Labour
running_cost_annual feeding Assets total
employee_overheads active category or output detected
total_productive_output missing or zero where hour-based recovery rate is required
no active assets exist and no_active_assets_confirmed is not true
module total business cost differs materially from P&L business cost
principal repayments feeding Assets, General Overheads, or Cost Summary
source modules include quote-level snapshot fields
```

## Warnings

A warning means the model can display but the user should review the result.

Examples:

```text
P&L benchmark variance exists
unallocated General Overheads remain
optional category detail missing
unusual cost split detected
no active assets exist but no_active_assets_confirmed is true
P&L asset benchmark is missing but the user has explicitly confirmed no active assets are required
P&L reported interest differs from Assets modelled interest
monthly actuals later show drift but no explicit model refresh has been made
```

# 19. DIAGNOSTIC TOTALS

Module Reconciliation may calculate:

```js
module_total_business_costs =
  total_labour_cost_annual
  + total_asset_cost_annual
  + total_general_overheads
```

```js
pnl_business_cost_variance =
  module_total_business_costs - total_business_costs
```

```js
pnl_business_cost_variance_percent =
  total_business_costs === 0
    ? 0
    : pnl_business_cost_variance / total_business_costs
```

These are diagnostics only.

They do not replace module totals.

They do not feed Cost Summary calculations.

They do not feed Recovery Summary, Cost Allocation, Business Outcome, Quote Engine, or Feedback Loop calculations.

# 21. BUSINESS SUMMARY / RECOVERY CHAIN BOUNDARY

Module Reconciliation does not create Business Summary, Recovery Summary, Cost Allocation, or Business Outcome outputs.

It may provide trust evidence that downstream outputs depend on.

Business Summary, Recovery Summary, Cost Allocation, and Business Outcome must inherit trust state through Model Readiness.

---

# 22. QUOTE ENGINE BOUNDARY

Quote Engine depends on trusted baseline/model outputs later.

Module Reconciliation must not:

```text
calculate quote gaps
calculate repair levers
store quote versions
store accepted quote snapshots
decide whether a quote should be accepted
```

Quote Engine must inherit trust state through Model Readiness.

If Model Readiness is blocked, Quote Engine outputs must be preview/not trusted only.

---

# 23. FEEDBACK LOOP / MONTHLY ACTUALS BOUNDARY

The Feedback Loop belongs later.

Module Reconciliation does not compare accepted quote predictions with monthly actuals.

Module Reconciliation does not rewrite source module assumptions based on actuals.

If monthly actuals later reveal drift, that belongs to Feedback Loop or explicit model refresh flow.

Reconciliation remains evidence for the current baseline model.

---

# 26. FINAL DEFINITION

Module Reconciliation v3.8 is the evidence layer that checks whether P&L, General Overheads, Labour, and Assets are contract-compliant before Model Readiness issues the final trust verdict.

It does not calculate recovery strategies, allocation statuses, business outcomes, quotes, scenarios, feedback-loop results, or source-of-truth totals.

Assets may only pass without active asset records when the user explicitly confirms that no active assets are required.

Missing asset data is not confirmation.
