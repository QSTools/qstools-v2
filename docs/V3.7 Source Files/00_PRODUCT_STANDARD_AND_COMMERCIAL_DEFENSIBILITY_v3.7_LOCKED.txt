# QS Tools — Product Standard & Commercial Defensibility Brief

## v3.7 — QUOTE ENGINE AND FEEDBACK LOOP STANDARD UPDATED

**STATUS:** ACTIVE SYSTEM PRINCIPLE
**VERSION:** v3.7
**REPLACES:** Product Standard & Commercial Defensibility v3.6
**UPDATED FOR:** Quote Engine leak detection, accepted quote snapshots, monthly actuals, macro feedback loop, and homepage positioning

---

## PURPOSE

This brief defines the product quality standard for QS Tools.

QS Tools is not being built to be merely functional.

QS Tools is being built to be commercially defensible, first-principles aligned, traceable, and difficult to pick holes in.

The target standard is 10/10 across every core product metric.

---

## PRODUCT STANDARD

QS Tools must be:

**Good enough to defend, not just good enough to work.**

A working number is not enough.

A trusted number must be:

* traceable
* explainable
* reconciled
* owned by one module
* free from hidden assumptions
* supported by visible treatment decisions
* safe to use downstream
* clearly marked as historical, current, forward-looking, diagnostic, scenario-based, prediction-based, or actual-based

---

## CORE COMMERCIAL AMBITION

QS Tools should become the best commercial decision tool for construction / business costing.

It should not compete with accounting platforms as a bookkeeping system.

It should sit beside accounting systems and answer:

* what does this business really cost?
* which cost belongs where?
* what assumptions are hidden?
* what needs review?
* can the model be trusted?
* what revenue / rate is required to recover the true cost burden?
* what Margin Pool does the business actually generate?
* does that Margin Pool cover the operating burden?
* what changes if the user models rates, costs, productivity, or margins?
* is this quote making the margin the user thinks it is?
* should the business take this work at this price?
* is the business actually performing against the model it set?

---

## ACCOUNTING SYSTEM POSITION

Accounting systems such as Xero are the record-of-truth for bookkeeping and financial reporting.

QS Tools is not trying to replace the accounting system.

QS Tools uses accounting data as input evidence and turns it into a commercial decision model.

The accounting system answers:

**What happened?**

QS Tools answers:

**What does this mean commercially?**
**What must the business recover going forward?**
**Can the model be trusted?**
**Does this quote recover the model?**
**Is the business performing against the model?**

---

## CORE PRODUCT QUESTION

The product should keep returning to this question:

**Do your assumptions about how your business runs match reality?**

Supporting positioning:

**Simple in hindsight. Invisible beforehand.**

**We mirror your business back to you.**

## FORWARD-LOOKING COMMERCIAL MODEL RULE

QS Tools is not only a historical reporting layer.

Historical P&L values are input evidence.

Forward-looking commercial cost is the decision basis.

The system must not blindly price from historical P&L values when a specialised module owns a clearer current or forward-looking cost.

Examples:

* Labour owns current productive output and true people cost
* Assets owns current / forward-looking asset ownership and finance interest
* General Overheads owns current clean non-asset overhead burden
* Cost Summary owns forward-looking operating cost burden
* Revenue / COGS owns Margin Pool from classified income and direct costs
* Business Summary compares current trading output with current operating burden
* Business Modelling tests future changes from a locked baseline
* Quote Engine tests individual quotes against the selected trusted model
* Feedback Loop later compares accepted quote predictions against monthly actuals

The P&L is evidence.

The model is the decision layer.

---

## FIRST-PRINCIPLES PRODUCT RULE

Every part of QS Tools must follow:

**First principles for everything.**

**Simple in hindsight. Invisible beforehand.**

The system should reveal what was previously hidden in the business.

The user should be able to look at the result and say:

**That is obvious now that I can see it.**

But before QS Tools exposed it, the issue was hidden in the P&L, accounting treatment, cost structure, quoting process, cash-flow pressure, or operational setup.

## DEFENSIBILITY RULE

Every important output must survive challenge.

If someone questions a number, QS Tools must be able to show:

* where the number came from
* what source line or module produced it
* what category owns it
* what assumptions were made
* whether the user manually resolved it
* whether it is included or excluded
* whether it blocks readiness
* what downstream output it affects
* whether it is historical, current, forward-looking, diagnostic, scenario-based, prediction-based, or actual-based

If the system cannot explain a number, the number should not be trusted.

## NOT GOOD ENOUGH

The following are not acceptable for a market-grade QS Tools build:

* numbers that work but cannot be explained
* costs included only because they were convenient
* WIP automatically treated as COGS
* WIP automatically treated as General Overheads
* WIP automatically excluded
* penalties silently included in recovery
* finance costs mixed between Assets and General Overheads without visibility
* asset finance interest taken from historical P&L when Assets owns the current finance model
* paid-off asset finance still creating current cost
* principal repayments included in Cost Summary
* principal repayments hidden from future Cash Flow visibility
* staff overheads treated as Labour
* vehicle running costs treated as Assets
* raw P&L values feeding Cost Summary directly
* downstream modules rebuilding upstream maths
* unassigned costs hidden behind summary totals
* warnings that do not tell the user what to fix
* trusted outputs produced from unresolved source data
* gross margin treated as proof of profit
* required recovery rate treated as a labour charge-out rate
* labour recovery hidden inside material margin
* subcontract labour confused with internal productive labour
* micro / activity classifications used as core calculation drivers before they are locked
* scenario changes mutating the baseline or source modules
* accepted quote snapshots overwritten by later edits
* monthly actuals overwriting quote predictions
* monthly actuals overwriting source assumptions automatically
* feedback-loop analysis treated as job-level costing before the macro loop is proven

## COST OWNERSHIP STANDARD

Every cost must belong to exactly one owner.

No duplicate cost ownership is allowed.

Examples:

Salary & Wages → Labour
Employer KiwiSaver → Labour
ESCT → Labour
Employer ACC Levy → Labour
Staff expenses / welfare / PPE / recruitment → General Overheads
Vehicle fuel / servicing / repairs / tyres / registration → General Overheads
Asset ownership / lease / finance cost → Assets
Asset finance interest → Assets
Non-asset interest / overdraft / working capital interest → General Overheads
Revenue and Direct Costs / COGS → Revenue / COGS
Quote-level direct cost packages → Quote Engine
Accepted quote snapshots → Quote Engine / Feedback Loop later
Monthly actuals → P&L / Revenue / COGS actuals layer later
Principal repayments → Cash Flow later

If ownership is unclear, the item must remain visible and unresolved until classified.

---

## ASSET FINANCE STANDARD

Assets is the source of truth for asset finance interest.

P&L interest may be historical accounting context.

P&L interest must not override asset finance interest calculated by Assets.

If P&L interest includes asset finance, the system must not treat the P&L value as the asset finance authority.

Asset finance interest must be current / forward-looking.

The purpose is to estimate the asset finance interest the business needs to recover going forward, not to reconstruct the exact historical interest charged in the previous P&L period.

Any mismatch between historical P&L interest and forward-looking asset interest is informational only.

The system may show an explanatory interest / finance bridge, but that bridge must not alter source-of-truth module outputs.

---

## CASH FLOW SEPARATION STANDARD

Principal repayments are real cash pressure.

Principal repayments are not operating cost.

Principal repayments must not feed:

* total_asset_cost_annual
* total_general_overheads
* total_cost_burden
* required_recovery_rate
* Cost Summary

Principal belongs in the future Cash Flow layer.

Interest belongs in cost.

Principal belongs in cash flow.

QS Tools must eventually make principal and finance payment pressure visible without corrupting operating cost logic.

Cash Flow should answer:

**Can the business survive the payment pressure?**

Cost Summary answers:

**What does the business cost to operate per productive hour?**

These must remain separate.

---

## COST SUMMARY STANDARD

Cost Summary is the forward-looking operating cost layer.

Cost Summary must answer:

**What does the business cost to operate going forward?**

It must produce:

* total_people_cost_annual
* total_asset_cost_annual
* total_business_overheads
* total_business_cost_annual
* total_cost_burden
* total_productive_output
* required_recovery_rate
* model_readiness_status

Cost Summary must not include:

* Revenue
* COGS / Direct Costs
* Margin Pool
* Gross Margin
* Target Profit
* Quote Margin
* Quote Engine outputs
* Accepted quote snapshots
* Monthly actuals
* Principal repayments
* Cash-flow pressure
* Scenario outputs

Cost Summary must not use raw P&L lines directly.

Cost Summary must not use historical P&L asset interest as the source of truth for asset finance.

Cost Summary must consume upstream outputs only.

---

## PRODUCTIVE OUTPUT STANDARD

Required Recovery Rate must be based on final productive output.

The denominator must be Labour’s final productive output after entitlement and productivity logic has already been applied.

Cost Summary must not use:

* paid hours
* raw working hours
* pre-productivity hours
* guessed hours
* locally recalculated hours

If the productive output value cannot be traced to Labour, the recovery rate is not trusted.

Required Recovery Rate is not a labour charge-out rate.

It is the business break-even pressure per productive hour.

---

## REVENUE / COGS STANDARD

Revenue / COGS owns the trading side of the model.

It must answer:

What did the business sell?
What did the business directly spend to create those sales?
What Margin Pool remains?

Core formula:

```text
Revenue
- Direct Costs
= Margin Pool
```

Margin Pool is not final profit.

Margin Pool is the pool available to carry the business operating cost burden.

Revenue / COGS must not calculate Cost Summary.

Revenue / COGS must not calculate Net Position.

Revenue / COGS must not define pricing.

Revenue / COGS must not model scenarios.

Revenue / COGS consumes classified P&L income and COGS / direct cost outputs.

Micro / activity classifications may be captured for future insight, but they must not drive v3.7 core Revenue / COGS calculations unless a later locked brief explicitly promotes them.

---

## BUSINESS SUMMARY STANDARD

Business Summary is the mirror layer.

It must show what the business currently is.

Business Summary combines:

```text
Margin Pool from Revenue / COGS
```

with:

```text
Total Cost Burden from Cost Summary
```

to show:

```text
Net Position
```

It must also show operational reality:

* Required Recovery Rate
* Current Margin per Productive Hour
* Recovery Gap per Hour

Business Summary must not suggest fixes.

Business Summary must not model scenarios.

Business Summary must not test individual quotes.

Business Summary must not mutate source modules.

The page should feel like:

**Here is the business as it currently stands.**

It should not feel like:

**Here is what you should do next.**

---

## BUSINESS MODELLING STANDARD

Business Modelling is the control layer.

It starts from the current business mirror.

It must create:

* locked baseline snapshot
* editable scenario copy
* baseline / upside / downside scenario roles
* live delta against baseline

Baseline = reality.

Scenario = control.

The baseline must not be overwritten automatically.

Minimum recoverable rate is a reference line, not a scenario.

Business Modelling may support:

* Upside / improvement scenario
* Downside / pressure scenario
* rate changes
* margin changes
* productivity changes
* operating cost changes

Business Modelling must not mutate P&L, Labour, Assets, General Overheads, Cost Summary, Revenue / COGS, or Business Summary.

---

## QUOTE ENGINE STANDARD

Quote Engine is the quote-level leak detection layer.

It should answer:

**Does this quote recover what the business actually needs?**

Quote Engine should show:

* quote price
* direct costs
* true internal labour cost where relevant
* margin / recovery pool
* required business recovery
* quote gap
* hourly gap
* max viable hours
* required price at allowed hours
* repair levers
* original quote vs adjusted quote
* model trust state

Quote Engine must not treat gross margin as proof of profit.

Quote Engine must not ask the user for true labour cost when Labour already owns it.

Quote Engine must not rebuild upstream maths.

Quote Engine must preserve accepted quote snapshots later.

---

## ACCEPTED QUOTE SNAPSHOT STANDARD

A quote may be revised many times.

Only the accepted version becomes the prediction record for the future feedback loop.

Accepted quote snapshots must be immutable.

An accepted quote snapshot represents:

**What we believed when we priced the job.**

Monthly actuals must not overwrite accepted quote snapshots.

Later changes should create new versions or variations.

## FOUNDATION-FIRST RULE

Monthly P&L import, Xero API integration, quote acceptance tracking, cash-flow forecasting, and month-by-month performance analysis must not be built until the baseline model is commercially defensible.

The foundation must reach the required standard first:

P&L classification
module ownership
reconciliation
model readiness
Cost Summary
Revenue / COGS
Business Summary
Business Modelling
Quote Engine baseline build
Feedback Loop later
Cash Flow later

The system must not automate a weak model.

A weak baseline creates weak monthly analysis.

A trusted baseline creates meaningful variance analysis.

---

## ONE-TIME SETUP / REUSABLE MAPPING RULE

The first QS Tools setup may take time because the system is making the business’s P&L structure explicit.

This setup effort is intentional.

Once a P&L line has been classified and resolved, QS Tools should be able to reuse that treatment for future P&L periods.

The system should progressively build a reusable business mapping layer:

```text
P&L account / Xero account / report line
→ QS category
→ review subcategory
→ treatment
→ owning module
→ readiness impact
```

This means future P&L imports should become faster and more consistent.

---

## MONTHLY P&L IMPORT RULE

Future monthly P&L imports should use the saved explicit P&L structure.

When a new monthly P&L is imported:

* known lines should map automatically
* known excluded items should remain excluded
* known review-required items should remain review-required
* known General Overheads subcategories should remain aligned
* known Labour items should flow to Labour
* known Asset ownership items should flow to Assets
* known income and COGS / direct cost items should flow to Revenue / COGS
* WIP / accounting adjustments should still require explicit review unless a locked treatment rule exists

New or changed lines should be highlighted for review.

Monthly import should not remove the need for commercial review.

Monthly import should reduce repeated setup effort.

## BASELINE VS ACTUAL PERFORMANCE LOOP

QS Tools should eventually compare:

```text
accepted quote predictions
```

against:

```text
monthly actual P&L performance
```

The first feedback loop is macro-level, not job-level.

The system should show whether the business is performing in line with the model it set.

Future monthly analysis should answer:

* did actual sales align with accepted quote predictions?
* did actual Margin Pool align with expected Margin Pool?
* did actual labour cost align with the baseline?
* did actual general overheads align with the baseline?
* did actual asset costs align with the baseline?
* did accepted quotes produce the expected recovery?
* where did actual performance drift from the model?
* did cash-flow pressure align with the finance structure later?

The Quote Engine should create quote predictions.

The system should learn from:

* quotes prepared

This evidence should help the user understand whether the business is quoting in line with its commercial model.

The system must not invent rules or act as a black box.

QS Tools compares the model to reality.

## MODEL FEEDBACK STANDARD

QS Tools should eventually connect:

planned model
quoted work
accepted work
actual P&L
monthly variance
cash-flow pressure later

The purpose is not only to calculate a rate.

The purpose is to create a feedback loop that shows whether the business model is working in reality.

The user does not need perfect job-level tracking to see whether the business model is working.

Job-level tracking may be useful later, but the first powerful layer is business-level model control.

The long-term QS Tools loop is:

Baseline setup
→ trusted cost model
→ Revenue / COGS
→ Business Summary
→ Business Modelling
→ Quote Engine

→ monthly P&L import

The value is not only knowing what to charge.

## LABOUR RECOVERY LEAK RULE

QS Tools must expose labour recovery as early as possible in the model.

In fixed-price / quote-based businesses, labour recovery is often the main hidden leak.

A quote may appear profitable because total margin looks acceptable, while internal labour is actually being under-recovered and subsidised by:

* material margin
* owner labour
* unpaid time
* underestimated productive hours
* ignored downtime
* overhead absorption errors
* asset margin or other non-labour recovery

QS Tools must make this visible.

---

## MARGIN POOL RECOVERY RULE

For fixed-price and quote-based businesses, QS Tools must not treat gross margin as proof of profit.

A quote must generate enough margin / recovery pool to recover:

* internal labour cost
* general overheads
* asset ownership costs
* target profit later

The core fixed-price test is:

```text
Revenue
- external/direct costs
= margin / recovery pool
```

Then:

```text
margin / recovery pool
- required model recovery
= recovery surplus / shortfall
```

If the recovery surplus / shortfall is negative, the quote is under-recovering against the model, even if it appears profitable using a simple gross-margin view.

---

## INTERNAL RECOVERY POOL RULE

For labour recovery analysis, QS Tools must distinguish between:

* revenue
* external/direct costs
* subcontract labour
* internal labour
* general overhead recovery
* asset ownership recovery
* target profit

The defensible baseline concept is:

```text
Revenue
- external/direct costs
= margin / recovery pool
```

Subcontract labour must be handled carefully.

Subcontract labour is not internal labour capacity, but it may represent bought-in labour required to deliver the work.

QS Tools must not automatically treat subcontract labour as internal labour recovery.

---

## QUOTE-BASED BUSINESS RULE

For quote-based businesses, especially fixed-price work, QS Tools must not rely only on total quote margin.

The system must distinguish between:

```text
quote appears profitable
```

and:

```text
quote recovers the business model correctly
```

A quote can pass a simple total-margin check while still failing labour recovery, overhead recovery, asset recovery, target profit recovery, or cash-flow pressure.

---

## CASH-FLOW RESULT RULE

QS Tools should eventually produce a clear cash-flow result separate from the commercial recovery result.

The system should be able to say:

* this model covers operating cost but creates cash-flow pressure
* this model is commercially viable but finance payments are tight
* this model is cash-flow exposed because principal repayments are high

Cash-flow pressure must not be confused with operating cost.

Cash-flow pressure must not corrupt required recovery rate.

Cash Flow is a separate visibility layer.

---

## USER TRUST RULE

The user must never accidentally trust an incomplete model.

If the model is incomplete, QS Tools must say so clearly.

If a number is preliminary, QS Tools must say so clearly.

If a cost is excluded, QS Tools must keep it visible.

If a material item is unresolved, QS Tools must block readiness.

If a model is ready, the user should have confidence that:

* source data has been classified
* module ownership is clean
* reconciliation has been checked
* downstream outputs are safe to use
* quote tests inherit the correct trust state
* accepted predictions preserve the trust state they were created with

---

## BUILD DISCIPLINE

Every build should follow this sequence:

1. first-principles rule
2. source brief update
3. narrow implementation task
4. build passes
5. visual / workflow check
6. commit
7. move to the next small piece

Do not make broad changes when a narrow change is enough.

Do not hide a conceptual problem with code.

Do not allow convenience to override commercial correctness.

Do not run Codex against unclear behaviour.

If the rule is not written, update the brief before coding.

---

## FUTURE XERO API RULE

Xero API integration should come after the core engine is stable.

The Xero API should be used to import P&L / accounting data into an already-defined QS Tools classification structure.

The API should not define the commercial model.

The commercial model defines how imported Xero lines are treated.

Future Xero flow:

```text
Xero P&L line
→ saved QS Tools mapping
→ category
→ review subcategory
→ treatment
→ readiness state
```

If a Xero line is new, changed, or unmapped, QS Tools must flag it for review.

The API makes the workflow faster.

It does not replace first-principles classification.

---

## FINAL STANDARD

QS Tools is successful when the user can trust the model because every cost, assumption, treatment, variance, and downstream number can be explained.

The product goal is not to produce a number quickly.

The product goal is to produce a number that can be defended.

The long-term product goal is to connect baseline, quote behaviour, accepted work, actual monthly P&L performance, and cash-flow pressure so the business can see what is happening commercially at any given moment.

QS Tools must show whether a fixed-price quote actually recovers the business model.

The system must not allow a quote to look commercially healthy merely because gross margin is positive.

The key question is:

**Does the margin / recovery pool recover the business model?**

If not, QS Tools must make the shortfall visible.

Cash-flow pressure must also be visible later, but it must remain separate from operating cost and required recovery rate.
