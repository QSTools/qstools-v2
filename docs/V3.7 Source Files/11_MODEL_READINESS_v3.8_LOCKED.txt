# 1. PURPOSE

Model Readiness determines whether the QS Tools model is complete, reconciled, and clean enough to trust downstream outputs.

Model Readiness gates:

```text
Cost Summary
Revenue / COGS
Business Summary
Recovery Summary
Cost Allocation
Business Outcome
Business Modelling later
Quote Engine later
Cash Flow later
Production Capacity later
Feedback Loop later
```

Model Readiness answers:

```text
Can downstream outputs be trusted as final, or are they preview/warning/blocked?
```

---

# 2. PRODUCT ROLE

Model Readiness is the trust verdict layer.

Module Reconciliation is the evidence layer.

Model Readiness does not create totals.

Model Readiness does not recalculate modules.

Model Readiness does not create recovery strategy.

Model Readiness does not create allocation status.

Model Readiness does not create business outcome verdicts.

Model Readiness does not create quote logic.

Model Readiness does not create feedback-loop logic.

It decides whether downstream outputs can be treated as trusted.

---

# 3. CORE PRINCIPLE

If the model is not ready, downstream pages are not trusted.

Reconciliation is the evidence layer.

Model Readiness is the verdict layer.

Model Readiness does not create cost totals.

Model Readiness does not recalculate modules.

Model Readiness decides whether downstream outputs can be treated as trusted.

The trust state must remain visible through:

```text
Cost Summary
Business Summary
Recovery Summary
Cost Allocation
Business Outcome
Business Modelling later
Quote Engine later
```

```js
model_ready =
```

# 9. STATUS RULES

## ready

Use when:

```text
model_ready = true
no blocking checks exist
no major warnings exist
```

## warning

Use when:

```text
model_ready = true
no blocking checks exist
warning checks exist
```

Downstream outputs may be used, but warnings must remain visible.

Warnings do not block Cost Summary if source modules are structurally valid.

## blocked

Use when:

```text
model_ready = false
one or more blocking checks exist
```

Downstream outputs may display but must not be treated as final.

# 10. MODULE READINESS REQUIREMENTS

Model Readiness requires each source module to pass its own readiness contract.

## P&L readiness must confirm

```text
P&L data exists
required macro lines are classified
unassigned_balance = 0
no active employee_overheads category remains
benchmark totals are available where required
unresolved review-required lines are handled according to P&L rules
```

Optional micro / activity classifications must not block P&L readiness unless a later locked brief changes this.

## General Overheads readiness must confirm

```text
total_general_overheads exists
category totals exist
no employee_overheads category remains
staff overheads are category-only
vehicle running costs are included where applicable
asset ownership costs are excluded
asset finance interest is not duplicated
employer ACC levy is excluded
benchmark variance is visible where applicable
no banned downstream overhead variables are exposed
```

## Labour readiness must confirm

```text
total_labour_cost_annual exists
total_productive_output exists
total_productive_output is Labour’s final productive output
productive output has already had entitlement and productivity logic applied
Labour aggregates all active saved Labour profiles
inactive Labour profiles are excluded
employer ACC levy is included in Labour, not General Overheads
```

Cost Summary must not use paid hours, raw working hours, or pre-productivity hours.

## Assets readiness must confirm

```text
total_asset_cost_annual exists
ownership-only asset rules are followed
running_cost_annual does not feed total_asset_cost_annual
vehicle running costs are excluded
labour costs are excluded
asset finance interest is owned by Assets
asset finance interest is forward-looking where finance applies
paid-off or term-ended finance does not create current finance cost
principal repayments do not feed total_asset_cost_annual
principal repayments do not feed Cost Summary
no_active_assets_confirmed is explicit if no active assets exist
```

Assets readiness must not depend on P&L asset benchmark matching.

P&L benchmark variance is diagnostic only.

---

# 11. ASSET INTEREST READINESS RULE

Assets is the source of truth for asset finance interest.

If financed assets exist and finance is active, Assets should expose asset finance interest through the Assets contract where available.

Model Readiness must not require P&L interest to match Assets interest.

A mismatch between historical P&L interest and forward-looking asset interest may create a warning, but it must not block readiness if:

```text
Assets is valid
General Overheads is valid
Module Reconciliation is valid
no unresolved interest treatment blocker exists
```

The mismatch is informational.

It must not alter source-of-truth outputs.

---

# 12. CASH FLOW SEPARATION READINESS RULE

Principal repayments are cash-flow pressure, not operating cost.

Model Readiness must treat principal as excluded from Cost Summary.

Principal must not block Cost Summary readiness unless it is incorrectly feeding:

```js
total_asset_cost_annual
total_general_overheads
total_cost_burden
required_recovery_rate
```

Future Cash Flow may consume principal outputs later.

That is outside Cost Summary readiness.

---

# 13. RECONCILIATION RELATIONSHIP

Module Reconciliation owns evidence checks.

Model Readiness consumes the reconciliation verdict.

Business benchmark variance may create warning status.

Business benchmark variance must not block readiness when all source modules are valid and there are no blocking checks.

Historical P&L values are benchmark evidence.

They are not source-of-truth calculation inputs for Cost Summary.

---

# 14. READINESS VS VARIANCE RULE

Readiness means:

```text
source modules are structurally complete
source module outputs are valid
blocking checks are clear
```

Variance means:

```text
benchmark comparison needs review
```

Readiness and variance must not be treated as the same thing.

A model may be ready with warnings if benchmark variance exists but all source modules are valid.

# 15. GENERAL DOWNSTREAM BEHAVIOUR

If `model_ready = false`:

```text
Cost Summary totals may display
Cost Summary must show not trusted state
Revenue / COGS may display only if its own inputs exist, but combined downstream outputs are not trusted
Business Summary is not final
Recovery Summary is not final
Cost Allocation is not final
Business Outcome is not final
Business Modelling is not final
Quote Engine is not final
Cash Flow later is not final
Production Capacity later is not final
Feedback Loop later is not final
```

If `model_ready = true` and `model_readiness_status = "warning"`:

```text
Cost Summary may be used
downstream outputs may be used with visible warnings
warnings must remain visible
Recovery Summary may be used with visible warnings
Cost Allocation may be used with visible warnings
Business Outcome may produce a warning-state verdict
Quote Engine later may test quotes but must show inherited warning state
```

If `model_ready = true` and `model_readiness_status = "ready"`:

```text
Cost Summary outputs are trusted
downstream modules may consume trusted upstream contracts
Recovery Summary outputs may be trusted subject to its own warnings
Cost Allocation outputs may be trusted subject to its own warnings
Business Outcome may issue a trusted verdict subject to its own warnings
Quote Engine later may use trusted baseline/model inputs where its own quote inputs are valid
```

# 16. BUSINESS SUMMARY TRUST RULE

Business Summary must inherit Model Readiness state.

If Model Readiness is blocked:

```text
Business Summary may display preview values only.
Business Summary must not be treated as a final baseline.
```

If Model Readiness is warning:

```text
Business Summary may be used with visible warnings.
```

If Model Readiness is ready:

```text
Business Summary baseline may be treated as trusted, subject to its own page warnings.
```

---

```text
Recovery Summary may display preview only.
Recovery Summary must not convert an untrusted Business Summary into a trusted recovery plan.
```

If Business Summary is warning-state:

```text
Recovery Summary may be used with visible warnings.
```

If Business Summary is trusted:

```text
Recovery Summary outputs may be treated as trusted, subject to its own warnings.
```

# 21. QUOTE ENGINE TRUST RULE

Quote Engine must inherit Model Readiness state through Cost Summary, Business Summary, Business Outcome, or Business Modelling later.

If Model Readiness is blocked:

```text
Quote Engine may display preview only.
Quote outputs must be marked not trusted.
Accepted quote snapshots should not be treated as trusted predictions unless the user explicitly accepts a warning-state snapshot under a later locked rule.
```

If Model Readiness is warning:

```text
Quote Engine may produce outputs.
Inherited warnings must remain visible.
Accepted quote snapshots should preserve model_trust_state = warning.
```

If Model Readiness is ready:

```text
Quote Engine may use trusted baseline/model values.
```

Quote Engine must not bypass Model Readiness.

---

# 22. FEEDBACK LOOP TRUST RULE

The Feedback Loop belongs later.

When built, it must preserve the trust state of accepted quote snapshots.

If an accepted quote was created from a warning or blocked model state, that trust state must remain attached to the snapshot.

Monthly actuals must not rewrite historical model readiness.

The Feedback Loop compares predictions to reality; it does not rewrite the trust state that existed when the prediction was made.

# 29. FINAL DEFINITION

Model Readiness v3.8 is the trust gate for QS Tools.

It consumes module readiness flags and Module Reconciliation results, then decides whether Cost Summary and downstream commercial outputs can be trusted.

It does not create or alter cost totals.

It does not reconcile by changing module outputs.

It does not create recovery strategy, allocation status, business outcome verdicts, quote logic, or feedback-loop logic.

It ensures the forward-looking model is trusted only when P&L, General Overheads, Labour, Assets, and Module Reconciliation are structurally ready.

Recovery Summary, Cost Allocation, Business Outcome, Business Modelling, Quote Engine, and Feedback Loop must inherit and preserve this trust state.
