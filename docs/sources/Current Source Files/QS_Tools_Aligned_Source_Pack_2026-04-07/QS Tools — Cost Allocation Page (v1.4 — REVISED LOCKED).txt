# 📄 QS Tools — Cost Allocation Page (v1.4 — REVISED LOCKED)

## STATUS

✅ Revised to align with current system direction:

* Cost Summary remains **cost of business summary only**
* Recovery interpretation moves to a **future Recovery Layer**
* Cost Allocation remains **pure structure only**

This version replaces any wording that made Cost Allocation feed recovery interpretation directly into Cost Summary.

---

## PURPOSE

The Cost Allocation Page defines the **structural recovery framework** of the business.

It is responsible for:

* setting the active revenue model
* defining valid Labour ↔ Asset structural relationships
* preparing structural outputs for downstream modules
* supporting a future Recovery Layer without changing cost ownership boundaries

It is **not** responsible for cost calculation, pricing, or recovery analysis. 

---

## SYSTEM ROLE (CRITICAL)

Cost Allocation is a:

**structural linking module**

Position in system:

```text
Input Modules
↓
Cost Allocation
↓
Cost Summary
↓
Recovery Layer
↓
Budget
```

It must:

* define structure only
* define the active revenue model
* define valid active links between Labour and Assets
* expose structural outputs for downstream use

It must NOT:

* calculate labour cost
* calculate employee overhead cost
* calculate asset cost
* calculate total business burden
* calculate required recovery
* interpret business performance

---

## CORE PRINCIPLE

Cost Allocation answers:

> How is the business structurally set up to recover cost?

It does **not** answer:

> What does the business cost?
> What must be charged out?
> Is the business under-recovering?

Those belong elsewhere:

* **Cost Summary** = cost truth
* **Recovery Layer** = recovery interpretation later

---

The Cost Allocation Page owns:

* page layout for the structural linking layer

* display and editing of valid structural links
* version/profile management for allocation structures
* display of structural warnings and completeness state

The Cost Allocation Page does **not** own:

* labour identity creation
* asset identity creation
* cost formulas

* recovery formulas

## COSTALLOCATIONMAINCARD STRUCTURE

### A. CostAllocationProfilesCard

Purpose:

* save/load/delete allocation profiles
* manage active structural profile
* preserve history and versioning

Rules:

* one active allocation profile only
* editing creates new version
* previous versions remain preserved
* profile controls remain separate from link editing

---

### B. CostAllocationForm

Purpose:

* define the active structural recovery model

#### Required input

* `revenue_model`

Allowed values:

* `labour_only`
* `asset_driven`

Rules:

* value must be explicit
* no inferred fallback in UI
* model choice affects structural expectations only
* model choice does not calculate cost totals

#### Behaviour

##### `labour_only`

* asset links may be empty
* labour is treated as the primary structural recovery path
* no asset link requirement for validity

##### `asset_driven`

* active asset links are expected
* missing links should surface warnings
* structure indicates that recovery is expected through asset-linked delivery context

This remains consistent with the locked Cost Allocation source brief. 

---

### C. CostAllocationLinkTable

Purpose:

* define active structural links between Labour and Assets

Links active V1 records using:

* `staff_id` from Labour
* `asset_id` from Assets

#### Active link record

* `asset_labour_link_id`
* `asset_id`
* `staff_id`
* `is_active`
* `created_at`
* `updated_at`

Rules:

* IDs only
* display names for display only
* no exact duplicate active pairs
* link history preserved
* only active links used downstream

#### Behaviour

##### `labour_only`

* links optional
* empty active-link state allowed

##### `asset_driven`

* links expected
* missing active links trigger structural warnings
* unlinked active staff should be surfaced clearly

* `effective_from`

## IMPORTANT OWNERSHIP RULE

Cost Allocation may read upstream active records for structural linking and display, but it must not take ownership of upstream data.

That means:

* Labour remains owner of `staff_id`
* Assets remains owner of `asset_id`
* Employee Overheads remains owner of staff-overhead outputs
* Cost Allocation only references upstream IDs for structure

## DOWNSTREAM HANDOFF RULE (REVISED)

### Cost Summary

Cost Summary may consume Cost Allocation outputs for **structural visibility only**, such as:

* showing active recovery model
* showing linked/unlinked counts
* surfacing structural warnings

But Cost Summary must remain a **cost-only summary layer**.

That means Cost Summary must NOT:

* perform recovery interpretation
* alter cost totals based on revenue model
* infer labour-vs-asset recovery performance
* become a structural analysis engine

### Recovery Layer (future)

The future Recovery Layer may consume both:

* Cost Summary outputs
* Cost Allocation structural outputs

This is where recovery interpretation belongs.

So the downstream split is:

* **Cost Summary** = cost truth
* **Recovery Layer** = interpretation
* **Cost Allocation** = structure

This is aligned to your revised system direction and preserves module boundaries.
