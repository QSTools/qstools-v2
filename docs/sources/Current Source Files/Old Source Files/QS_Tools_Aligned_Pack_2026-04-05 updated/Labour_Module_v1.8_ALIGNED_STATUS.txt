# Labour Module — Source of Truth (v1.8 — LOCKED)

---

## 🎯 PURPOSE

The Labour Module is the foundation module of QS Tools.

It is responsible for:

- creating and owning `staff_id`
- defining working time structure
- calculating full employer labour cost
- producing commercial labour outputs

This is the entry point of the system.

---

## 🔒 SYSTEM ROLE (CRITICAL)

identity owner + labour cost engine

Labour must:

- generate and own `staff_id`
- calculate full employer cost
- produce clean outputs for downstream modules

Labour must NOT:

- include employee overhead costs
- include asset costs
- include allocation logic
- include business overhead logic
- include scenario persistence logic

## 📊 CALCULATION FLOW (LOCKED)

### Time Model

`paid_hours_per_year` =
`hours_per_week × 52`

---

`daily_hours` =
`hours_per_week / days_per_week`

---

`non_productive_paid_hours` =
(`annual_leave_weeks × hours_per_week`)
+ (`public_holiday_days × daily_hours`)
+ (`sick_days × daily_hours`)
+ (`bereavement_days × daily_hours`)
+ (`family_violence_days × daily_hours`)

---

`productive_hours` =
(`paid_hours_per_year - non_productive_paid_hours`)
× (`productivity_percent / 100`)

---

## 💰 Employer Contributions (UPDATED)

Employer contributions must reflect actual employer-funded super cost.

### Employer KiwiSaver Gross

If `employee_kiwisaver_enabled`:

`employer_kiwisaver_gross` =
`labour_rate × paid_hours_per_year × employer_kiwisaver_rate`

Else:

`employer_kiwisaver_gross = 0`

---

## 🧮 ESCT CALCULATION (UPDATED AND LOCKED)

ESCT is not a universal fixed percentage assumption.

ESCT must be determined from the employee’s earnings position using a graded threshold approach aligned to NZ payroll treatment.

### ESCT determination rule

ESCT must be derived from:

- wage earnings over the relevant annualised period
- employer KiwiSaver contribution amount
- the matching ESCT threshold band

### Required behaviour

Process:

1. calculate wage earnings from Labour inputs
2. calculate `employer_kiwisaver_gross`
3. determine the applicable annualised earnings basis
4. determine the matching ESCT threshold band
5. apply that ESCT band to the employer KiwiSaver contribution
6. produce `esct_amount`

### Rule

`esct_amount` must be a derived outcome.

It must NOT be modelled as:

`employer_kiwisaver_gross × one fixed default percentage for all cases`

### Contract stability rule

The module may still expose:

- `esct_rate`
- `esct_amount`

but `esct_rate` must represent the derived band result, not a hardcoded universal assumption.

---

### Total Employer Contribution Cost

`total_employer_contribution_cost` =
`employer_kiwisaver_gross + esct_amount`

---

## 💰 Labour Cost Model

`base_labour_cost_annual` =
`labour_rate × paid_hours_per_year`

---

`total_labour_cost_annual` =
`base_labour_cost_annual + total_employer_contribution_cost`

---

`loaded_labour_cost_rate` =
`total_labour_cost_annual / paid_hours_per_year`

---

`productive_labour_cost_rate` =
`total_labour_cost_annual / productive_hours`

---

`labour_cost_rate` =
`productive_labour_cost_rate`

---

## 💼 Commercial Outputs

`minimum_charge_out_rate` =
`productive_labour_cost_rate / (1 - margin_target_percent)`

---

`actual_margin_percent` =
(`charge_out_rate - productive_labour_cost_rate`) / `charge_out_rate`

---

`margin_gap` =
`charge_out_rate - minimum_charge_out_rate`

---

`profit_per_hour` =
`charge_out_rate - productive_labour_cost_rate`

---

`above_recovery` =
`charge_out_rate - minimum_charge_out_rate`

## 💾 STORAGE MODEL

Each profile stores:

- `staff_id`
- all input fields
- all calculated outputs
- `profile_version`
- `effective_from`
- `is_active`
- `created_at`
- `updated_at`

---

## 🔁 VERSION RULES

- editing creates new version
- only one active profile per `staff_id`
- history preserved
- downstream modules use active version only

---

## ⚠️ CORE RULES

- `staff_id` must be system-generated
- `staff_name` must never be used as key
- no employee overhead inputs allowed
- no asset inputs allowed
- no allocation logic allowed
- no general overhead logic allowed
- variable names must not change
- all outputs must match contract exactly
- all calculations must live in calculation files, not UI components

---

## 🖥 UI RULES

- no inputs allowed until profile created
- show real-time calculation updates
- highlight invalid or missing inputs
- keep layout clean and structured
- do not expose internal IDs
- collapsible cards are allowed for long explanatory sections
- profile controls must remain distinct from core input cards

---

## 🔗 INTEGRATION

### Employee Overheads

- links via `staff_id`

### Cost Summary

- consumes Labour outputs

### Budget

- consumes Cost Summary outputs, not Labour scenario outputs directly

### Labour Scenario Layer

- may reuse Labour outputs and Labour calculation engine
- must remain isolated from core Labour persistence
- must not overwrite live Labour state

---

## 🚫 NOT ALLOWED

- manual `staff_id` entry
- using names as keys
- merging employee overhead logic into Labour
- merging asset logic into Labour
- renaming variables
- skipping versioning
- mixing Scenario persistence into the Labour core
- duplicating Labour formulas inside React components
