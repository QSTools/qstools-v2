QS Tools — Responsive Layout Layer / Desktop Enhancement Brief (v1.0 — LOCKED)

STATUS: PHASE 4 LAYOUT CONTRACT

PURPOSE
This brief defines the post-Phase-3 responsive layout layer for QS Tools.

It exists to restore a stronger tablet and desktop decision-engine experience without undoing:
- the Phase 3 mobile-first cleanup
- the token-based UI system
- the page / hook / calculation separation
- the locked downstream module contracts

This brief is a layout brief only.

It does not change:
- calculations
- hooks
- selectors
- storage
- output contracts
- identity ownership

SYSTEM ROLE (CRITICAL)
The Responsive Layout Layer is a presentation-layer enhancement system.

It must:
- preserve mobile-first usability
- add controlled tablet layouts
- restore stronger desktop decision visibility
- centralise responsive layout behaviour
- prevent layout drift from returning

It must NOT:
- reintroduce ad hoc responsive grids
- change module logic
- change page data contracts
- move calculations into components
- allow page-specific layout hacks

CORE DEVICE MODEL
QS Tools must now be treated as a 3-layer responsive product:

1. Mobile
- stacked layout
- one column only
- heavy use of collapsible sections
- no persistent right rail
- no dense side-by-side blocks

2. Tablet / Pad
- limited split layout allowed
- maximum 2-column structure
- right-side summary allowed
- sticky behaviour limited or optional
- moderate collapse still encouraged
- readability preferred over density

3. Desktop
- controlled split layout allowed
- right-side decision rail allowed
- sticky rail allowed where useful
- less collapse required for key outputs
- highest visibility for cost flow, drivers, and summary

GLOBAL RESPONSIVE RULES
- Mobile remains the base
- No return to uncontrolled md:grid-cols-* culture
- Layout may change, logic may not
- Macro → Breakdown → Drilldown remains the UX model
- Full-width tools stay full-width where appropriate

GLOBAL LAYOUT CONTROLLER (REQUIRED)
Reusable page modes:
- stack
- split
- split_sticky
- full_width_analysis

RIGHT-RAIL RULES
A right rail may contain:
- summary cards
- cost flow cards
- driver/insight cards
- structural status summaries
- headline outputs

A right rail must NOT become:
- a second input column full of unrelated controls
- a place for duplicated cards
- a place for business logic
- a place for layout hacks

NOT ALLOWED
- rewrite module logic
- alter Labour outputs
- alter Cost Summary outputs
- consume Scenario in Cost Summary
- reintroduce raw md:grid-cols-* clutter
- create device-specific duplicate business components
- use display names as join keys
- move summary calculations into page.jsx
- bypass the token system
- break Scenario isolation
- break module boundaries

DEFINITION OF DONE
Phase 4 is complete when:
- mobile layout remains stable from Phase 3
- tablet uses a guided split layout where appropriate
- desktop restores stronger decision visibility
- Labour regains a controlled right-side output rail
- Scenario remains full-width and isolated
- Cost Summary remains macro-first and structurally clear
- no module logic changed
- no contract drift introduced
- no styling drift reintroduced
