QS Tools — Naming Convention & Source File Structure Brief (v1.0 — LOCKED)

STATUS: GLOBAL NAMING + SOURCE ORGANISATION RULE

PURPOSE
This brief defines:
1. Mandatory naming conventions across the entire QS Tools system
2. The required source file pack structure
3. Rules to prevent naming drift and file chaos as the system scales

CORE PRINCIPLE
If you cannot predict a file name or variable name without searching, the system is already drifting.

MODULE NAMING RULES
- Folder names = kebab-case
- Code/module names = PascalCase

FILE NAMING
Hooks:
- use<Module>.js

Calculations:
- <module>Calculations.js
Exception:
- costAllocationRules.js remains as-is

Selectors:
- <module>Selectors.js

Storage:
- <module>Storage.js
- <module>ProfileStorage.js

Identity:
- <module>Identity.js where applicable

COMPONENT NAMING
All components must be prefixed with the module name.

Standard components:
- <Module>StatusStrip.jsx
- <Module>MainCard.jsx
- <Module>SummaryCard.jsx
- <Module>HelpPanel.jsx
- <Module>Form.jsx
- <Module>ProfilesCard.jsx
- <Module>InsightCard.jsx
- <Module>LinkTable.jsx

Not allowed:
- Card.jsx
- Summary.jsx
- Panel.jsx
- Overview.jsx
- DataCard.jsx

VARIABLE NAMING RULE
- snake_case only

Examples:
- gross_wages_annual
- total_people_cost_annual
- employee_overheads_total_annual
- productive_labour_cost_rate
- minimum_charge_out_rate

IDENTITY NAMING RULE
Required IDs:
- staff_id
- asset_id
- allocation_profile_id
- asset_labour_link_id

Not allowed:
- id
- staffId
- assetName

OUTPUT CONTRACT NAMING
These names are system contracts and must NEVER change without migration.

Examples:
- gross_wages_annual
- entitlements_annual
- total_employee_overheads_annual
- total_people_cost_annual
- total_cost_burden
- required_recovery_rate

SOURCE FILE PACK STRUCTURE
A. Active System
- Active System Briefs
- Source Files Guardrail Brief

B. Module Contracts
- Labour
- Employee Overheads
- Assets
- General Overheads
- Cost Allocation
- Cost Summary
- Budget

C. UI / Layout
- UI System & Mobile Conversion Brief
- Responsive Layout Layer (Phase 4)

D. Build Templates
- New Module Build Template
- Naming Convention Brief
- New Page Checklist
- Audit Script Notes

FILE ORGANISATION RULES
Keep files grouped:
- components/<module>/
- hooks/
- lib/calculations/
- lib/selectors/
- lib/storage/

IMPORT PATH CONSISTENCY
Use predictable absolute imports:
- @/hooks/useLabour
- @/components/labour/LabourSummaryCard
- @/lib/calculations/labourCalculations

Not allowed:
- ../../hooks/useLabour
- ../components/Card

DRIFT PREVENTION RULE
Before creating a file, confirm:
- Does the name match the pattern?
- Does the module prefix exist?
- Does the file belong in the correct folder?
- Does the name match existing module naming?
- Would someone else guess this name correctly?

NOT ALLOWED
- freestyle naming
- multiple names for same concept
- missing module prefixes
- camelCase internal variables
- using display names as IDs
- renaming contract variables casually
- dumping files into incorrect source folders
- copying old files with inconsistent names
