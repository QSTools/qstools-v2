QS Tools — UI Migration Plan (v1.1 — LOCKED)

---

PURPOSE

This brief defines how existing QS Tools pages are migrated to the shared UI system.

It ensures:

* pages are rebuilt once only
* UI consistency across the system
* no disruption to business logic
* stable rollout across modules

---

CORE PRINCIPLE

Rebuild pages onto the shared UI system without changing logic.

---

SCOPE

This migration applies to:

* all pages using legacy or mixed styling
* all future page builds

---

WHAT THIS MIGRATION DOES

* replaces page-level styling with shared UI classes
* standardises layout structure
* aligns pages to mobile-first behaviour
* prepares system for future visual updates

---

WHAT THIS MIGRATION DOES NOT DO

* change calculations
* change module behaviour
* change data contracts
* change ownership boundaries
* redesign business logic
* restructure page flows

---

LOCKED BASELINE

Migration must use:

* shared UI classes from Implementation Standard v1.1
* current globals.css as the single design source

---

MIGRATION RULES

1. Replace page styling with shared UI classes
2. Do not create new styling systems
3. Do not mix old and new systems inconsistently
4. Preserve all existing logic and data flow
5. Preserve page order and structure
6. Use approved layout helpers only

---

PAGE MIGRATION PATTERN

Each page must be rebuilt using:

<main className="ui-page">
  <div className="ui-page-stack">
    ...
  </div>
</main>

Then apply:

* ui-section for major blocks
* ui-panel for inner cards
* shared inputs, buttons, and spacing

---

RESPONSIVE MIGRATION RULE

Mobile

* vertical stacking
* no forced columns

Tablet

* controlled ui-split-2 usage

Desktop

* approved layouts only
* no custom grid systems

---

ORDER OF MIGRATION

Phase 1 — Homepage (complete)

* validate UI system
* confirm spacing, typography, layout

Phase 2 — Simple pages

* pages with minimal interaction

Phase 3 — Core modules

* Labour
* Cost Summary
* P&L

Phase 4 — Downstream pages

* Recovery Summary
* Cost Allocation
* Recovery Outcome

---

VALIDATION AFTER EACH PAGE

Each migrated page must be checked for:

* correct UI class usage
* no inline styles
* no duplicate styling systems
* responsive behaviour
* unchanged logic

---

BRIEF UPDATE RULE

After migration:

* update the page brief to the next version
* confirm shared UI system usage
* confirm no logic changes

Examples:

* v1.0 → v1.1
* v3.0 → v3.1

---

FAILURE CONDITIONS

Migration fails if:

* pages introduce custom styling systems
* logic is altered unintentionally
* layout becomes inconsistent across pages
* responsiveness is broken

---

SUCCESS CRITERIA

* all pages use shared UI classes
* consistent layout across modules
* no duplicated UI logic
* visual updates possible via CSS only

---

FINAL DEFINITION

This migration locks the transition from page-level styling to a shared UI system, ensuring QS Tools can evolve visually without requiring another full rebuild.
