# QS Tools — Build Order Brief (v1.1 — LOCKED)

STATUS: EXECUTION PLAN (DEPENDENCY ORDER)

---

## CORE PRINCIPLE

> Build in dependency order — never build a page before its inputs and shared UI system exist.

---

## BUILD STRATEGY

* Start with shared CSS and layout foundation
* Then build upstream data and cost layers
* Then build downstream logic
* Then finish with validation layers

---

## PHASE 1 — UI FOUNDATION FIRST

### 1. CSS Merge Implementation

Purpose:

* merge the improved demo UI language into the main shared CSS safely

Includes:

* preserve current main tool globals.css foundation
* add missing reusable utilities
* confirm theme consistency
* confirm responsive behaviour
* confirm sidebar shell is unaffected

Rule:

* extend only
* do not replace blindly

---

### 2. Layout Separation

Purpose:

* separate marketing / entry pages from application pages

Includes:

* homepage without sidebar
* Quick Start without sidebar
* full tool with sidebar starting at P&L

Rule:

* use separate layout contexts
* do not use conditional sidebar hacks

---

### 3. UI Implementation Standard Lock-in

Purpose:

* ensure every page is built using the shared ui-* system

Rule:

* apply UI Implementation Standard while building every page from this point onward

---

## PHASE 2 — CORE INPUT LAYER

### 4. P&L Page

Purpose:

* capture real data
* assign costs correctly
* produce benchmarks

Must be completed before anything else downstream.

---

### 5. Cost Modules

Build in this order:

1. Labour
2. Employee Overheads
3. Assets
4. General Overheads

Rules:

* include reconciliation strips
* include readiness outputs
* no downstream logic
* no styling deviations

---

### 6. Model Readiness

Purpose:

* gate the system

Rule:

* if not ready → downstream pages not trusted

---

## PHASE 3 — FINANCIAL BASELINE

### 7. Cost Summary

Purpose:

* single source of truth

Outputs:

* total_cost_burden
* required_revenue

Rule:

* no other page modifies this

---

## PHASE 4 — BUSINESS STRUCTURE

### 8. Business Model

Purpose:

* mirror the business back to the user
* reveal financial reality

---

### 9. Recovery Summary

Purpose:

* define how the business is structured to earn

Rule:

* no pricing
* no time logic
* no capacity logic

---

## PHASE 5 — PRICING LAYER

### 10. Rate Models

Purpose:

* convert model into real-world rates

Outputs:

* $/hour
* $/m²
* $/unit
* other real earning formats

Rule:

* no time logic
* no capacity logic

---

## PHASE 6 — OPERATIONAL REALITY

### 11. Production Capacity

Purpose:

* convert revenue into time
* check whether the business can realistically deliver it

Outputs:

* time_required
* time_available
* utilisation
* capacity status
* intensity / pressure

---

## PHASE 7 — SYSTEM TRUTH

### 12. Recovery Outcome

Purpose:

* mirror the full business back to the user

Includes:

* financial reality
* pricing reality
* production reality

Rule:

* no new calculations

---

## PHASE 8 — REAL-WORLD VALIDATION

### 13. Quote Benchmark

Purpose:

* test real jobs against the business model

Checks:

* pricing vs rates
* job vs capacity
* overall business impact

---

## BUILD RULE

Apply UI structure as you build each page:

* do NOT build then redesign later
* follow UI Implementation Standard immediately
* keep pages inside the shared CSS system from the start

---

## TESTING RULE

After each phase:

* layout works
* sidebar remains stable
* inputs work
* outputs are correct
* calculations unchanged
* responsive behaviour correct

---

## CRITICAL RULES

1. Do not skip phases
2. Do not build out of order
3. Do not combine page responsibilities
4. Do not introduce new logic mid-build
5. Do not redesign pages outside the shared CSS system
