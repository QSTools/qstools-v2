# QS Tools — Active System Brief (v3.1 — LOCKED)

STATUS: MASTER SYSTEM CONTROL

---

## PURPOSE

This brief defines the **active QS Tools system** in one place.

It exists to:

* keep development aligned
* prevent system drift
* act as a quick reference during build
* define what each part of the system is allowed to do

---

## CORE PRINCIPLE

> QS Tools is a business alignment system that mirrors a business, defines what it must recover, and validates whether real work achieves that.

---

## SYSTEM STRUCTURE

QS Tools operates across three layers:

---

### 🔹 1. DEMO (QUICK START)

**Purpose:**

* create awareness
* expose gap
* trigger user realisation

**Role:**

* entry point only

**Does:**

* shows what work produces
* converts output into perceived rate
* reveals gap via Labour Reality

**Does NOT:**

* model the business
* define pricing
* provide solutions

---

### 🔹 2. CORE SYSTEM

**Purpose:**

* model the business correctly

**Flow:**

P&L
↓
Modules (Labour / Assets / Overheads / Materials)
↓
Cost Summary
↓
Business Model
↓
Recovery Summary

---

#### P&L

* captures all financial data
* separates:

  * revenue
  * COGS
  * business costs
* produces benchmark totals

---

#### Modules

Own cost logic only:

* Labour → labour cost engine
* Employee Overheads → staff overhead cost
* Assets → owned asset cost
* General Overheads → fixed business cost
* Materials → margin contribution layer

Rules:

* no duplication
* no cross-ownership
* no downstream influence

---

#### Cost Summary

* single source of truth
* defines:

  * total_cost_burden
  * required_revenue

Rule:

> No other page may create or modify required_revenue

---

#### Business Model

* compares:

  * actual vs required

* generates insight only

Rule:

> no recommendations, no optimisation

---

#### Recovery Summary

* user defines:

  * how revenue is recovered

* converts required revenue into streams

Rule:

> user decides, system explains

---

### 🔹 3. EXECUTION LAYER

**Purpose:**

* apply the model to real work

**Flow:**

Rate Models
↓
Cost Allocation
↓
Quote Benchmark
↓
Recovery Outcome

---

#### Rate Models

* converts:

  * cost + recovery + output
    → required rate

Rule:

> output must be defined by user

---

#### Cost Allocation

* checks:

  * capacity vs required output

Rule:

> only model constraints

---

#### Quote Benchmark

* compares:

  * real job vs required rate

Rule:

> user inputs, system evaluates

---

#### Recovery Outcome

* determines:

  * system viability

Rule:

> no recalculation, no optimisation

---

## FULL SYSTEM LOOP

1. Build business model
2. Define required recovery
3. Input job / quote
4. Validate outcome

---

## CORE SYSTEM RULES (NON-NEGOTIABLE)

### Data Integrity

* each cost entered once
* each cost owned by one module
* no duplication across system

---

### Flow Control

* P&L → modules → Cost Summary → downstream only
* no backward influence
* no circular logic

---

### Cost Separation

* COGS = delivery cost
* business costs = recovery base

---

### Ownership Rules

* modules own calculations
* summary pages do not calculate
* execution pages do not redefine cost

---

### Readiness Rules

* all modules must be reconciled
* model must be ready before execution

---

## UI SYSTEM RULE

> Modules control logic. CSS controls appearance.

* no module-level colour systems
* use shared UI framework only
* consistent layout across all pages

---

## PRODUCT DEFINITION

QS Tools is:

* a system that models real business cost
* a system that defines required recovery
* a system that validates real-world decisions

QS Tools is NOT:

* a calculator
* a quoting tool
* a pricing shortcut

---

## FINAL DEFINITION

QS Tools mirrors a business, defines its financial baseline, and validates whether real-world work aligns with that reality.
