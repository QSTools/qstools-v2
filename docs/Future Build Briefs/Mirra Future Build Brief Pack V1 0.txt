## 3. Core Product Principle

Mirra is a closed-loop business model system.

It does not only ask:

* What did the business do?
* What is the cash position?
* Did this job make money?

It asks:

* What should the business recover?
* Did the quote support that recovery?
* Did the job perform against the quote?
* Did the invoice recover what it should?
* Did the cash arrive when expected?
* Where is the pattern of loss or profit?
* What should be adjusted before the next quote?

---

## 4. Core Loop

```text
Quote
→ Job
→ Actuals
→ Invoice
→ Cash
→ Insight
→ Better Quote
```

This is the complete feedback loop.

The system behaves like a CFO by:

1. detecting variance
2. aggregating patterns
3. ranking impact
4. recommending which lever to pull
5. feeding results back into future quotes

---

## 5. System Boundary Rule

Do not turn Mirra into an accounting system.

### Mirra owns

* pricing logic
* recovery logic
* job baseline logic
* job performance interpretation
* invoice validation before send
* cash explanation
* variable learning

### Xero / accounting software owns

* ledger
* GST / tax compliance
* official invoice records
* debtor ledger
* payment status
* statements
* accounting reports

### Correct split

```text
Mirra = decision engine
Xero = accounting engine
```

# Quote Checker Refinement Brief v1.0

**STATUS:** FUTURE BUILD BRIEF

## Purpose

The Quote Checker validates whether a proposed quote supports the business recovery model before the job is accepted.

It answers:

> Should we accept this job at this price?

## System role

Quote Checker is a job-level commercial validation layer.

It must consume recovery outputs from the system and compare them against quote inputs.

It must not rebuild Cost Summary or Recovery Summary logic.

## Position

```text
Recovery Outcome
↓
Quote Checker
↓
Accepted Job Baseline
```

## Inputs

### Quote identity

* quote_id
* quote_name
* client_name
* job_type
* quote_date
* quote_version
* quote_status

### Allowed quote_status

* draft
* pending
* waiting
* accepted
* lost
* superseded

### Quote financial inputs

* quoted_sale_price
* material_cog
* subcontractor_cog
* other_direct_cog
* total_direct_cog
* quoted_labour_hours
* labour_cog
* quoted_margin
* quoted_margin_percent

## System inputs consumed

From Cost Summary / Recovery Summary:

* total_cost_burden
* required_revenue
* required_recovery_rate
* required_labour_recovery_rate
* active_recovery_model

## Calculations

```text
total_direct_cog =
material_cog + subcontractor_cog + other_direct_cog
```

```text
gross_profit =
quoted_sale_price - total_direct_cog
```

```text
gross_margin_percent =
gross_profit / quoted_sale_price
```

```text
quote_recovery_per_hour =
gross_profit / quoted_labour_hours
```

```text
quote_required_labour_recovery =
quoted_labour_hours × required_labour_recovery_rate
```

```text
quote_recovery_gap =
gross_profit - quote_required_labour_recovery
```

## Decision states

* above_floor
* near_floor
* below_floor
* invalid

## Outputs

* quote_recovery_per_hour
* quote_required_labour_recovery
* quote_recovery_gap
* gross_profit
* gross_margin_percent
* quote_viability_status
* quote_status
* quote_version
* recommended_action

## Core rules

* Must consume required recovery outputs only.
* Must not calculate total business cost.
* Must not change accepted job data after quote acceptance.
* Accepted quote version must be preserved.

# Draft Invoice Checker Brief v1.0

Draft Invoice Checker builds and validates invoices inside Mirra before they are pushed to Xero.

It answers:

> If we send this invoice, what does it mean?

## System role

Draft Invoice Checker is the pre-commit diagnosis layer.

## Position

```text

+ PO / Cost Reconciliation

Draft Invoice Checker
↓
Xero Push / Debtor Sync
↓
Cash Flow Intelligence
```

## Inputs

* draft_invoice_id
* job_id
* claim_period
* claim_stage
* draft_invoice_date
* draft_invoice_lines[]
* draft_invoice_total
* labour_claimed
* materials_claimed
* subcontractor_claimed
* other_claimed
* notes

## System inputs consumed

From Accepted Job Baseline:

* accepted_contract_value
* expected_labour_hours
* expected_material_cog
* expected_total_cog
* expected_margin
* expected_recovery

From Labour Tracking:

* actual_labour_hours_by_job
* actual_labour_cost_by_job
* labour_hours_variance

From PO / Cost Reconciliation:

* reconciled_material_cost
* reconciled_subcontractor_cost
* material_cost_variance

From Recovery Summary / Quote Checker:

* required_labour_recovery_rate
* required_recovery_rate
* quote_recovery_gap

## Calculations

```text
job_cost_to_date =
actual_labour_cost_by_job
+ reconciled_material_cost
+ reconciled_subcontractor_cost
+ other_actual_cost
```

```text
revenue_to_date_after_invoice =
previous_invoiced_total + draft_invoice_total
```

```text
job_margin_to_date =
revenue_to_date_after_invoice - job_cost_to_date
```

```text
labour_overrun_value =
max(0, actual_labour_cost_by_job - expected_labour_cog)
```

```text
material_margin_remaining =
accepted_contract_value
- actual_labour_cost_by_job
- reconciled_material_cost
- reconciled_subcontractor_cost
```

```text
job_recovery_gap =
revenue_to_date_after_invoice - required_recovery_to_date
```

## Outputs

* invoice_recovery_status
* labour_overrun_value
* material_margin_remaining
* job_margin_to_date
* job_recovery_gap
* invoice_margin_impact
* send_warning
* push_to_xero_ready
* primary_issue_key
* recommended_action

## Decision states

* healthy
* near_floor
* under_recovered
* margin_eroding
* loss_locked_in
* invalid

## Core warning examples

* Labour has exceeded allowance and is consuming material margin.
* This invoice keeps the job below required recovery.
* Even full contract value may not recover the job.
* This invoice is healthy against current job performance.
* Draft invoice is missing required line detail.

## Core rules

* Draft Invoice Checker does not replace user judgement.
* It does not override contract value.
* It does not suggest unlawful or unsupported claims.
* It explains consequence before push.
* It must not become accounts receivable.

Cash Flow Intelligence projects and explains cash pressure using real debtors, payment timing, cost commitments, and the business recovery model.

It answers:

> Is cash pressure caused by timing, under-recovery, under-claiming, volume, or cost structure?

* actual_payment_dates
* outstanding_balance

### Cost timing inputs

* PO commitments
* supplier invoice due dates
* payroll timing
* overhead timing
* asset payment timing
* tax/GST timing later

### System intelligence inputs

* required_revenue
* required_recovery_rate
* required_labour_recovery_rate
* quote_recovery_gap
* job_recovery_gap
* invoice_recovery_status
* labour_overrun_value
* material_margin_remaining

## Outputs

* projected_cash_balance_by_period[]
* cash_low_point
* cash_pressure_period
* cash_shortfall_amount
* timing_issue_flag
* under_recovery_flag
* under_claiming_flag
* late_debtor_flag
* volume_pressure_flag
* cost_structure_flag
* cash_cause_summary
* recommended_action

## Period model

Start simple with weekly periods.

For each period:

```text
opening_cash
+ expected_receipts
- expected_payments
= closing_cash
```

## Cause classification

### Timing issue

Recovery is healthy but receipts are late or costs land before cash.

### Under-recovery

Jobs/invoices are below recovery even if payment timing improves.

### Under-claiming

Work/cost has been incurred but invoice recovery is behind.

### Volume pressure

There is not enough accepted or active work to support required recovery.

### Cost structure pressure

Fixed costs exceed what current productive output can support.

## Core rules

* Cash Flow Intelligence must use real debtors where available.
* It must not rely only on model projection.
* It must explain why cash moves, not only show the bank curve.
* It must not replace accounting.
* It must consume previous layers rather than rebuild their logic.

# Performance Intelligence / Variable Learning Brief v1.0

**STATUS:** FUTURE BUILD BRIEF

## Purpose

Performance Intelligence aggregates expected vs actual performance across jobs and time to detect repeating patterns, identify profit leaks, and improve future quoting assumptions.

It answers:

> Where are we consistently wrong, and what should we adjust?

## System role

This is the CFO learning loop.

It turns job history into improved future pricing and quoting.

## Position

```text
Quote
→ Job
→ Actuals
→ Invoice
→ Cash
→ Performance Intelligence
→ Better Quote
```

## Inputs

### From Quote Checker

* quote_id
* quote_version
* quoted_sale_price
* quoted_labour_hours
* quoted_material_items[]
* quoted_cog
* quoted_margin

### From Accepted Job Baseline

* job_id
* accepted_contract_value
* expected_material_items[]
* expected_labour_hours
* expected_margin

### From Labour Tracking

* actual_labour_hours_by_job
* labour_hours_variance
* actual_labour_cost_by_job

### From PO / Cost Reconciliation

* reconciled_material_cost
* material_quantity_variance
* material_cost_variance
* supplier_invoice_match_status

### From Draft Invoice Checker

* job_recovery_gap
* labour_overrun_value
* material_margin_remaining
* invoice_recovery_status

### From Cash Flow Intelligence

* debtor_timing_variance
* cash_cause_summary
* late_payment_flag

## Outputs

* top_variance_drivers
* quoted_vs_actual_by_item
* quoted_vs_actual_by_category
* quoted_vs_actual_by_supplier
* labour_productivity_variance
* material_quantity_variance
* material_cost_variance
* margin_erosion_by_item
* margin_erosion_by_job_type
* recommended_variable_adjustments
* confidence_score
* trend_direction

## Core variable principle

```text
Structure is fixed.
Variables are emergent.
```

Users do not define variables upfront.

Variables emerge from repeated expected vs actual differences.

## Universal variable types

### 1. Quantity variables

How much was used vs expected.

### 2. Efficiency variables

How fast work was done vs expected.

### 3. Cost variables

Unit cost actual vs expected.

### 4. Price variables

Actual recovered price vs expected.

### 5. Timing variables

Actual payment/cost timing vs expected.

## Example — construction

Concrete 252RSS:

```text
quoted_total_quantity = 1,000 m3
actual_total_quantity = 1,080 m3
variance = +8%
impact = actual additional cost
recommendation = review quoting factor
```

## Example — coffee truck

Coffee beans:

```text
expected_yield = 60 cups/kg
actual_yield = 55 cups/kg
variance = -8.3%
impact = higher cost per cup
recommendation = review recipe/waste factor
```

## Example — retail

Shirt SKU:

```text
listed_price = $90
realised_average_price = $78
variance = -13.3%
impact = margin compression
recommendation = review markdown strategy
```

## Insight output examples

Top profit leaks:

1. Labour overrun — $42,000
2. Concrete quantity variance — $18,400
3. Supplier price drift — $9,700

Recommended actions:

* Review labour allowance on slab prep.
* Increase 252RSS allowance factor from 1.00 to 1.08.
* Review supplier pricing for concrete items.

## Core rules

* Do not require perfect line-item classification.
* Aggregate first; refine later.
* Start broad and segment only where data supports it.
* Do not create an inventory system.
* This page is performance intelligence, not stock management.
* Recommendations must be based on repeated patterns, not single-job noise.

* loss of the core decision-engine purpose

## Active future principle

Mirra must remain a decision engine.

It may create and validate operational records, but only where those records support recovery, performance, invoice consequence, cash explanation, or future quote improvement.

## System boundary rule

### Mirra owns

* business model
* recovery model
* quote validation
* accepted job baseline
* job performance interpretation
* invoice validation
* cash explanation
* variable learning

### Xero / accounting software owns

* ledger
* tax/GST
* official invoice records
* debtor ledger
* statements
* compliance
* payment status

## Not allowed

Mirra must not become:

* a full accounting system
* a full accounts receivable system
* a full stock/inventory system
* a full payroll system
* a full ERP
* a generic job-management clone

## Future page / hook / calculation separation

All future modules must preserve:

```text
page = orchestration only
hook = integration
calculations = business logic
selectors = shaping
components = UI only
```

No business maths in components.

## Downstream contract rule

Every future module must consume upstream outputs rather than rebuild them.

## Variable system rule

Do not ask users to define variables upfront.

Variables must emerge from:

```text
expected vs actual
```

Across:

* quantity
* cost
* price
* productivity
* timing

Do not create a complex variable setup UI in v1.

## Performance intelligence rule

The Performance Intelligence page is not inventory.

It must not manage stock on hand, reorder points, or warehouse movements.

It must aggregate:

* quoted vs actual
* expected vs actual
* allowance vs usage
* price vs realised price
* labour allowance vs actual labour

Its purpose is to find repeated performance patterns and recommend quoting improvements.

## Draft invoice rule

The Draft Invoice Checker must not force invoice values.

It must explain the consequence of the invoice before it is pushed to Xero.

## Cash flow rule

Cash Flow Intelligence must not be generic cash forecasting.

It must explain cash pressure by cause:

* timing
* under-recovery
* under-claiming
* volume
* cost structure

It must use real debtor/payment data where available.

## Quote / job version rule

Accepted job baseline must preserve the quote version accepted.

Future quote edits must not rewrite job baseline.

Any scope changes must be captured as controlled variations later.

## Build order rule

Do not build Cash Flow Intelligence before:

1. Quote Checker
2. Accepted Job Baseline
3. Labour Tracking
4. Purchase Order Engine
5. PO / Cost Reconciliation
6. Draft Invoice Checker
7. Xero Push / Debtor Sync

Cash is the outcome layer. It must consume cause layers.

## Final definition

This future guardrail preserves Mirra as a closed-loop business decision engine that validates pricing, tracks execution, checks invoice recovery, explains cash pressure, and learns from actual performance to improve future quotes.

Mirra is a closed-loop business system that checks pricing, tracks execution, validates recovery before invoicing, explains cash pressure, and learns from actual performance to improve future quotes.
